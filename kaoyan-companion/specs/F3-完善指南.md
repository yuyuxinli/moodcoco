## 1. 功能概述

每天早 8 点基于昨日完成情况、遗忘成本、各科提分 ROI、状态机模式，动态生成当日学习计划并通过微信推送。

**权重**：有用性 **高** | 实事求是 **高** | 被看见感 中 | 安全性 低（门槛 8.0）

## 2. 测试场景

F3 是 Functional 类型，测试重点：**A（基础跑通）+ C（跨 session 记忆持久）**。核心验证点：计划是否基于 tracker 数据个性化生成。

### 测试 A：基础跑通（简单对话）

#### A1：首次计划推送（无 tracker 数据）

**前置条件**：用户已完成 F2 诊断（`kaoyan_diagnosis_date` 非空），无任何 tracker 文件，无 plan_history

**输入**：Cron 08:00 触发，或用户说"帮我排计划"

**期望输出**：
- 推送 2 条 ai_message（开场 + 计划），不合成一条
- 开场使用"第一天"话术："这是你的第一份每日计划。说实话，第一周的计划不会特别准......"
- 计划基于 F2 诊断数据（`kaoyan_daily_hours`、`kaoyan_estimated_scores`、`kaoyan_self_eval`）生成
- 各科时间分配符合当前阶段权重（距 2026-12-19 计算 remaining_days）
- 时间粒度为 0.5h 步长，科目保底不低于 MIN_DAILY
- 所有科目时间之和 == available_hours
- 写入 `memory/kaoyan/plan_history.md`，`day_number: 1`，`mode` 与 `kaoyan_plan_state` 一致
- 写入 USER.md：`kaoyan_plan_day_count: 1`

**验证要点**：
- [ ] 2 条消息分发（非合成一条）
- [ ] 首日开场话术正确
- [ ] plan_history.md 文件已写入，格式符合 Layer-2 契约 2.5 plan_history schema
- [ ] USER.md `kaoyan_plan_day_count` 已更新

#### A2：target_score 自动计算（F2 只写了总分线）

**前置条件**：USER.md 中 `kaoyan_target_scores` 为空，`kaoyan_retest_line` 非空，`kaoyan_estimated_scores` 和 `kaoyan_subject_scores` 已由 F2 写入

**输入**：F3 首次运行

**期望输出**：
- 按 SKILL.md "target_score 科目拆解"算法计算各科目标分
- 每科目标分 <= 满分 * 0.9（90% 封顶）
- 单科线约束：`target_score[s] >= single_subject_line[s]`
- 封顶后溢出重分配到未封顶科目
- 若 sum(cap) < retest_line，推送中坦诚提示目标差距
- 写入 USER.md：`kaoyan_target_scores`、`kaoyan_target_achievable`、`kaoyan_target_total`

**验证要点（用具体数据验算）**：
- [ ] 假设用户：政治自评 5/10（满分 100）、英语 5/10（满分 100）、专业课 3/10（满分 300），retest_line = 360
- [ ] estimated: 政治 50、英语 50、专业课 90，total = 190，gap = 170
- [ ] ease_factor: 政治 1.0、英语 1.0、专业课 1.5（ratio<0.5 → 1.5 倍 × 满分）
- [ ] raw_gap 归一化后按比例分配 170 分 gap
- [ ] 封顶 90%：政治 <= 90、英语 <= 90、专业课 <= 270
- [ ] 结果写入 USER.md，数值自洽

#### A3：正常模式计划推送（有昨日 tracker 数据）

**前置条件**：
- `kaoyan_plan_state: "normal"`
- 昨日 tracker 文件存在：`status: "done_most"`，`mood: "normal"`，`stuck_points: [{subject: "数学", normalized_topic: "极限-洛必达法则"}]`

**输入**：Cron 08:00 触发

**期望输出**：
- 开场话术感知昨日状态："昨天{done}做了，{skip}没来得及......"
- 计划中数学任务包含复习标记（`priority: "复习"`），因 stuck_points 匹配
- 数学时间分配有 weakness_boost（近 7 天 stuck_points > 0 → +15%）
- 昨天未完成的任务出现在今日计划（`priority: "补漏"`）
- focus_hint 引用具体薄弱知识点："上次极限-洛必达法则错了，今天重点攻这个。"

**验证要点**：
- [ ] 开场引用昨日具体完成情况（非模板感知，是数据感知）
- [ ] stuck_points 影响了今日任务优先级和时间分配
- [ ] 有"复习"或"补漏"标记的任务

#### A4：轻松模式触发（昨天 bad_day）

**前置条件**：

**输入**：Cron 08:00 触发

**期望输出**：

- 开场话术："昨天歇了一天。没事，今天轻一点，先找回手感。"
- 计划使用轻松模式格式："今天轻一点，不逼你。......做了就是赢。做不完也没关系。"
- 任务描述简化（只做最核心部分）
- plan_history 记录 `mode: "easy"`

**验证要点**：
- [ ] 状态机转换正确（normal → easy）
- [ ] available_hours 降为 60%
- [ ] 推送格式切换为轻松模式

#### A5：最小行动模式触发（连续 2 天 done_none）

**前置条件**：
- `kaoyan_plan_state: "easy"`
- 昨日 tracker：`status: "done_none"`
- 前日 tracker：`status: "done_none"`
- `streak.consecutive_none: 2`

**输入**：Cron 08:00 触发

**期望输出**：

- 推送使用最小行动格式："今天只需要做一件事：→ {单一任务，30 分钟以内}......做完了告诉我。不做也行，我明天还会发。"

**验证要点**：
- [ ] 状态机转换正确（easy → minimal）
- [ ] 只推送 1 个任务，30 分钟以内
- [ ] 话术零压力、无评判

### 来自 Codex 独立评审（eval_codex_groupB_v4_independent.md）

以下问题在现有 SKILL.md 中可能仍存在，需重点检查：

- [ ] **两套时间分配算法并存**：SKILL.md Step 2 按 `phase_weight` 直接分配 vs Step 2 末尾按 `normalize(ROI_score)` 分配。实现应统一为 ROI_score 路径（phase_weight 是 ROI 公式的因子之一），确认 SKILL.md 中无歧义
- [ ] **redistribute_overflow() 日计划版本缺失**：SKILL.md §Step 2 末尾调用的 `redistribute_overflow(subjects, available_hours)` 是"从权重最低科目减时间直到 sum==available_hours"，与 target_score 封顶版本语义不同，确认实现正确
- [ ] **exam_subjects / current_phase / subject.type 来源未声明**：这些变量在算法中使用但未明确定义来源。exam_subjects 应来自 USER.md 的科目代码（F2 诊断写入），current_phase 应由 remaining_days 计算，subject.type 应与科目代码映射（101→政治、201→英语等）
- [ ] **weekly_recommendations.plan_intensity 系数映射**：Layer-2 契约 2.5 已定义映射（normal→×1.0, reduced→×0.8, minimum→×0.6），确认 SKILL.md 实现与契约一致
- [ ] **politics_minimum 字符串格式与时间粒度不兼容**：weekly_cache 中 `politics_minimum: "每天 15min 保底"` 是字符串，但 F3 计划引擎需要数值。需定义解析规则或改为数值格式
- [ ] **"目标缺口提示"场景模板缺失**：§target_score 拆解中提到 `sum(cap) < retest_line` 时坦诚提示，但推送消息格式章节无对应模板
- [ ] **"同一句 7 天内不重复"无存储机制**：开场话术去重规则只有规则没有存储——plan_history 不记录 opening_id，无法回溯最近 7 天用了哪句开场
- [ ] **last_studied_content 来源未定义**：Step 3 任务模板中 `"快速回顾{last_studied_content}"` 的 last_studied_content 应来自最近一天该科 tracker 的 plan[].topic，需确认
- [ ] **momentum 模式触发后仍不加量验证**：确认 momentum 模式的 available_hours == base（100%），不会因为连续完成而增加任务量
