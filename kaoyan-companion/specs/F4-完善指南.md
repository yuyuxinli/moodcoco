## 1. 功能概述（一句话）

每晚 22:00 Cron 推送或用户主动触发，1 次点击完成每日学习打勾（核心路径 ≤15 秒），记录 status/completed/stuck_points/streak 到 `tracker/YYYY-MM-DD.md`，供 F3 调整计划、F5 生成周报、F6 检测崩溃。

## 2. 测试场景

F4 是 Functional 类型，测试重点 **A（基础跑通）+ C（跨 session 记忆持久）**。

### 测试 A：基础跑通（简单对话）

#### A1：Cron 触发 - done_all 完整路径

**前置条件：** 用户已完成 F2 诊断（USER.md 有 `kaoyan_target_school`），当天 F3 已推送计划（plan_history 有今日记录，包含 3 个任务：数学极限10题、英语阅读2篇、政治马原第3章）。`kaoyan_tracker_preference` 不为 `dislike`。`kaoyan_plan_state` 不为 `dormant`。

**输入：** 22:00 Cron 触发

**期望输出：**
1. 可可发送"今天辛苦了，来打个勾吧"+ 5 选项 poll（全部搞定/做了大部分/做了一点点/今天没学/状态不好）
2. 用户点击"全部搞定"
3. 可可回应肯定，**必须引用今日具体任务内容**（如"极限那 10 道题都做了"），不能泛泛说"今天做得好"
4. 进入错题追问（可选）：6 选项 poll（没有都还顺利/数学有卡住/英语有卡住/政治有卡住/专业课有卡住/不想说了）
5. 用户选"数学有卡住"→ 可可追问"数学哪块卡住了？"→ 用户输入"洛必达"→ 可可结束打勾

**数据验证：** 检查 `memory/kaoyan/tracker/{today}.md` 文件：
- `status: "done_all"`
- `trigger: "cron"`
- `plan[]` 有 3 个任务且与 plan_history 今日记录一致
- `completed[]` 全部 `done: true`
- `stuck_points[0].subject: "数学"`，`stuck_points[0].normalized_topic` 包含"洛必达"相关标准化名
- `streak.consecutive_days` 正确计算

#### A2：done_most 分支 + 未完成任务追问

**输入：** Cron 触发 → 用户点"做了大部分"

**期望输出：**
1. 可可回应肯定已完成部分，**不提未完成的**，问"哪个没来得及做？"
2. 选项动态生成：基于今日 F3 推送的具体任务列表 + "好几个都没做完"
3. 用户选择具体未完成任务

**数据验证：**
- `status: "done_most"`
- `completed[]` 中被选中的任务 `done: false`，其余 `done: true`

#### A3：done_some 分支

**输入：** Cron 触发 → 用户点"做了一点点"

**期望输出：**
1. 可可回应"做了就算数"正向框架，问做了什么（不问为什么没做完）
2. 选项动态生成：基于今日任务 + "做了点别的"
3. **不追问错题**（done_some 不触发错题追问）

**数据验证：**
- `status: "done_some"`
- `completed[]` 按用户选择填充

#### A4：done_none 分支

**输入：** Cron 触发 → 用户点"今天没学"

**期望输出：**
1. 可可回应零压力、零评判（不说"没关系明天补上"、不说"休息也很重要"）
2. **不追问原因，不追问任何东西，直接结束**

**数据验证：**
- `status: "done_none"`
- `completed[]` 为空或全 `done: false`
- `mood: "normal"`（done_none 不等于 mood:bad）

#### A5：bad_day 分支 → F6 路由

**输入：** Cron 触发 → 用户点"状态不好"

**期望输出：**
1. 可可发送"怎么了？想说说吗？"
2. 等待用户回复后路由到 F6 kaoyan-crisis Phase 1
3. **打勾流程结束，不继续追问学习情况**

**数据验证：**
- `status: "bad_day"`，`mood: "bad"`
- F6 crisis 流程被触发

#### A6：用户主动打勾

**输入：** 用户在非 22:00 时间发送"今天做完了"或"打个勾"

**期望输出：**
1. **跳过"今天辛苦了"开场**，直接进入 5 选项 poll
2. 后续流程与 Cron 触发一致

**数据验证：**
- `trigger: "user_initiated"`

#### A7：重复打勾

**输入：** 当天已打过勾，用户再次说"打个勾"

**期望输出：**
1. 可可回应"今天已经打过勾了，要更新一下吗？"
2. 用户选"是"→ 覆盖今日记录
3. 用户选"否"→ 结束，不修改

#### A8：用户直接打字不点选项

**输入：** Cron 触发后，用户不点 poll，直接打字"今天做完了，极限那块终于搞懂了"

**期望输出：**
1. AI 从文本提取 status → done_all
2. AI 从文本提取正面反馈 → 存入 note 字段
3. 不强制走 ai_options 流程

**数据验证：**
- `status: "done_all"`
- `note` 字段包含用户正面反馈内容

#### A9：F3 当天未推送计划

**前置条件：** 当天 F3 未推送（plan_history 无今日记录）

**输入：** 22:00 Cron 触发 → 用户点"做了大部分"

**期望输出：**
1. 打勾正常触发
2. 追问选项不基于具体任务（因无计划），仅记录 status

**数据验证：**
- `plan[]` 为空
- `completed[]` 为空
- `status: "done_most"` 正常记录

#### A10：用户说"别每天问我"

**输入：** 打勾流程中或任何时候用户表达不想被每天追问

**期望输出：**
1. 立即停止 Cron 推送
2. 确认话术（非道歉，不施压）

**数据验证：**
- USER.md 新增 `kaoyan_tracker_preference: "dislike"`
- 后续 22:00 Cron 不再触发（前置条件不满足）

## 3. 支撑层检查

### 记忆系统

- **写入位置：** `memory/kaoyan/tracker/YYYY-MM-DD.md`，一天一个文件
- **写入方式：** `diary_write(date="YYYY-MM-DD", type="kaoyan-tracker", entry=content)`
- **YAML schema 一致性：** 写入文件必须与 Layer-2 契约 2.5 中定义的 tracker schema 完全一致（必填字段：date, weekday, check_in_time, trigger, status, plan, completed, mood, note, streak）
- **streak 计算：** 每次写入时重新计算，不沿用旧值。读取前 N 天 tracker 文件逐天回溯

### 知识库

- F4 不直接依赖知识库。知识库在 F2 诊断时使用（`references/kaoyan-knowledge/`）

### 时间感知

- **22:00 推送时间：** 固定，不随用户状态变化（v1 统一 22:00）
- **凌晨回复（00:00-06:00）：** date 取推送日期（昨天），不取回复日期
- **崩溃锁定期抑制：** 处于 F6 崩溃锁定期（`last_crisis_date` 距今 <= 3 天）时，22:00 不推送打勾（Layer-2 契约 2.7 Cron 3 抑制条件）
- **周日互斥：** 周日 20:00 已发周报 → 22:00 不发打勾（周报含打勾功能）
- **dormant 状态：** `kaoyan_plan_state == "dormant"` 时不推送

### 跨 skill 状态

| 场景 | 期望行为 | 验证方式 |
|------|---------|---------|
| F4 status → F3 消费 | 昨日 done_none → F3 今日降级 60%；昨日 mood:tired → 降级 20%；昨日 completed 未完成任务 → 今日补入 | 检查 Day N+1 F3 推送的计划内容和任务量 |
| F4 status → F6 触发 | 连续 3 天 done_none（不含 done_some）→ F6 崩溃接住 | 检查第 3 天后 F6 是否被触发 |
| F4 status → F5 消费 | 本周 7 天 tracker → F5 计算各科完成率、薄弱点排行、连续天数 | 检查周日 F5 周报引用的数据与 tracker 文件一致 |
| F4 bad_day → F6 路由 | bad_day 选项 → 立即路由 F6，打勾流程结束 | 检查 F6 crisis 流程启动 |
| F6 崩溃 → F4 抑制 | F6 触发后 3 天内，22:00 不推送打勾 | 检查崩溃后 Day 1-3 无打勾推送，Day 4 恢复 |
| 状态机转换 | done_none 在 normal 状态 → 切 easy；done_all 在 easy 状态 → 切 normal | 检查 USER.md `kaoyan_plan_state` 字段变化 |

## 4. 已知问题（从 Codex 评审继承）

以下问题从 Round 1 Codex 评审（eval_codex_groupB v1-v4）中提取，按优先级排列。

### P0（必须修复，影响数据有效性和可执行性）

- [ ] **mood:tired 产出规则缺失**（Codex v4 independent, F4 P0-1）：tracker schema 中 `mood` 有 `normal | tired | bad` 三个枚举值，`bad` 在 bad_day 分支产出，`normal` 为默认值，但 **`tired` 何时产出没有定义**。F3 依赖 `mood == "tired"` 做次日降级 20%，但 F4 当前没有任何路径会写入 `tired`。需要定义：哪个打勾分支 / 用户表述会触发 `mood: "tired"`（例如：done_some + 用户表达疲惫类关键词）
- [ ] **done_most/done_some 多选落库规则不明确**（Codex v4 independent, F4 P0-2）：done_most 追问"哪个没做"时，选项包含具体任务和"好几个都没做完"（multiple）。用户可能同时选具体任务 + multiple，此时 `completed[]` 怎么写没有定义。需要明确：(a) multiple 与具体任务互斥？(b) 同时选时以哪个为准？(c) completed[] 的落库逻辑

### P1（建议修复，不阻塞基础功能但影响严谨性）

- [ ] **HEARTBEAT.md Cron 表达式语法错误**（Codex v4 independent, F4 P1-3）：sub-PRD 中 general_check_in 的 Cron 表达式写成 `0 21 30 * * *`（6 字段，标准 cron 是 5 字段），应为 `30 21 * * *`。注意：Layer-2 契约 2.7 中 F4 自身的 Cron 表达式 `0 22 * * *` 是正确的
- [ ] **凌晨主动打勾日期归属未覆盖**（Codex v4 independent, F4 P1-4）：现有规则只覆盖"凌晨回复昨天 Cron 推送"→ 记到昨天。但"凌晨用户主动打勾（非回复推送）"该记到哪天没有定义。需要补充规则（建议：00:00-06:00 主动打勾统一记到昨天）
- [ ] **completed[] 字段"必填"与"可为空"矛盾**（Codex v4 independent, F4 数据有效性扣分）：Layer-2 契约 2.5 标注 `plan` 和 `completed` 可为空 Array（F3 未推送时），但 SKILL.md 中 completed[] 在字段用途表标"必填"。需统一表述：completed[] 在有计划时必填，F3 未推送时允许为空
- [ ] **missing_day → F3 消费分支未显式定义**（Codex v4 independent, 跨文档 P0-1 的 F4 侧）：F4 定义了 missing_day = 无文件，F3 应按 done_some 保守估计。但 F3 算法中没有显式的"无 tracker 文件"处理分支。F4 侧需确认：missing_day 的语义（无文件 = 该天不存在 tracker）是否在所有下游（F3/F5/F6）的读取代码中有对应处理

## 附录：Layer-2 契约关键引用

### tracker YAML schema（2.5 数据契约）

```yaml
---
date: "2026-04-05"
weekday: "Saturday"
check_in_time: "22:03"
trigger: "cron"                  # cron | user_initiated
status: "done_most"              # done_all | done_most | done_some | done_none | bad_day

plan:                            # Array，F3 未推送时为空
  - id: "task_1"
    subject: "数学"
    topic: "极限-洛必达法则"
    description: "张宇1000题 第96-105题"
    estimated_minutes: 90

completed:                       # Array，与 plan 一一对应
  - task_id: "task_1"
    done: true

stuck_points:                    # Array，可为空
  - subject: "数学"
    description: "极限洛必达那块"
    normalized_topic: "极限-洛必达法则"

mood: "normal"                   # normal | tired | bad
note: ""

streak:
  consecutive_days: 3
  consecutive_complete: 2
  consecutive_none: 0
---
```

### 状态机转换（2.6 状态机，与 F4 相关的行）

| 当前状态 | done_all | done_none | bad_day | F6崩溃 |
|---------|----------|-----------|---------|--------|
| normal | 保持（连续3天→momentum） | →easy | →easy + F6 路由 | →minimal（锁3天） |
| momentum | 保持 | →normal | →easy + F6 路由 | →minimal（锁3天） |
| easy | →normal | 保持（连续2天→minimal） | 保持 + F6 路由 | →minimal（锁3天） |
| minimal | →easy | 保持 | 保持 + F6 路由 | →minimal（锁3天） |

### Cron 3 调度（2.7 调度契约）

- **触发时间：** 每日 22:00
- **前置条件：** `kaoyan_target_school` 存在 且 `kaoyan_tracker_preference != "dislike"` 且 `kaoyan_plan_state != "dormant"`
- **抑制条件：** 崩溃锁定期（last_crisis_date 距今 <= 3 天）→ 不推送
- **互斥条件：** 周日 20:00 已发周报 → 22:00 不发打勾
- **读取文件：** plan_history.md（今日计划）, tracker/{yesterday}.md, tracker/{day-before-yesterday}.md
- **写入文件：** tracker/{today}.md
