## 1. 功能概述

每周日 20:00，基于本周 7 天 F4 tracker 数据生成学习报告——先展示进步再温和提薄弱，输出 `weekly_cache` JSON 供 F3 下周消费。

**类型**：Functional
**标准权重**：有用性 **高** / 实事求是 **高** / 被看见感 中 / 安全性 低

---

## 2. 测试场景

F5 是 Functional 类型，测试重点 **A（基础跑通）+ C（记忆持久）**。核心验证点：周报数据是否来自真实 tracker 文件，而非 AI 编造。

### 测试 A：基础跑通

#### A1：标准周（5 天数据，有进步）

**输入准备**：

在 `memory/kaoyan/tracker/` 创建本周一至周五 5 个 YAML 文件，数据特征：
- 数学：6 个 plan，完成 5 个（83%）
- 英语：4 个 plan，完成 3 个（75%）
- 政治：3 个 plan，完成 3 个（100%）
- stuck_points：`极限-洛必达法则` 出现 2 次
- mood：4 天 normal、1 天 tired

在 `memory/kaoyan/weekly_cache/` 创建上周 JSON 文件，上周数据：
- 数学完成率 60%，英语完成率 70%，政治完成率 80%
- active_days: 4

**触发**：模拟周日 20:00 Cron 触发

**期望输出**：
- [ ] 4 段消息分段发送（Part 1 开场 + Part 2 进步 + Part 3 薄弱 + Part 4 结尾）
- [ ] Part 2 引用具体数据：数学完成率从 60% → 83%（+23%），英语 70% → 75%
- [ ] Part 3 薄弱点不超过 2 个，提及 `洛必达法则` 卡了 2 次
- [ ] Part 3 每个薄弱点带具体行动指引（如"下周计划里会加几道洛必达法则专项练习"）
- [ ] Part 4 包含 Poll（"想聊聊" / "看看就好"）
- [ ] **所有数字可在 tracker 文件中追溯**——不编造数据

**有用性检查**：
- [ ] 行动指引具体到知识点级别，不是泛泛"加强数学"
- [ ] `recommendations` JSON 写入 `weekly_cache`，`plan_intensity` 有值

**实事求是检查**：
- [ ] 不出现因果猜测（"可能是时间不够""是不是那天课多"）
- [ ] 不出现结果预测（"后面会顺很多""完成率会更好"）
- [ ] 不出现虚高鼓励（"功夫没白费"——这是因果推断，不是事实描述）

#### A2：首周（无上周缓存）

**输入准备**：

`memory/kaoyan/tracker/` 放 4 天数据，`weekly_cache/` 为空。

**期望输出**：
- [ ] 走首周模板：不做跨周对比
- [ ] 只展示本周原始数据（各科 plan 数、完成数）
- [ ] 明确说明"从下周开始我就能帮你看出更多了"（坦诚数据不够）
- [ ] 不编造"趋势"——首周没有趋势

#### A3：数据不足（< 3 天）

**输入准备**：

`memory/kaoyan/tracker/` 仅放 2 天数据。

**期望输出**：
- [ ] 不发周报
- [ ] 发送关怀消息："这周辛苦了。下周想让我帮你盯着学习进度吗？"
- [ ] 不评判用户（不说"你只打了 2 天卡"）

#### A4：全部 done_none / bad_day

**输入准备**：

7 天 tracker 文件全部 `status: done_none` 或 `status: bad_day`。

**期望输出**：
- [ ] 不发周报
- [ ] 走关怀路径："这周不容易吧。不想聊学习也行，聊聊别的？"
- [ ] 不发任何数据分析

#### A5：退出机制

**输入**：用户说"别搞这种总结"

**期望输出**：
- [ ] 回复"好，不发了。"
- [ ] USER.md 写入 `kaoyan_weekly_preference: disabled`
- [ ] 后续周日 20:00 不再触发周报

#### A6：薄弱点表述温度

**输入准备**：

英语完成率 40%、政治 0 数据（本周无 plan）。

**期望输出**：
- [ ] 英语低完成率：用"在爬坡（40%）"而非"只有 40%，太低了"
- [ ] 政治缺勤：用"没顾上，下周哪怕每天 15 分钟也行"而非"完全没学"
- [ ] 薄弱点总数不超过 2 个

### 测试 C：记忆持久（跨 session）

#### C1：weekly_cache 写入持久化

**Session 1**：触发周报生成，让 F5 完成全部流程。

**Session 2**（新 session）：
- [ ] 检查 `memory/kaoyan/weekly_cache/YYYY-WNN.json` 文件是否存在
- [ ] JSON 格式是否符合 Layer-2 §2.5 schema
- [ ] `summary.subjects` 各科 `plan_count`/`done_count`/`completion_rate` 与 tracker 原始数据一致
- [ ] `stuck_topics` 聚合正确（按 `normalized_topic` 计数）
- [ ] `progress_signals` 和 `weaknesses` 数组非空且类型正确
- [ ] `recommendations.plan_intensity` 值为 `normal | reduced | minimum` 之一

#### C2：F5→F3 联动验证

**Session 1**：F5 周报生成，`recommendations` 写入 `weekly_cache`。
- 制造场景：本周状态差（bad_days >= 2），使 `plan_intensity` = `reduced`

**Session 2**（新 session，次周一 08:00）：
- [ ] F3 读取最新 `weekly_cache` 的 `plan_intensity`
- [ ] `plan_intensity: reduced` 时 F3 实际减量（`kaoyan_plan_adjustment × 0.8`）
- [ ] F3 采纳 `math_focus` / `english_adjustment` 等科目级建议

#### C3：连续多周进化感

**Session 1**：创建第 1 周 `weekly_cache`，触发第 2 周周报。
**Session 2**：创建第 2 周 `weekly_cache`，触发第 3 周周报。
**Session 3**：创建第 3 周 `weekly_cache`，触发第 4 周周报。

逐 session 验证：
- [ ] 第 2 周：出现"本周 vs 上周"二维对比，薄弱点精度到知识点级
- [ ] 第 4 周：出现多周趋势表述（如"连续 X 周在涨"），薄弱点精度到知识点+频次
- [ ] `week_number` 字段正确递增
- [ ] 不再出现首周的"数据还不多"坦诚说明

#### C4：weekly_cache 保留策略

**准备**：在 `weekly_cache/` 放入 10 周历史 JSON 文件。

**触发**：生成新一周周报。

- [ ] 写入新文件后，最早 2 个文件被清理，仅保留最近 8 周
- [ ] 清理不影响本次周报对比（本次读取的是上周文件）

## 3. 支撑层检查

### 记忆系统

- [ ] F5 只读 `memory/kaoyan/tracker/` 和 `weekly_cache/`，不写 tracker
- [ ] F5 写入 `weekly_cache/YYYY-WNN.json` 后，`memory_search` 不索引 JSON 文件（JSON 不被 memory_search 索引，这是正确行为）
- [ ] F5 读取 USER.md 的 `kaoyan_target_school`/`kaoyan_weekly_preference` 等字段

### 跨 skill 状态

- [ ] F5 `recommendations` → F3 下周计划：`plan_intensity` 系数映射（normal=1.0, reduced=0.8, minimum=0.6）生效
- [ ] F5 检测到危机信号（tracker 中 mood:bad 连续 + note 含危机关键词）→ 不发周报，走 F6
- [ ] F4 连续 3 天 `done_none` → F5 该周可能不触发（< 3 天有效数据）或触发时走关怀路径

### 时间感知

- [ ] Cron 只在周日 20:00 触发，其他时间用户问"我的周报呢"不主动生成
- [ ] 日期范围计算正确（本周一到本周日），不跨周

### 数据口径一致性

- [ ] `tracker.plan[].subject` 与 `weekly_cache.summary.subjects` 的 key 名一致（中文科目名）
- [ ] `tracker.stuck_points[].normalized_topic` 与 `weekly_cache.summary.subjects[].stuck_topics` 的 key 一致
- [ ] `tracker.completed[].task_id` 与 `tracker.plan[].id` 一一对应

## 4. 已知问题（从 Codex 评审继承）

来源：`.evolve-round1-backup/eval_codex_groupB_v4_independent.md`（Codex 独立评审 v4）

### P0（必须修复）

- [ ] **示例话术中的因果推断未清干净**：SKILL.md 中"你在极限上的功夫没白费""攻下来后面会顺很多""完成率会更好""计划量和节奏越来越匹配了"均为因果/结果预测，必须改为纯事实描述。
  - 修复方向：所有进步表述只陈述数据变化（"数学完成率从 60% 到 83%"），不推断原因或预测结果。
  - 涉及文件：`skills/kaoyan-weekly/SKILL.md` §Part 2 进步亮点、§薄弱点表述规则

- [ ] **weakness_improving 只检测"双周都出现同一 topic"，卡点消失捕不到**：上周出现的 stuck_topic 本周未出现，应该也算进步信号（`type: weakness_resolved`）。
  - 修复方向：在 `detect_progress()` 中增加逻辑——遍历上周 `stuck_topics`，如果本周该 topic 未出现 → 生成 `weakness_resolved` 信号。
  - 涉及文件：`skills/kaoyan-weekly/SKILL.md` §Step 3 进步检测

### P1（建议修复）

- [ ] **detect_progress/detect_weakness 运行时对象与 weekly_cache.summary 字段映射缺失**：函数中使用 `this_week.subjects`、`this_week[subject].completion_rate` 等运行时属性，但与 `weekly_cache.summary.subjects["数学"].completion_rate` 的持久化结构没有显式映射。
  - 修复方向：在 SKILL.md 数据汇总章节增加一个映射表，明确运行时对象属性与 `weekly_cache` JSON 路径的对应关系。例如：
    ```
    this_week.active_days        ← summary.active_days
    this_week[subject].completion_rate ← summary.subjects[科目名].completion_rate
    this_week.stuck_topic_counts ← summary.subjects[科目名].stuck_topics
    last_week.*                  ← 上周 weekly_cache 同路径
    ```

- [ ] **坏数据异常处理缺失**：当前异常处理只覆盖业务场景（< 3 天、全 done_none 等），不覆盖坏数据场景。
  - 需要补充的坏数据场景：
    - tracker YAML 缺少必填字段（如无 `status`、无 `plan`）→ 跳过该天，视为 missing_day
    - `completed[].task_id` 找不到对应 `plan[].id` → 忽略该条 completed 记录
    - `weekly_cache` JSON 格式损坏或缺字段 → 视为无上周数据，走首周逻辑
    - `mood` 值不在枚举范围内 → 视为 `normal`
  - 涉及文件：`skills/kaoyan-weekly/SKILL.md` §异常处理
