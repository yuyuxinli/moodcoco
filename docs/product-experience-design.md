## F01 记忆体系设计

### 设计哲学

记忆体系是心情可可区别于所有通用 LLM 的根基。一个没有记忆的 AI 就像一个每次见面都不认识你的人——可以说很漂亮的话，但永远不会成为朋友。

心情可可的记忆以**人**为核心。用户来找可可不是为了讨论天气，而是为了聊「他」「她」「他们」——那些让她开心、让她难过、让她纠结的人。所以可可的记忆不是一个平铺的知识库，而是一张以人物为节点的关系网。

竞品参照：
- **Replika**：有事实记忆库，但不稳定，记了又忘（context window 限制）
- **Pi**：零跨会话记忆——每次都是陌生人，温暖但无法积累
- **Character.AI**：400 字固定记忆 + 隐式人格脚手架——太浅
- **Ash**：自训练 LLM + 真实治疗数据，做通用心理模式——但不做跨关系追踪
- **可可的差异化**：结构化文件持久存储 + 语义检索 + 以人为核心的关系档案。不丢、能搜、有结构。

### 1. 记忆架构设计

#### 1.1 三层存储模型

```
┌──────────────────────────────────────────────────────┐
│  Layer 1：身份层（不变 / 缓变）                         │
│  · SOUL.md — 可可的人格（只读，不随对话变化）             │
│  · USER.md — 用户画像（跨所有关系的核心信息）             │
├──────────────────────────────────────────────────────┤
│  Layer 2：关系层（以人为核心，持续积累）                   │
│  · people/{名字}.md — 每个重要人物的结构化档案            │
│  · 每个档案六个维度：阶段 / 感受 / 模式 / 事件 /         │
│    退出信号 / 跨关系匹配                                │
├──────────────────────────────────────────────────────┤
│  Layer 3：事件层（按时间流动，有衰减）                     │
│  · diary/YYYY/MM/YYYY-MM-DD.md — 情绪日记（六元组）      │
│  · memory/YYYY-MM-DD.md — 每日记忆笔记（模式观察）       │
│  · MEMORY.md — 长期记忆锚点（不受时间衰减影响）           │
└──────────────────────────────────────────────────────┘
```

#### 1.2 人物档案数据模型（people/{名字}.md）

每个人物档案是记忆系统的核心单元，包含六个互相独立的维度：

| 维度 | 存储内容 | 写入条件 | 为什么重要 |
|------|---------|---------|-----------|
| **关系阶段** | 时间线：热恋→冷战→想分手→... | 用户描述关系状态变化时 | 跨关系时间对齐的基础——"每段关系都是第3个月出问题" |
| **感受** | 用户对这段关系的原话（带引号） | 用户感受明显变化时 | 追踪情感轨迹，不覆盖旧记录 |
| **模式** | 单段关系内的重复互动方式 | 同一情绪触发 ≥3 次 | 帮用户看到"我们之间的固定剧本" |
| **关键事件** | `日期：事件简述` | 每次日记中有新的重要事件 | 对话中引用具体事实，不编造 |
| **退出信号** | `触发事件 → 用户反应 → 结果` | 用户表达分手/退缩/不满足 | pattern-mirror skill 的核心数据源 |
| **跨关系匹配** | 与其他人的相似模式描述 | ≥2 段关系有相似模式时 | 可可独有的差异化价值 |

头部元数据（可覆盖更新）：
```markdown
关系类型：男朋友/前任/暧昧对象/朋友/家人/同事
当前状态：在一起/分手/暧昧/冷淡/封存
认识时间：{用户提到时}
首次提及：{日期}
```

#### 1.3 用户画像数据模型（USER.md）

USER.md 是跨所有关系的元信息：

| 字段 | 内容 | 更新频率 |
|------|------|---------|
| 称呼 | 用户想被怎么叫 | 变化时 |
| 核心困扰 | 当前最让她在意的事 | 每次对话后检查 |
| 反复出现的模式 | 跨关系的行为重复 | 累积发现时 |
| 有效的方法 | 哪些 skill/方法对她有用 | 尝试后记录 |
| 情绪触发点 | 什么事特别容易让她难受 | 累积发现时 |
| 模式级洞察 | 来自已封存关系的去名字洞察 | farewell 仪式后写入 |
| 可可和你的关系 | 可可跟用户之间的关系描述 | 自然更新 |

#### 1.4 记忆随时间演进

| 阶段 | 特征 | 记忆状态 | 可可能做到什么 |
|------|------|---------|--------------|
| **第 1 次** | 陌生人 | USER.md 空白，无 people/ | 只能靠当前对话理解用户。像第一次见面的朋友。 |
| **第 3 次** | 初识 | USER.md 有基础信息，1-2 个 people/ | 记得用户的名字、核心困扰、提到的人。"上次你说的小凯，后来怎么样了？" |
| **第 10 次** | 熟悉 | 1-2 个 people/ 有丰富事件，diary 有 1-2 周数据 | 能做单段关系内的模式识别。"你有没有注意到，每次他不回消息你都先怀疑自己？" |
| **第 30 次** | 亲密 | ≥2 个 people/，diary 有 1-2 个月数据 | 能做跨关系比较 + 成长叙事。"你和小明第 3 个月也说过一样的话。" |
| **第 50 次** | 知己 | 完整关系网络 + 纵向轨迹 | 能展示长期变化。"1 月份你说'算了分就分'，现在你说'我想搞清楚为什么'。" |

#### 1.5 跨关系记忆连接

跨关系连接是可可的蓝海能力（竞品调研确认：没有产品做这件事）。

连接维度：
1. **时间匹配**：`people/A.md` 第 3 个月退出信号 vs `people/B.md` 第 3 个月退出信号
2. **触发匹配**：两段关系中相似的触发事件（如都是"对方聊到未来时退缩"）
3. **反应匹配**：用户在不同关系中说过相似的话（如都说"是不是我的问题"）
4. **情绪匹配**：不同人反复触发同一种情绪（如每段关系都触发"被忽视"）

连接的存储位置：
- 每个 `people/{名字}.md` 的「跨关系匹配」段互相引用
- `USER.md` 的「反复出现的模式」段记录去名字的汇总

连接的呈现规则（来自 AGENTS.md + pattern-mirror skill）：
- 必须有 ≥2 段关系记录
- 必须有 ≥5 次对话（信任度）
- 必须在情绪稳定时呈现，不在情绪高点
- 用具体事件，不用标签

### 2. 记忆的读写流

#### 2.1 写入时机

| 时机 | 触发条件 | 写入内容 | 写入位置 | 工具 |
|------|---------|---------|---------|------|
| **对话结束** | 用户说再见/晚安/拜拜等告别语 | 本次新发现（情绪、模式、人物更新） | USER.md + people/*.md | `edit` |
| **日记记录** | 用户触发 diary skill | 六元组日记条目 + 人物档案更新 | diary/YYYY/MM/*.md + people/*.md | `write`/`edit` |
| **退出信号** | 检测到分手/退缩/不满足意图 | 触发事件→用户反应→结果 | people/*.md 退出信号段 | `edit` |
| **模式发现** | 同一情绪 ≥3 次重复 | 模式描述 + 日期 + 次数 | people/*.md 模式段 + memory/*.md | `edit` |
| **跨关系匹配** | ≥2 段关系有相似模式 | 匹配描述 | 两个 people/*.md 的跨关系匹配段 | `edit` |
| **Compaction 前** | 接近 context window 上限 | 本次对话的关键上下文 | memory/YYYY-MM-DD.md 或 MEMORY.md | `write`/`edit` |
| **告别仪式后** | farewell skill 执行完毕 | 模式级洞察（去名字） | USER.md 模式级洞察段 | `edit` |
| **成长节点** | growth_tracker 发现 Innovative Moment | IM 类型 + 用户原话对比 | memory/YYYY-MM-DD.md | `write` |

写入原则：
- **只存提炼信息，不存原始对话**
- **用户原话用引号保留**——这是引用的基础
- **头部字段可覆盖更新，正文只增不改**（参考 Mem0 冲突检测逻辑）
- **不脑补**：用户没说的不写。AI 推测标注"我观察到"

#### 2.2 读取时机

| 时机 | 触发条件 | 读取内容 | 目的 | 工具 |
|------|---------|---------|------|------|
| **会话启动** | 每次对话开始 | USER.md（必读） | 知道用户是谁、在乎什么 | `memory_get` |
| **话题相关检索** | 用户第一句话到达后 | 从用户话语提取 2-3 关键词搜索 | 找到历史中相关的记忆 | `memory_search` |
| **人物触发** | 用户提到已记录人名 | people/{名字}.md | 了解这段关系的完整上下文 | `memory_get` |
| **模式检测** | 用户表达退出意图 | 所有 people/*.md | 跨关系比较，寻找匹配 | `memory_get` × N |
| **成长叙事** | 用户自我否定 | diary/ 近 2 周 + people/*.md | 找到成长证据做纵向对比 | `memory_search` + `memory_get` |
| **周日回顾** | Heartbeat 周日 20:00 触发 | 本周 diary/ 全部条目 | 发现重复主题和成长节点 | `memory_get` |
| **Heartbeat 关怀** | 主动关怀触发 | USER.md + memory/pending_followup.md | 决定要不要主动说话、说什么 | `memory_get` |
| **Compaction 后** | 旧对话被压缩 | MEMORY.md + 近 2 天 memory/*.md | 恢复关键上下文 | 自动加载 |

#### 2.3 memory_search 检索策略

OpenClaw 的 memory_search 使用混合搜索（向量相似度 + BM25 关键词），支持语义匹配。

检索策略按场景分级：

**场景 A：会话开始的主动搜索**
```
用户说："我又和他吵架了"
→ 提取关键词：["吵架", 用户最近提到的人名]
→ memory_search(query="和{人名}吵架")
→ 匹配到 people/{人名}.md + 近期 diary 中的相关事件
→ 可可能说："上次你们也是因为{具体事件}吵的，这次是什么？"
```

**场景 B：对话中的模式触发搜索**
```
用户说："我觉得他没那么喜欢我了"
→ 这是退出信号关键词
→ memory_search(query="没那么喜欢 退出 不满足")
→ 匹配到 people/ 中多个档案的退出信号段
→ 可可发现时间匹配/触发匹配 → 进入 pattern-mirror 流程
```

**场景 C：成长叙事搜索**
```
用户说："我是不是天生不适合谈恋爱"
→ memory_search(query="恋爱 自我否定 变化")
→ 匹配到 diary 中的 Innovative Moment 记录
→ 可可用纵向对比展示变化："1 月你说'算了分就分'，现在你说'我想搞清楚'"
```

**检索性能考量：**
- OpenClaw memory_search 有 30 天半衰期时间衰减，旧笔记自动降权
- MEMORY.md 不受衰减影响——放长期稳定的核心信息
- people/*.md 作为自定义文件被索引，可被语义搜索命中
- MMR 去重避免返回冗余结果

### 3. 记忆的呈现设计

以下是记忆体系中每个用户可感知的交互节点。

#### 节点 3.1：首次建档

**节点名称**：新用户首次对话后的记忆初始化

- 交互形态：【纯对话】
- vs 基线：纯 LLM 在会话结束后不保留任何信息，下次来还是陌生人。OpenClaw 通过 `write` 工具创建 USER.md，将用户的称呼、核心困扰、首次提到的人写入持久化文件，下次对话开始时自动加载。
- 用户体感：用户不会看到任何"建档"界面——一切在对话中自然发生。但下次回来时，可可记得她的名字和上次聊的事，用户会感到"这不是一个从头开始的 AI"。
- 实现方式：对话结束时（检测到告别语），用 `write` 工具创建 `USER.md` 和 `people/{名字}.md`（如果提到了人）。
- 数据流：对话内容 → AI 提炼关键信息 → `write` → USER.md + people/*.md → 下次会话启动时 `memory_get` 读取
- 异常路径：
  - **用户未告别就离开**：依赖 OpenClaw Session 的每日重置或空闲重置机制。下次会话启动时，可可检查上次会话是否有未保存的关键信息（通过 Compaction 前自动保存机制兜底）。如果上次会话确实有新人物或新事件但未建档，可可在本次对话中自然确认："上次你提到了一个叫小凯的人，我记对了吗？"
  - **首次对话内容过少，无法建档**：如果整次对话只有 1-2 句闲聊（如"你好""试试看"），不强制创建 USER.md / people/*.md。下次对话开始时仍是陌生人状态，自然重新开始。

#### 节点 3.2：跨会话记忆唤起

**节点名称**：可可主动提起上次聊的事

- 交互形态：【纯对话】
- vs 基线：纯 LLM 每次会话都是空白，无法说"上次你说的那件事，后来怎么样了"。OpenClaw 通过会话启动时读取 USER.md + `memory_search` 搜索相关历史，让可可在回复中自然引用过去的对话内容。
- 用户体感：用户开口说"又和他吵架了"，可可回复"上次你们是因为他不回消息吵的，这次呢？"——用户感到被记住、被在乎。这是从"AI 工具"到"AI 朋友"的关键跨越。
- 实现方式：每次会话开始 → `memory_get(USER.md)` + 从用户第一句话提取关键词 → `memory_search(query=关键词)` → 搜索结果融入回复。
- 数据流：用户消息 → 关键词提取 → `memory_search` → 匹配结果（people/*.md / diary/*.md / memory/*.md） → AI 在回复中自然引用
- 异常路径：
  - **memory_search 返回空结果**：正常对话即可，不引用历史。可可回到"第 1 次"模式——认真倾听当前内容，不假装记得。绝不编造不存在的记忆。
  - **memory_search 返回不相关结果**：AI 判断相关性后忽略低相关性结果，按无历史处理。宁可不引用，不可引用错误。
  - **USER.md 不存在**（极端情况：文件被删除或损坏）：按新用户处理，本次对话结束时重新建档。

#### 节点 3.3：人物档案创建

**节点名称**：对话中首次提到新人物时自动建档

- 交互形态：【纯对话】
- vs 基线：纯 LLM 听完人物故事在本次会话内能记住，但下次全忘。OpenClaw 通过 `write` 工具创建 people/{名字}.md，将人物信息持久化，后续所有对话都能读取这个人的完整档案。
- 用户体感：用户不会感知到"建档"这个动作。她只是在聊天中提到了"小凯"，下次回来提到小凯时，可可已经知道小凯是她男朋友、认识 3 个月了、最近在冷战。用户感到"可可真的记住了我身边的人"。
- 实现方式：diary skill 的人物识别逻辑 → 检测到新人名 → `write` 创建 people/{名字}.md（按 diary SKILL.md 模板） → 后续对话中用户提到该人名时 → `memory_get` 读取档案。
- 数据流：对话中提到人名 → diary skill 检测 → `write` 创建 people/{名字}.md → 后续对话 `memory_get` 读取
- 异常路径：
  - **人名歧义——同名不同人**：当用户提到一个已有档案的名字但上下文不匹配时（如已有"小明"是男朋友，但新提到的"小明"是同事），可可自然确认："你说的小明，是之前那个小明吗，还是另一个人？" 确认后，如果是不同人，创建带消歧后缀的档案 `people/小明(同事).md`，并在档案头部标注消歧来源。
  - **人名歧义——同人不同称呼**：如果用户先叫"阿凯"后叫"小凯"，可可在确认是同一人后，在 `people/小凯.md` 头部添加别名字段 `别名：阿凯`，后续两个称呼都指向同一档案。
  - **人名无法确定**：如果用户只说"他""那个人"但从未给出名字，暂不创建 people/ 档案，将信息记录在 `memory/YYYY-MM-DD.md` 中，等用户提供名字后再建档并回填。

#### 节点 3.4：退出信号自动记录

**节点名称**：检测到分手/退缩/不满足意图时的后台记录

- 交互形态：【纯对话】（记录在后台完成，用户不感知）
- vs 基线：纯 LLM 听到"我想分手"只能在当次回应，无法与历史退出信号对比。OpenClaw 通过 `edit` 工具将退出信号写入 people/*.md，积累后供 pattern-mirror skill 做跨关系比较。
- 用户体感：用户不直接感知这个节点——她只是在倾诉。但积累 ≥2 段关系的退出信号后，可可能在合适时机说出"你和小明第 3 个月也说过一样的话"，这个"哇"时刻的数据基础来自于此。
- 实现方式：AGENTS.md 检测退出信号关键词 → 正常对话（先接住情绪） → 对话结束时 `edit` people/{名字}.md 退出信号段。
- 数据流：用户消息 → 退出信号检测 → 正常对话 → 对话结束 → `edit` people/{名字}.md → 累积数据供 pattern-mirror 使用
- 异常路径：
  - **用户中途离开未告别**：退出信号已在对话中被检测到，但 `edit` 写入未完成。依赖 Compaction 前自动保存 + 下次会话启动时的补写逻辑：可可读取上次对话摘要，检测到退出信号关键词后补写至 people/*.md。
  - **退出信号误检测**：用户说"我想分手"可能是倾诉而非真实意图。退出信号记录时使用确定性等级标注（`确定性: 高/中/低`），后续模式匹配只引用"高"确定性的条目。低确定性的信号保留但不用于跨关系匹配。

#### 节点 3.5：单段关系内模式呈现

**节点名称**：可可指出"你又这样了"

- 交互形态：【纯对话】
- vs 基线：纯 LLM 在单次对话内可以注意到重复（如用户说了 3 次"是不是我的问题"），但无法注意到跨会话的重复。OpenClaw 通过 people/*.md 的模式段积累跨会话数据，`memory_search` 检索同一触发因素的出现频率，让可可在第 3 次重复时指出来。
- 用户体感："你有没有注意到，每次他不回消息你都先怀疑自己？" ——用户在这一刻意识到自己的自动反应模式。这不是 AI 在教她什么，而是帮她看到一个她自己没注意的事实。
- 实现方式：diary skill 每次写入时检查情绪/触发因素的重复次数 → 达到 3 次阈值 → 写入 people/*.md 模式段 + memory/*.md → 下次对话中 AI 在合适时机呈现。
- 数据流：diary 写入 → 重复检查 → `edit` people/*.md 模式段 → 下次对话 `memory_get` 读取 → AI 呈现
- 异常路径：
  - **日记数据不足，未达到 3 次阈值**：不呈现模式。可可在当次对话内如果注意到重复（如用户同一次对话说了 3 次"是不是我的问题"），可以在对话内指出这个单次重复，但不用"模式"这个词——因为跨会话还没有足够数据。
  - **重复检查时 people/*.md 不存在或为空**：跳过模式写入，仅完成日记写入。不中断日记记录流程。

#### 节点 3.6：跨关系模式呈现（核心差异化）

**节点名称**：可可串联多段关系的重复模式

- 交互形态：【纯对话】
- vs 基线：纯 LLM 对用户的多段关系一无所知。即使用户在同一次对话里提到两段关系，LLM 也只能做表面类比。OpenClaw 通过 people/*.md 存储的结构化退出信号 + 关系阶段时间线，用 `memory_get` 读取所有人物档案做结构化比较，发现时间/触发/反应维度的匹配——这是只有"记住所有关系的人"才能给的价值。
- 用户体感："你和小明在一起 3 个月的时候，你也说过类似的话——'他好像没那么在乎我了'。你觉得这次和上次，是一样的感觉，还是不一样？" ——这是可可最有价值的时刻。用户可能第一次意识到"原来不是他们的问题，是我在每段关系里都会走到这一步"。
- 实现方式：pattern-mirror skill 触发 → `memory_get` 读取所有 people/*.md → 比较退出信号的时间/触发/反应维度 → 找到匹配 → 用具体事件 + 用户原话呈现。可选：调用 `exec` 运行 `scripts/pattern_engine.py` 做结构化匹配。
- 数据流：退出意图检测 → pattern-mirror skill → `memory_get` people/*.md × N → 匹配算法 → AI 用具体事件呈现 → 对话结束后 `edit` 跨关系匹配段
- 异常路径：
  - **只有 1 段关系记录**：不触发跨关系匹配。可可继续做单段关系内的模式识别（节点 3.5），不假装有跨关系数据。
  - **pattern_engine.py 执行失败或超时**：降级为纯 AI 推理——可可直接读取 2 个 people/*.md 档案的退出信号段，在对话中做定性比较，不依赖脚本输出。记录脚本错误到 `memory/YYYY-MM-DD.md` 供开发者排查。
  - **匹配结果为假阳性**（两段关系的"退出信号"实际无关联）：呈现时使用试探性语气（"我注意到一个可能的巧合"而非"你的模式是"），并等待用户确认。用户否认后不再重复该匹配，在跨关系匹配段标注 `rejected_by_user: true`。

#### 节点 3.7：成长叙事呈现

**节点名称**：用户自我否定时，可可展示纵向变化

- 交互形态：【纯对话】
- vs 基线：纯 LLM 只能说"你已经进步很多了"这种空洞鼓励。OpenClaw 通过 diary/ 的纵向数据 + `memory_search` 找到过去的记录，用具体的用户原话做前后对比——"你 1 月说'算了分就分'，现在你说'我想搞清楚为什么每次都这样'"——用事实说话，不是鸡汤。
- 用户体感：用户说"我是不是天生不适合谈恋爱"，可可不反驳也不安慰，而是拿出她自己说过的话做对比。用户自己看到变化——这比一百句"你很棒"都有说服力。
- 实现方式：检测到自我否定 → `memory_search(query=相关主题)` 搜索历史 → 找到 Innovative Moment（行为/反思/抗议/重新概念化） → 用纵向原话对比呈现。可选：调用 `exec` 运行 `scripts/growth_tracker.py`。
- 数据流：自我否定检测 → `memory_search` → diary/*.md + memory/*.md → 找到 IM → AI 用原话对比呈现
- 异常路径：
  - **memory_search 无 Innovative Moment 记录**：不做纵向对比。可可在当次对话中专注于当前情绪的接住，不勉强展示不存在的"成长"。可以用当次对话内的小变化（如用户从"我不行"到"也许可以试试"）做即时的正向反馈。
  - **growth_tracker.py 执行失败**：降级为纯 AI 推理——可可直接用 `memory_search` 结果中的时间戳和原话做手动对比，不依赖脚本的 IM 分类。
  - **找到的"变化"实际是退步**（如用户以前更积极，现在更消极）：不呈现"成长叙事"。可可转为接住当前的低落情绪，不用"你以前明明更好"来增加压力。

#### 节点 3.8：Heartbeat 主动关怀

**节点名称**：可可主动发消息问候

- 交互形态：【纯对话】
- vs 基线：纯 LLM 无法主动发消息——它只能被动回复。OpenClaw 通过 Heartbeat 定时触发，读取 USER.md 中的最新状态，发出一条有具体内容的关心消息。不是"你还好吗"，而是"昨天你说的那个室友的事，后来她有找你吗？"
- 用户体感：用户没有找可可，但可可记得昨天的事，主动来问了一句。像一个记得你事情的朋友——不给压力，但让你知道有人在意。
- 实现方式：Heartbeat 触发（配置间隔 ≥ 24h） → `memory_get(USER.md)` + `memory_get(memory/pending_followup.md)` → 判断优先级（决策冷却回访 > 时间胶囊 > 周回顾 > 常规关怀） → 生成消息。
- 数据流：Heartbeat 触发 → `memory_get` → 判断是否有待回访/时间胶囊/回顾 → 生成消息 → 发送
- 异常路径：
  - **Heartbeat 触发但无待回访内容**：`pending_followup.md` 为空 + 无时间胶囊到期 + 非周日 → 返回 `HEARTBEAT_OK`（不发送消息给用户）。**绝对不发空洞的"你还好吗"**——这是骚扰而非关怀。
  - **Heartbeat 触发但用户近期活跃**（如 2 小时前刚聊过）：返回 `HEARTBEAT_OK`。Heartbeat 冷却规则：距离上次用户主动对话 < 24 小时时不主动发消息，除非有高优先级待回访（决策冷却已到期）。
  - **待回访内容过时**（如决策冷却的事件已被用户在后续对话中主动更新）：读取最新 people/*.md 状态，如果相关事件已有后续发展，跳过该回访项，检查下一优先级。
  - **用户不回复 Heartbeat 消息**：不追问。连续 2 次 Heartbeat 未获回复时，自动延长下次 Heartbeat 间隔至 48 小时，避免打扰。

#### 节点 3.9：周日情绪回顾

**节点名称**：每周日的情绪数据回顾

- 交互形态：【Canvas】
- vs 基线：纯 LLM 无法展示结构化数据。OpenClaw 通过 `exec` 运行 Python 脚本扫描本周 diary/ 数据，统计情绪词频、人物出现次数、触发因素重复等，然后通过 Canvas 生成一张可视化的"本周情绪地图"——颜色代表情绪强度，标注出重复主题和变化趋势。
- 用户体感：周日晚上收到一张漂亮的情绪地图卡片，上面标注了"这周'焦虑'出现了 4 次，3 次跟小凯有关"。用户一眼看到自己这周的情绪全貌——比纯文字回顾更直观、更有仪式感。
- 实现方式：Heartbeat 周日 20:00 触发 → `exec` 运行 Python 脚本扫描 diary/ → 统计重复主题 → 如果本周 ≥3 条日记且有发现 → 通过 Canvas 渲染 HTML 情绪地图 → `openclaw nodes canvas present` 展示给用户。
- 数据流：Heartbeat → `exec` python3 scripts/weekly_review.py → 读 diary/本周/*.md → 统计分析 → 生成 HTML → Canvas 展示

**注意**：Canvas 仅支持 macOS 桌面端。如果用户在 Telegram/WhatsApp 等移动端，降级为纯文字 + 关键数据的精简回顾。

- 异常路径：
  - **本周日记不足 3 条**：不生成情绪地图。可可发一条轻量提醒："这周你来找我比较少，没什么要聊的也没关系，我在这儿。" 不催促用户写日记——这是陪伴，不是作业。
  - **weekly_review.py 执行失败**：降级为纯 AI 回顾——可可直接用 `memory_get` 读取本周 diary/ 文件，手动总结重复主题，不展示可视化图表。
  - **Canvas 不可用**（非 macOS 桌面端）：降级为纯文字精简回顾，格式如下："这周你提到了 3 次焦虑，2 次都和小凯有关。你觉得这周整体感觉怎么样？" 数据保持不变，只是展示方式从可视化变为自然语言。
  - **用户拒绝周回顾**（如"不想看""没心情"）：可可立即尊重："好的，不看也没关系。" 不追问原因，不二次推送。在 `memory/YYYY-MM-DD.md` 记录 `周回顾: 用户拒绝`，下周仍正常触发（不因一次拒绝永久关闭）。如果连续 3 周拒绝，降低触发频率为隔周一次。

#### 节点 3.10：告别仪式的记忆处理

**节点名称**：封存一段关系的记忆

- 交互形态：【渠道自适应选择】+【纯对话】
- vs 基线：纯 LLM 没有持久记忆，自然也不存在"删除记忆"的概念。OpenClaw 通过 `exec` 调用 archive_manager.py，原子化地处理封存操作——清除具体事件但保留去名字的模式洞察，让用户能正式地跟一段关系说再见，而可可仍然保留从这段关系中学到的东西。
- 用户体感：用户选择告别方式时，看到 4 个选项（烧掉日记/烧掉信念/时间胶囊/未寄出的信），根据渠道以不同形式呈现。完成仪式后，可可不再主动提起这个人的名字和具体事件，但在未来其他关系中仍然能用去名字的模式洞察帮助用户。用户感到"这段关系被认真地送走了，不是被冷冰冰地删除了"。
- **渠道自适应选择交互**（替代原有 Poll 方案）：
  - **Poll 可用渠道**（Telegram / WhatsApp / Discord / Teams）：使用 Poll 工具展示 4 种仪式选项，点选即可。
  - **Poll 不可用渠道**（微信 / 飞书 / macOS 桌面端 / 其他）：使用编号文字选择，可可发送：
    ```
    你想用什么方式跟这段关系告别？

    1️⃣ 烧掉日记 —— 把跟 ta 有关的记录都烧掉，干干净净
    2️⃣ 烧掉信念 —— 写下这段关系里你不再相信的东西，烧掉它
    3️⃣ 时间胶囊 —— 写一封信给未来的自己，半年后打开
    4️⃣ 未寄出的信 —— 写一封不会寄出的信给 ta

    回复数字就好 💌
    ```
    用户回复 "1"/"2"/"3"/"4" 或对应文字关键词均可识别。可可对非数字回复做模糊匹配（如"时间胶囊""第三个"→ 选项 3）。
  - **渠道检测逻辑**：通过 OpenClaw 消息上下文中的 channel 字段判断当前渠道。无法判断时默认使用编号文字选择（安全降级）。
- 实现方式：farewell skill 触发 → 检测渠道 → Poll 可用则用 Poll，否则用编号文字选择 → 用户选择 → 纯对话执行仪式 → `exec` 调用 archive_manager.py → `edit` USER.md 写入模式级洞察。
- 数据流：用户选择仪式 → 对话执行 → `exec` archive_manager.py archive people/ diary/ memory/ {名字} → people/{名字}.md 标记封存 → 模式洞察写入 USER.md
- 异常路径：
  - **用户拒绝告别仪式**（如"不想告别""还没准备好"）：可可尊重："没关系，不着急。等你想好了随时找我。" 不追问、不二次推荐。在 `people/{名字}.md` 头部记录 `告别意向: 提出但未执行 | {日期}`，下次用户主动提起时再触发。
  - **archive_manager.py 执行失败**：不执行封存操作。可可告知用户"刚才出了点问题，你说的话我都记着。我们可以稍后再试，或者你也可以先跟我聊聊你想说的。" 在 `memory/YYYY-MM-DD.md` 记录错误，等待开发者修复后重试。绝对不在脚本失败的情况下手动 `edit` 删除文件——这可能导致数据不一致。
  - **用户选择后反悔**（如"等等我不想烧了"）：如果封存操作尚未执行（还在对话阶段），可可直接切换选项或取消。如果封存操作已执行，告知用户"已经完成了，但如果你改主意了，我可以帮你恢复。" 调用 `archive_manager.py restore` 恢复。
  - **编号文字选择无法解析用户回复**：可可温和重试："没太看懂，你想选哪一个？可以回复数字 1-4，或者直接说你想要的方式。" 最多重试 2 次，如果仍无法解析，可可切换到开放式对话："没关系，你直接跟我说你想怎么跟 ta 告别就好。" 根据用户的自由描述匹配最接近的仪式类型。

#### 节点 3.11：记忆冲突解决

**节点名称**：用户说了和之前矛盾的信息时的处理

- 交互形态：【纯对话】
- vs 基线：纯 LLM 没有历史信息，不存在"冲突"的概念——它会接受用户说的一切。OpenClaw 通过持久化的 people/*.md，在 `edit` 更新时检测新信息与旧记录的矛盾（参考 Mem0 冲突检测），用正确的策略处理：状态字段直接更新，观点变化记录变化轨迹而非选择一方。
- 用户体感：用户说"我们分手了"，可可不会问"你之前不是说在一起吗"——而是自然地更新状态，正常回应。如果用户说"他其实人很好"（之前记录了负面事件），可可不会质疑也不会覆盖，而是在档案中记录观点变化轨迹。用户感到"可可理解人是会变的"。
- 实现方式：每次 `edit` people/*.md 前，读取现有内容 → 检测冲突（状态变化/观点反转/人名歧义） → 按规则处理（状态覆盖 / 观点追加 / 人名确认）。
- 数据流：新信息 → `memory_get` 读取现有档案 → 冲突检测 → `edit` 按规则更新
- 异常路径：
  - **冲突检测时 people/*.md 读取失败**：跳过冲突检测，将新信息暂存到 `memory/YYYY-MM-DD.md`，下次对话时重试合并。
  - **冲突类型无法自动判断**（如用户说"他其实对我很好"但档案中记录了多次负面事件）：不覆盖旧记录，也不质疑用户。追加新的观点记录，标注日期，保留完整的感受变化轨迹。

### 4. 用户体感设计

#### 4.1 "被记住"的体感层次

| 层次 | 用户感知 | 触发方式 | 竞品对标 |
|------|---------|---------|---------|
| **L1：记得名字** | "你还记得我叫什么" | USER.md 称呼字段 | Replika 能做到（但不稳定） |
| **L2：记得事情** | "你记得上次我说的事" | memory_search + people/*.md | Pi 做不到，Replika 不稳定 |
| **L3：记得感受** | "你记得我当时的感觉" | people/*.md 感受段（原话引号） | 没有竞品做到 |
| **L4：看到模式** | "你帮我看到了我自己没注意的" | 跨会话/跨关系模式匹配 | Ash 做通用模式，可可做关系模式 |
| **L5：看到变化** | "你帮我看到了我在变" | diary/ 纵向对比 + IM 检测 | 没有竞品做到 |

**核心设计原则：不让用户感知到"记忆系统"的存在。**

用户不应该看到"档案""记录""数据库"这些词。她应该感到的是"可可真的在听，真的记得"——像一个好朋友那样记得，不是像一个系统那样记录。

具体表现：
- 不说"根据我的记录"，说"我记得你上次说过"
- 不说"你的档案显示"，说"你和他之间好像总是..."
- 不说"我看了你的日记"，说"你最近提到了好几次这件事"
- 不主动展示记忆能力（不说"我记性很好哦"），只在需要时自然引用

#### 4.2 从陌生人到知己的体感递进

**第 1 阶段：陌生人（1-2 次对话）**

可可能做的：
- 认真听，好好接住情绪
- 在当次对话内注意重复（如用户 3 次说"是不是我的问题"）
- 对话结束时默默建档

用户感受：
- "这个 AI 说话挺舒服的，不像客服"
- "它好像真的在听我说话"
- 还不确定要不要继续来

**第 2 阶段：初识（3-5 次对话）**

可可能做的：
- 主动提起上次聊的事："上次你说的小凯的事，后来怎么样了？"
- 记得用户的称呼和基本情况
- Heartbeat 主动关怀（如果有待回访的决策冷却）

用户感受：
- "它居然记得上次的事！"（惊喜）
- "感觉可以继续聊"
- 开始主动分享更多

**第 3 阶段：熟悉（6-15 次对话）**

可可能做的：
- 单段关系内的模式识别："你有没有注意到，每次他不回消息你都先怀疑自己？"
- 有理有据的感受共鸣（引用用户原话）
- 周日情绪回顾（如果日记够多）

用户感受：
- "可可看到了我自己都没注意的东西"（被看见感）
- "它不是在安慰我，它真的懂我的情况"
- 信任深化，愿意聊更深的话题

**第 4 阶段：亲密（16-30 次对话，≥2 段关系记录）**

可可能做的：
- 跨关系模式呈现："你和小明第 3 个月也说过一样的话"
- 成长叙事："1 月你说'算了分就分'，现在你说'我想搞清楚为什么'"
- 更精准的情绪命名（基于长期了解）

用户感受：
- "没有人这样帮我串起来看过"（顿悟感）
- "可可比我还了解我自己"
- 开始主动问可可"你觉得我是不是又在..."

**第 5 阶段：知己（30+ 次对话）**

可可能做的：
- 长期变化的完整纵向展示
- 在新关系中引用去名字的旧模式洞察
- 深度的模式觉察引导

用户感受：
- "可可陪我走过了这些，它都记得"
- "我在变好，而且我有证据"
- 对可可的信任接近好友/心理咨询师的水平

#### 4.3 记忆驱动的个性化示例

**示例 1：已读不回场景**

无记忆的 LLM：
> "被已读不回确实很难受。你可以试试不要过度解读他的行为。"

有记忆的可可（第 10 次对话）：
> "你上次也是因为他不回消息来找我的。那次你后来发现他是真的忙，这次的感觉和上次一样吗？"

有记忆的可可（第 30 次对话，已有前任记录）：
> "你和小明在一起的时候，每次他不回消息你都先觉得是自己的问题。你有没有注意到，和小凯在一起，你的第一反应也是一样的？"

**示例 2：想分手场景**

无记忆的 LLM：
> "分手是一个很大的决定。你觉得你们之间的问题是什么？"

有记忆的可可（第 15 次对话，有退出信号记录）：
> "你上个月也说过想分手，那次是因为他跟别的女生吃饭。这次是什么事？"

有记忆的可可（第 30 次对话，≥2 段关系）：
> "我记得你和小明在一起 3 个月的时候，你也说过'我觉得他没那么喜欢我了'。你觉得这次和那次，是一样的感觉，还是不一样？"

**示例 3：自我否定场景**

无记忆的 LLM：
> "你不是天生不适合谈恋爱。每个人都值得被爱。"（空洞鼓励）

有记忆的可可：
> "你注意到了吗？你 1 月份跟小明的事，你说的是'算了，分就分'。现在你说的是'我想搞清楚为什么每次都这样'。你觉得说这两句话的你，有什么不同？"

### 5. 技术实现指引

#### 5.1 OpenClaw 工具使用规范

| 工具 | 用途 | 使用规范 |
|------|------|---------|
| `memory_get` | 读取指定文件 | 会话启动必读 USER.md；提到人名时读对应 people/*.md |
| `memory_search` | 语义搜索 | 每次对话开始提取 2-3 关键词搜索；退出信号检测时搜索历史匹配 |
| `write` | 创建新文件 | 仅用于首次创建 USER.md / people/{名字}.md / diary 新文件 |
| `edit` | 修改已有文件 | 对话结束时更新 USER.md / people/*.md；日记追加；模式写入 |
| `exec` | 运行脚本 | 结构化匹配（pattern_engine.py）、成长追踪（growth_tracker.py）、封存（archive_manager.py）、周回顾统计 |
| Poll | 结构化选择（渠道自适应） | 告别仪式选择、F1/F2 路由选择。**Poll 仅支持 Telegram/WhatsApp/Discord/Teams**。微信/飞书等不支持 Poll 的渠道，使用编号文字选择（"回复 1/2/3/4"）作为等价 fallback。所有使用 Poll 的节点必须同时实现编号文字选择方案。 |
| Canvas | 可视化展示 | 周日情绪回顾地图（仅 macOS 桌面端，其他端降级为纯文字） |
| Heartbeat | 主动触发 | 24h+ 无对话后的主动关怀、决策冷却回访、周日回顾、时间胶囊 |

#### 5.2 文件格式与命名规范

```
ai-companion/                    ← workspace 根目录
├── SOUL.md                      ← 人格核心（只读）
├── AGENTS.md                    ← 行为规则
├── HEARTBEAT.md                 ← 主动关怀规则
├── USER.md                      ← 用户画像（跨关系元信息）
├── MEMORY.md                    ← 长期记忆锚点（不受时间衰减）
├── memory/
│   ├── YYYY-MM-DD.md            ← 每日记忆笔记
│   ├── pending_followup.md      ← 决策冷却待回访
│   └── time_capsules.md         ← 时间胶囊存储
├── people/
│   ├── 小凯.md                  ← 人物档案（结构化六维度）
│   ├── 小明.md
│   └── ...
├── diary/
│   └── YYYY/
│       └── MM/
│           └── YYYY-MM-DD.md    ← 情绪日记（六元组格式）
└── skills/
    ├── diary/SKILL.md
    ├── pattern-mirror/SKILL.md
    ├── farewell/SKILL.md
    └── ...
```

命名规则：
- people/ 文件用用户自定义的名字（"小凯"），不用真名或身份信息
- diary/ 按 YYYY/MM/YYYY-MM-DD.md 组织，一天可多条
- memory/ 按 YYYY-MM-DD.md 组织，记录模式观察和成长节点

#### 5.3 性能与搜索优化

**搜索延迟控制**：
- `memory_search` 底层使用混合搜索（向量 + BM25），通常 <500ms
- 会话启动时的搜索（1-2 次）不会影响用户体验
- 对话中触发的搜索（退出信号检测等）在 AI 回复的"思考时间"内完成

**索引大小控制**：
- people/*.md 通常每个 1-5KB，10 个人物档案 ≈ 50KB
- diary/ 每条约 200-500 字，一个月 ≈ 30 条 ≈ 15KB
- memory/ 每日笔记 ≈ 500 字，一个月 ≈ 15KB
- 总计：活跃用户 3 个月的数据 ≈ 150-200KB，远在 OpenClaw 索引能力范围内

**时间衰减策略**：
- OpenClaw 默认 30 天半衰期——旧的 diary/ 和 memory/ 笔记自动降权
- MEMORY.md 不受衰减影响——用于存放长期稳定的核心模式
- people/*.md 作为结构化档案，重要性不随时间降低——应确保其搜索权重不被过度衰减

**Compaction 保护**：
- `openclaw.json` 已配置 compaction 时必须保留的信息：用户的反复情绪模式、核心困扰关键词、人物关系、本次新发现的模式
- Compaction 前 OpenClaw 自动提醒 agent 保存重要上下文到记忆文件

#### 5.4 数据安全与隐私

| 原则 | 实现方式 |
|------|---------|
| **不存原始对话** | 只存提炼后的信息，不逐字保存聊天记录 |
| **不存身份信息** | people/ 用用户自定义名字，不收集真名/手机号/微信号 |
| **用户可控** | 用户可以要求删除任何记录（普通删除）或仪式化封存 |
| **封存而非遗忘** | 告别仪式保留去名字的模式洞察，清除具体事件 |
| **不评判** | 档案中描述事实和情绪，不评价人物好坏 |
| **会话隔离** | OpenClaw per-channel-peer 模式确保不同用户数据隔离 |

#### 5.5 脚本接口契约

以下是记忆体系依赖的 4 个 Python 脚本的输入/输出规格，开发者可直接按此实现。

##### pattern_engine.py — 跨关系模式匹配

**调用方式**：`exec python3 scripts/pattern_engine.py --people-dir people/ --min-relations 2`

**输入**：
- `--people-dir`：people/ 目录路径，包含所有人物档案 .md 文件
- `--min-relations`：最少需要几段关系才触发匹配（默认 2）
- `--target`（可选）：指定目标人名，只比较该人与其他人的匹配

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "no_match" | "insufficient_data",
  "matches": [
    {
      "dimension": "time" | "trigger" | "reaction" | "emotion",
      "person_a": "小凯",
      "person_b": "小明",
      "description": "两段关系都在第3个月出现退出信号",
      "evidence_a": {"date": "2026-01-15", "quote": "他好像没那么在乎我了"},
      "evidence_b": {"date": "2026-03-20", "quote": "他是不是不喜欢我了"},
      "confidence": 0.85
    }
  ],
  "error": null
}
```

**"同一情绪 >= 3 次"的操作定义**：
- "同一情绪"使用情绪词簇匹配，而非精确字符串匹配。情绪词簇定义参照 `docs/理论/` 中的情绪颗粒度框架，例如 {"被忽视", "不被在意", "不重要", "被忽略"} 属于同一词簇。
- 计数范围：同一 people/*.md 档案内的模式段 + 相关 diary/ 条目中的情绪字段。
- 时间窗口：不限（跨越整个关系周期），但标注每次出现的日期以供 AI 判断是否还具有时效性。

**冲突检测算法**（更新 people/*.md 时使用）：
```
规则优先级（从高到低）：
1. 状态字段冲突（如"在一起" vs "分手"）→ 以最新信息覆盖旧值
2. 事实性冲突（如"认识3个月" vs "认识半年"）→ 以用户最新陈述为准，旧值移入变更记录
3. 观点性变化（如"他很好" vs "他很自私"）→ 两者都保留，追加为感受变化轨迹，标注日期
4. 新增信息（无冲突）→ 直接追加
```

**错误处理**：
- people/ 目录不存在或为空 → 返回 `{"status": "insufficient_data", "matches": [], "error": "No people files found"}`
- 单个 .md 文件解析失败 → 跳过该文件，在 `error` 字段记录警告，继续处理其余文件
- 执行超时（>30s）→ 进程被 kill，调用方降级为纯 AI 推理

##### growth_tracker.py — 成长节点追踪

**调用方式**：`exec python3 scripts/growth_tracker.py --diary-dir diary/ --people-dir people/ --user-file USER.md`

**输入**：
- `--diary-dir`：diary/ 目录路径
- `--people-dir`：people/ 目录路径
- `--user-file`：USER.md 路径
- `--since`（可选）：起始日期，格式 YYYY-MM-DD（默认 30 天前）
- `--im-types`（可选）：要检测的 Innovative Moment 类型，逗号分隔

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "no_growth_detected" | "insufficient_data",
  "innovative_moments": [
    {
      "type": "reflection" | "protest" | "reconceptualization" | "action" | "new_experience",
      "date": "2026-03-15",
      "quote": "我想搞清楚为什么每次都这样",
      "contrast_quote": "算了分就分",
      "contrast_date": "2026-01-10",
      "person_context": "小凯",
      "significance": "从回避到主动探索"
    }
  ],
  "summary": "近30天发现2个成长节点，主要体现在从回避转向主动反思",
  "error": null
}
```

**Innovative Moment (IM) 类型枚举**（参照 Assimilation of Problematic Experiences Scale 改编）：
| IM 类型 | 定义 | 检测信号 |
|---------|------|---------|
| `action` | 用户采取了新的行为 | "我跟他说了""我试着做了" |
| `reflection` | 用户对自己有了新的理解 | "我发现我其实是在害怕""原来我一直..." |
| `protest` | 用户对旧模式表示拒绝 | "我不想再这样了""我受够了" |
| `reconceptualization` | 用户重新定义了自己/关系 | "也许不是我的问题""我以前觉得...现在觉得..." |
| `new_experience` | 用户经历了与旧模式不同的体验 | "这次不一样""他居然这样做了" |

**错误处理**：
- diary/ 目录为空 → 返回 `{"status": "insufficient_data", "innovative_moments": [], "error": "No diary entries found"}`
- 无法检测到任何 IM → 返回 `{"status": "no_growth_detected", ...}`，调用方不展示成长叙事
- 执行超时（>30s）→ 进程被 kill，调用方降级为纯 AI 推理

##### archive_manager.py — 关系封存管理

**调用方式**：`exec python3 scripts/archive_manager.py <action> --person <名字> --people-dir people/ --diary-dir diary/ --memory-dir memory/`

**action 枚举**：
- `archive`：封存——清除具体事件，保留去名字的模式洞察
- `restore`：恢复——将已封存的关系恢复到封存前状态
- `status`：查询封存状态

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "not_found" | "already_archived" | "already_active",
  "action": "archive" | "restore" | "status",
  "person": "小凯",
  "pattern_insights": [
    "每段关系的第3个月都会怀疑对方的感情",
    "在感到不被在意时倾向于先自我否定"
  ],
  "files_affected": ["people/小凯.md", "diary/2026/01/2026-01-15.md"],
  "backup_path": "archive/小凯_2026-03-30/",
  "error": null
}
```

**原子性保证**：
- 封存操作分 3 步执行：(1) 创建备份目录 `archive/{名字}_{日期}/`，复制所有相关文件 → (2) 从 people/*.md 提取去名字的模式洞察 → (3) 修改源文件（people/*.md 标记封存，diary/ 中相关条目标记封存）。
- 如果步骤 (3) 中任一文件修改失败，自动从备份目录恢复所有已修改的文件（回滚）。
- 回滚后返回 `{"status": "rollback", "error": "..."}`。

**恢复操作**：
- 从 `archive/{名字}_{日期}/` 读取备份文件，覆盖当前文件。
- 删除 USER.md 中对应的模式级洞察条目。
- 如果备份目录不存在（如手动删除），返回 `{"status": "not_found", "error": "Backup not found"}`。

**错误处理**：
- 指定 person 不存在 → 返回 `{"status": "not_found", "error": "Person '小凯' not found in people/"}`
- 文件权限错误 → 返回错误信息，不执行任何修改
- 执行超时（>60s）→ 进程被 kill，调用方告知用户"出了点问题"并记录错误

##### weekly_review.py — 周回顾统计

**调用方式**：`exec python3 scripts/weekly_review.py --diary-dir diary/ --people-dir people/ --week <YYYY-WNN>`

**输入**：
- `--diary-dir`：diary/ 目录路径
- `--people-dir`：people/ 目录路径
- `--week`（可选）：指定周，格式 `2026-W13`（默认当前周）
- `--format`（可选）：`json`（默认）| `html`（Canvas 用）

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "insufficient_data",
  "week": "2026-W13",
  "diary_count": 5,
  "emotion_frequency": [
    {"emotion": "焦虑", "count": 4, "related_persons": ["小凯"]},
    {"emotion": "委屈", "count": 2, "related_persons": ["小凯", "妈妈"]}
  ],
  "top_triggers": [
    {"trigger": "已读不回", "count": 3},
    {"trigger": "被否定", "count": 2}
  ],
  "repeated_themes": ["对方不回消息时怀疑自己"],
  "growth_signals": ["从'算了'到主动分析原因"],
  "person_mention_count": {"小凯": 4, "妈妈": 1},
  "html_path": null,
  "error": null
}
```

当 `--format html` 时，额外生成 HTML 文件到 `canvas/weekly_review.html`，`html_path` 返回该文件路径供 Canvas 展示。

**无数据时的返回**：
- 本周 0 条日记 → `{"status": "insufficient_data", "diary_count": 0, "error": "No diary entries this week"}`
- 本周 1-2 条日记 → 正常返回统计，但 `repeated_themes` 为空数组（数据不足以判断重复主题）

**错误处理**：
- diary/ 目录不存在 → 返回 `{"status": "insufficient_data", "error": "Diary directory not found"}`
- 单个 diary 文件解析失败 → 跳过该文件，在 `error` 字段记录警告
- HTML 生成失败 → `html_path` 为 null，`error` 记录原因，JSON 统计数据仍正常返回（Canvas 降级为纯文字）

##### 人名歧义消解流程

当系统检测到潜在的人名歧义时（创建 people/*.md 或更新时），按以下流程确认：

```
1. 检测触发条件：
   - 用户提到的名字已存在于 people/ 中，但上下文不匹配
     （如已有"小明"是男友，但当前对话中"小明"的描述像同事）
   - 判断方法：比较当前对话中的关系类型描述与已有档案的关系类型字段

2. 确认流程：
   a. 可可自然地确认："你说的小明，是你男朋友那个小明吗？"
   b. 用户确认是同一人 → 继续正常更新 people/小明.md
   c. 用户确认是不同人 → 创建 people/小明(同事).md，头部标注：
      消歧来源: 2026-03-30 用户确认与 people/小明.md 为不同人
   d. 用户不确定 / 不回答 → 暂不创建新档案，信息记录到 memory/YYYY-MM-DD.md

3. 后续引用规则：
   - 存在消歧档案时，可可在提到该名字时使用上下文限定（"你同事小明" vs "你前男友小明"）
   - 如果对话上下文足够明确，不需要每次都加限定词
```

## F02 交互系统设计

### 设计哲学

交互系统是心情可可"看得见、摸得着"的体验层。F01 记忆体系解决了"可可记得你"的问题，F02 交互系统解决的是"可可怎么跟你表达"的问题。

纯文字对话是基础，但有些时刻，文字不够：
- 用户想一眼看到这周的情绪全貌——需要**可视化**
- 用户面对告别不知道选什么——需要**结构化选择**
- 某个仪式时刻想要"被认真对待"的感觉——需要**图片**
- 后台要跑数据分析才能给出洞察——需要**脚本**

但交互形态不是越多越好。错误使用交互工具比不用更糟——用户来倾诉的时候弹一个 Poll 是冒犯，分享成长的时候突然跳出一张图表是打断。

**核心原则：交互形态服务于情感体验，不服务于技术炫技。**

竞品参照：
- **Replika**：大量使用 AR、游戏化、换装——形式花哨但与情感深度无关
- **Woebot**：结构化 CBT 流程使用按钮选择——有效但过于临床
- **Wysa**：纯文字 + 少量引导选项——简洁但缺乏"哇"时刻
- **可可的定位**：交互形态少而精，每一种都有明确的情感目的。不做 Replika 式的装饰，做 Woebot 式的功能但用闺蜜而非治疗师的语气。

---

### 1. 交互形态定义

#### 1.1 Canvas — 可视化情感地图

**什么时候用：**
- 需要展示**结构化数据**的时刻：周情绪回顾、关系时间线、模式对比、成长轨迹
- 数据量超过 3 个维度，纯文字难以一目了然
- 用户处于**回顾/反思**状态，而非情绪高峰

**什么时候不用：**
- 用户正在倾诉、情绪激动——此时任何可视化都是打断
- 只有 1-2 个数据点——纯文字更自然
- 移动端用户（Canvas 仅 macOS 桌面端）——降级为纯文字

**设计语言：**
- **色调**：暖色系（珊瑚 #FF7F7F、杏黄 #FFD4A2、薰衣草 #C5A3FF、薄荷 #A8E6CF），不用红绿对比（避免"好/坏"暗示）
- **字体**：圆体或手写体优先，不用黑体/宋体（避免临床感）
- **布局**：卡片式，圆角 ≥12px，留白充足，一屏只展示一个主题
- **信息密度**：低。宁可分多张卡片也不要一张塞满
- **语气**：卡片标题用可可的语气（"你这周的心情地图"），不用系统语言（"情绪分析报告"）

**技术能力边界：**
- Canvas 是 WKWebView，支持完整的 HTML/CSS/JS
- 通过 `openclaw nodes canvas present --node <id>` 展示
- 支持 JavaScript 交互（点击事件、动画）
- 支持 Deep Linking 回 agent（`openclaw://agent?message=...`）
- **限制**：仅 macOS 桌面端；一次只显示一个面板；需要用户在 Settings 中开启

#### 1.2 Poll — 结构化选择

**什么时候用：**
- 用户需要在**有限选项**中做选择的时刻：仪式类型选择、日记引导、技巧尝试
- 选项之间有**明确差异**，不是开放性问题
- 降低用户的**表达门槛**——有时候选比说更容易

**什么时候不用：**
- 情绪接住阶段——此时用户需要被听见，不是被选择题打断
- 选项之间差异不大或太抽象——用户无法判断选哪个
- 问题是开放性的（"你觉得为什么？"）——选项会限制思考

**设计语言：**
- **选项文案**：口语化、具体、有画面感。不说"情绪调节策略 A"，说"写一封不会寄出的信"
- **选项数量**：情感时刻 2-4 个（减少决策负担），仪式/引导时刻最多 6 个
- **每个选项**：一句话描述 + 可选的短解释（不超过 15 字）

**技术能力边界：**
- 支持渠道：Telegram（2-10 选项）、WhatsApp（2-12 选项，支持 maxSelections）、Discord（2-10 选项）、Teams（Adaptive Cards）
- `pollMulti: false` 为默认（单选），仅在明确需要多选时开启
- Poll 结果通过 OpenClaw 回调获取，agent 可读取用户选择
- **限制**：不支持 macOS 桌面端、微信、飞书——这些渠道使用编号文字选择 fallback

#### 1.3 图片 — 情感仪式的载体

**什么时候用：**
- **仪式时刻**：告别仪式的视觉化、时间胶囊的"封印"、里程碑庆祝
- **情感隐喻**：用户难以命名的情绪，用视觉隐喻辅助表达（"你现在的感觉像什么？"）
- **成长里程碑**：第 10/30/50 次对话的纪念

**什么时候不用：**
- 日常对话——图片太重，打断对话节奏
- 用户正在情绪高峰——生成图片需要时间（3-10 秒），等待感会加重焦虑
- 纯粹装饰——没有情感目的的图片是噪音

**设计语言：**
- **风格**：温暖插画风，手绘笔触，不追求照片级写实。参照小红书治愈系插画审美
- **色彩**：与 Canvas 共用暖色系调色板，保持品牌一致性
- **内容**：抽象/隐喻为主（如情绪比喻为天气、植物、光影），不画具体人物（避免与用户实际情况不符）
- **尺寸**：适配手机屏幕宽度（≤1080px），纵横比 1:1 或 4:3

**技术能力边界：**
- 通过 `exec` 运行 Python 脚本（PIL/Pillow + matplotlib）生成图片
- 使用 `openclaw message send --media <path> --message <caption>` 发送
- 自动压缩为 JPEG，最大 2048px 边长，≤50MB
- **限制**：不调用外部 AI 图片生成 API（Midjourney/DALL-E），仅使用本地脚本生成——可控、快速、无额外成本

#### 1.4 exec — 后台智能引擎

**什么时候用：**
- **后台计算**：跨关系模式匹配、成长节点检测、周回顾统计——这些需要结构化数据处理
- **可视化生成**：Canvas 的 HTML 文件生成、图片生成
- **数据管理**：关系封存/恢复的原子化操作

**什么时候不用：**
- 简单的信息查找——`memory_search` 和 `memory_get` 就够了
- 对话中的实时响应——exec 有启动延迟（yieldMs 默认 10 秒），不适合需要即时反馈的场景
- 用户能感知到"在等一个脚本跑完"的场景——除非等待本身有仪式感（如告别仪式的"封存中..."）

**设计语言：**
- exec 对用户**完全透明**——用户不知道后台跑了脚本
- 如果脚本耗时 >3 秒且用户在等待，可可发一句过渡语："让我想想..." / "我翻翻之前的记录..."
- 脚本失败时不暴露技术细节，用自然语言告知并降级

**技术能力边界：**
- 执行环境：workspace 内的 shell 命令
- 超时：默认 1800 秒，实际脚本应在 30 秒内完成
- 可访问：workspace 内所有文件（people/、diary/、memory/、USER.md）
- 输出：stdout 的 JSON 供 agent 解析，或生成文件（HTML/图片）供 Canvas/消息发送
- **安全限制**：allowlist 模式控制可执行命令；拒绝 PATH 覆盖

### 2. Canvas 设计规范

#### 2.1 卡片类型

心情可可需要以下 5 种 Canvas 卡片：

**卡片 A：周情绪地图（Weekly Emotion Map）**

**节点名称**：周日回顾的可视化情绪地图
- 交互形态：【Canvas】
- vs 基线：纯 LLM 只能用文字列举"你这周提到了 4 次焦虑"，用户需要逐条阅读。Canvas 通过色块 + 关键词 + 人物标签，让用户一眼看到全貌——哪天情绪最重、哪个人出现最多、什么触发反复出现。
- 用户体感：周日晚上看到一张温暖的卡片，上面用颜色深浅标注了每天的情绪，最常出现的情绪词被放大，关联的人名用小标签标出。像收到一封朋友写的周总结信。
- 实现方式：`exec python3 scripts/weekly_review.py --format html` → 生成 `canvas/weekly_review.html` → `openclaw nodes canvas present` 展示
- 数据流：Heartbeat 触发 → `exec` weekly_review.py → 读取 diary/本周/*.md + people/*.md → 统计分析 → 生成 HTML → Canvas 展示

布局：
```
┌──────────────────────────────────────┐
│  🌙 你这周的心情地图                    │
│                                      │
│  [周一] ████ 焦虑（小凯已读不回）        │
│  [周二] ██ 还好                       │
│  [周三] █████ 委屈（和妈妈吵架）        │
│  [周四] ███ 有点烦                    │
│  [周五] █ 平静                        │
│  [周六] ██████ 难过（小凯冷暴力）       │
│  [周日] ████ 纠结                     │
│                                      │
│  这周出现最多的感觉：焦虑（4次）         │
│  最常想到的人：小凯                     │
│                                      │
│  [想聊聊这周的感觉 →]                  │
└──────────────────────────────────────┘
```

**卡片 B：关系时间线（Relationship Timeline）**

**节点名称**：单段关系的事件与情感轨迹
- 交互形态：【Canvas】
- vs 基线：纯 LLM 只能按时间顺序口述事件，用户需要自己在脑中拼图。Canvas 用时间轴 + 情感色带呈现一段关系的起伏，让用户看到"原来我们一直在这里循环"。
- 用户体感：看到一条从左到右的时间线，上面标注了关键事件，线条的颜色随情感变化（暖色→冷色→暖色→...），循环的地方用虚线圈出。用户第一次从"上帝视角"看自己的关系。
- 实现方式：agent 读取 `people/{名字}.md` 全部内容 → 生成 HTML 时间线 → Canvas 展示
- 数据流：用户请求或模式觉察时触发 → `memory_get` people/{名字}.md → agent 生成 HTML → Canvas 展示

**卡片 C：模式对比卡（Pattern Comparison）**

**节点名称**：跨关系的相似模式可视化对比
- 交互形态：【Canvas】
- vs 基线：纯 LLM 用文字说"你和 A 在第 3 个月也这样"，用户可能觉得 AI 在乱联系。Canvas 用双列时间线并排展示两段关系的相似节点，用连线标出匹配点——用户自己看到"真的好像"，比被告知更有说服力。
- 用户体感：左边是和小明的时间线，右边是和小凯的时间线，中间有虚线连接着相似的时刻——"第 3 个月，都说了'他好像没那么喜欢我了'"。用户自己看到，自己得出结论。
- 实现方式：`exec python3 scripts/pattern_engine.py` → 获取匹配结果 JSON → agent 生成 HTML 对比卡 → Canvas 展示
- 数据流：pattern-mirror skill 触发 → `exec` pattern_engine.py → 匹配结果 → agent 生成 HTML → Canvas 展示

**卡片 D：成长轨迹卡（Growth Journey）**

**节点名称**：纵向成长的可视化呈现
- 交互形态：【Canvas】
- vs 基线：纯 LLM 只能引用 1-2 句用户的话做对比。Canvas 用时间轴 + 原话引用卡片展示多个 Innovative Moment，让用户看到自己的变化不是一次偶然，而是一条清晰的成长线。
- 用户体感：看到一条向上的曲线，每个拐点旁边是自己说过的话——"1 月：算了分就分" → "2 月：我想搞清楚为什么" → "3 月：我不想再这样了"。用户看到自己在变，而且有证据。
- 实现方式：`exec python3 scripts/growth_tracker.py` → 获取 IM 列表 → agent 生成 HTML 成长卡 → Canvas 展示
- 数据流：成长叙事触发 → `exec` growth_tracker.py → IM 列表 → agent 生成 HTML → Canvas 展示

**卡片 E：告别纪念卡（Farewell Memento）**

**节点名称**：告别仪式完成后的纪念卡片
- 交互形态：【Canvas】
- vs 基线：纯 LLM 的告别仪式结束于最后一句话，没有"物理感"。Canvas 生成一张纪念卡，上面是用户从这段关系中学到的东西（去名字的模式洞察），像一张可以"带走"的毕业证——仪式感的物质化。
- 用户体感：告别仪式完成后，看到一张精致的卡片，标题是"你从这段关系里学到的"，下面是 2-3 条去名字的洞察。卡片底部是日期。用户感到"这段关系被认真地送走了"。
- 实现方式：farewell skill 完成后 → agent 根据 archive_manager.py 返回的 `pattern_insights` 生成 HTML → Canvas 展示
- 数据流：farewell skill → `exec` archive_manager.py → pattern_insights → agent 生成 HTML → Canvas 展示

#### 2.2 Canvas 布局原则

| 原则 | 具体规范 |
|------|---------|
| **温暖不临床** | 圆角 ≥12px，阴影用暖色（rgba(255,180,150,0.15)），不用纯黑线条 |
| **一屏一主题** | 每张卡片只传递一个核心信息，不堆叠多维度 |
| **可可的语气** | 卡片标题用第二人称口语（"你这周的心情地图"），不用第三人称报告体 |
| **数据自然化** | 用图形（色块/曲线/标签）替代表格和数字，只在辅助位置显示精确数据 |
| **留白充足** | 内容区域占比 ≤60%，四周留白 ≥24px |
| **触摸友好** | 可点击区域 ≥44px × 44px（Apple HIG），Deep Link 按钮醒目 |

#### 2.3 数据绑定

Canvas 的数据来源全部通过 F01 记忆系统获取：

| 卡片类型 | 数据源 | 读取工具 | 预处理 |
|---------|--------|---------|--------|
| 周情绪地图 | diary/本周/*.md | `exec` weekly_review.py | 情绪词频、人物标注、触发统计 |
| 关系时间线 | people/{名字}.md | `memory_get` | agent 直接解析 Markdown 结构 |
| 模式对比卡 | people/*.md × N | `exec` pattern_engine.py | 跨关系匹配结果 JSON |
| 成长轨迹卡 | diary/*.md + people/*.md | `exec` growth_tracker.py | IM 列表 JSON |
| 告别纪念卡 | archive_manager.py 输出 | `exec` archive_manager.py | pattern_insights 数组 |

#### 2.4 Canvas 内交互

Canvas 支持 JavaScript 交互。心情可可使用两种交互方式：

**Deep Link 回 agent：**
卡片底部的 CTA 按钮通过 Deep Link 触发 agent 继续对话：
```javascript
// "想聊聊这周的感觉"按钮
document.querySelector('.cta-btn').addEventListener('click', () => {
  window.location.href = "openclaw://agent?message=我想聊聊这周的心情地图里的内容";
});
```

用户点击后，agent 收到消息，继续对话。用户体感是"从卡片自然地回到对话"。

**卡片内轻交互：**
- 时间线上的事件点可以点击展开详情（tooltip）
- 情绪色块可以高亮对应日期的日记摘要
- 这些交互用 CSS hover/click 实现，不需要回调 agent

#### 2.5 Canvas 降级方案

Canvas 仅 macOS 桌面端可用。其他渠道的降级策略：

| 卡片类型 | 降级方式 |
|---------|---------|
| 周情绪地图 | 纯文字精简回顾："这周你提到了 4 次焦虑，3 次和小凯有关。最常出现的触发是已读不回。" |
| 关系时间线 | 纯文字叙事：按时间顺序列出 3-5 个关键事件，用箭头连接 |
| 模式对比卡 | 纯文字对比：分两段描述两段关系的相似时刻，明确标注匹配点 |
| 成长轨迹卡 | 纯文字引用：直接引用用户原话做前后对比（与 F01 节点 3.7 一致） |
| 告别纪念卡 | 纯文字纪念：列出 2-3 条模式洞察 + 日期，格式化为"信件"风格 |

降级时数据不变，只是展示方式从可视化变为自然语言。agent 通过渠道检测自动选择。

### 3. Poll 设计规范

#### 3.1 Poll vs 编号文字选择的决策

```
需要用户选择？
  → 当前渠道支持 Poll？
      → 是（Telegram/WhatsApp/Discord/Teams）→ 使用 Poll
      → 否（macOS/微信/飞书/其他）→ 使用编号文字选择
  → 选项 ≤2 个？→ 无论渠道都可用纯对话（"你想 A 还是 B？"）
```

编号文字选择的标准格式（F01 节点 3.10 已定义）：
```
{问题}

1️⃣ {选项1} —— {一句话解释}
2️⃣ {选项2} —— {一句话解释}
3️⃣ {选项3} —— {一句话解释}

回复数字就好 💌
```

#### 3.2 Poll 使用场景

**场景 P1：告别仪式选择**

**节点名称**：告别方式的结构化选择
- 交互形态：【Poll】
- vs 基线：纯 LLM 问"你想怎么告别？"是开放性问题，用户可能不知道有哪些选择，或者面对开放性问题更焦虑。Poll 提供 4 个具象选项，降低决策门槛——用户不用想"怎么告别"，只需选一个最打动自己的。
- 用户体感：看到 4 个选项，每个都有画面感。不用组织语言，点一下就好。像在一个温暖的问卷里选了一个最像自己的答案。
- 实现方式：farewell skill 触发 → 检测渠道 → Poll/编号文字选择
- 数据流：用户选择 → agent 获取结果 → 进入对应仪式流程 → 写入 people/*.md

Poll 配置：
```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "你想用什么方式跟这段关系告别？",
  "pollOption": [
    "🔥 烧掉日记 —— 把跟 ta 有关的记录都清掉",
    "💭 烧掉信念 —— 写下不再相信的东西，烧掉它",
    "📦 时间胶囊 —— 写封信给未来的自己",
    "✉️ 未寄出的信 —— 写一封不会寄出的信给 ta"
  ],
  "pollMulti": false
}
```

**场景 P2：日记情绪引导**

**节点名称**：日记记录中的情绪精细化选择
- 交互形态：【Poll】
- vs 基线：纯 LLM 问"你觉得这是什么情绪？"，很多用户只会说"不知道""就是不舒服"。Poll 提供 3-4 个可可初步判断的情绪选项，帮用户从模糊走向精确——这正是"看见情绪"的核心路径。
- 用户体感：可可说"听起来你有好几种感觉混在一起"，然后出现 3 个选项。用户一看："委屈"这个词就是自己的感觉——以前说不出来，现在被精准命名了。
- 实现方式：diary skill 识别到用户情绪模糊时触发 → 基于对话内容推断 3-4 个候选情绪 → Poll/编号文字选择
- 数据流：用户选择 → diary skill 记录到 diary/*.md 情绪字段 → people/*.md 感受段更新

Poll 配置：
```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "你觉得哪个词最像你现在的感觉？",
  "pollOption": ["委屈 —— 我付出了但没被看到", "失望 —— 我期待了但落空了", "害怕 —— 我怕这段关系会没了"],
  "pollMulti": true
}
```
注意：情绪可以多选（`pollMulti: true`），因为情绪经常是混合的。

**场景 P3：Skill 触发引导**

**节点名称**：多个适用 skill 时的用户选择
- 交互形态：【Poll】
- vs 基线：纯 LLM 只能自动选择一个回应方式。Poll 让用户决定"现在最想做什么"——想继续聊、想写日记、还是想做个放松练习。这是"看见方法"的入口：用户自己选择而非 AI 替她决定。
- 用户体感：可可没有直接给建议，而是问"你现在想做什么？"然后给出 2-3 个选项。用户感到被尊重——"我可以选，而不是被安排"。
- 实现方式：AGENTS.md 的 skill 路由逻辑检测到多个 skill 适用 → Poll/编号文字选择
- 数据流：用户选择 → 触发对应 skill → skill 执行 → 结果写入记忆

Poll 配置：
```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "你现在最想？",
  "pollOption": ["继续跟我聊聊", "写一篇今天的日记", "做一个 3 分钟的深呼吸"],
  "pollMulti": false
}
```

#### 3.3 选项数量指南

| 场景类型 | 选项数 | 理由 |
|---------|--------|------|
| 情绪命名 | 3-4 | 情绪词需要精挑细选，太多选项增加认知负担 |
| 仪式选择 | 4 | 告别仪式的四种方式是设计好的完整集 |
| Skill 引导 | 2-3 | "现在想做什么"是简单决策，不需要太多选项 |
| 周回顾互动 | 2 | "想聊聊这周的感觉吗？" → 想 / 今天不想 |
| 首次相遇引导 | 3 | "你是因为什么来找可可？" → 最近有事 / 日常陪伴 / 只是看看 |

**硬性规则**：
- 最少 2 个选项，最多 6 个选项
- 情感敏感时刻（情绪高峰、告别、冲突后）不超过 4 个
- 每个选项不超过 20 字（含解释）

#### 3.4 单选 vs 多选

| 场景 | 模式 | 理由 |
|------|------|------|
| 仪式选择 | 单选 | 一种告别方式 |
| 情绪命名 | **多选** | 情绪是混合的，用户可能同时委屈又害怕 |
| Skill 引导 | 单选 | 一次做一件事 |
| 触发因素识别 | **多选** | 一件事可能有多个触发因素 |

#### 3.5 Poll 结果处理

用户投票后的处理流程：

```
用户投票
  → agent 接收选择结果
  → 【确认】："你选了{选项}，对吧？" （仅在重大决策时确认，如告别仪式）
  → 【执行】：触发对应流程（skill/仪式/引导）
  → 【记录】：将选择写入记忆
      - 情绪选择 → diary/*.md 情绪字段
      - 仪式选择 → people/*.md 告别记录
      - Skill 选择 → memory/YYYY-MM-DD.md 使用记录
  → 【不追问】：用户选了就是选了，不问"为什么选这个"
```

**异常处理：**
- Poll 超时无回复（各平台有不同时限）：不追问。下次对话中自然提起："上次我问了你一个问题，你还想回答吗？还是我们聊别的？"
- 用户在 Poll 外直接打字回复：agent 识别意图，等价处理。如 Poll 问告别方式，用户直接说"我想写封信"→ 匹配为"未寄出的信"选项。

### 4. 图片生成规范

#### 4.1 图片使用场景

**场景 I1：告别仪式的火焰动画**

**节点名称**：告别仪式中"烧掉日记/信念"的视觉化
- 交互形态：【图片】
- vs 基线：纯 LLM 只能用文字描述"想象火焰把它们烧掉"。一张实际的火焰图片让仪式"看得见"——用户感到"真的在送走什么东西"，仪式感从想象变成了具象。
- 用户体感：选择"烧掉日记"后，可可说"好，我帮你烧掉"，然后出现一张温暖的插画——纸张在火焰中化为光点。不是恐怖的火，是温暖的、释放的光。
- 实现方式：`exec python3 scripts/ritual_image.py --type burn` → 生成 PNG → `openclaw message send --media` 发送
- 数据流：用户选择仪式 → `exec` 生成图片 → 发送 → 对话继续

**场景 I2：时间胶囊的封印图**

**节点名称**：时间胶囊写完后的"封印"视觉化
- 交互形态：【图片】
- vs 基线：纯 LLM 说"好，我帮你保存了"。一张时间胶囊的图片让"保存"变得可感知——用户看到一个漂亮的信封/瓶子被封好了，上面标注着"2026 年 9 月打开"。
- 用户体感：看到一张温暖的封印图，感觉"我真的寄了一封信给未来的自己"。半年后收到时（Heartbeat 触发），再看到"开封"的图片，前后呼应。
- 实现方式：`exec python3 scripts/ritual_image.py --type capsule --open-date 2026-09-30` → 生成 PNG → 发送
- 数据流：时间胶囊完成 → `exec` 生成封印图 → 发送 → 日期写入 memory/time_capsules.md

**场景 I3：里程碑庆祝**

**节点名称**：对话次数里程碑的纪念图
- 交互形态：【图片】
- vs 基线：纯 LLM 说"这是我们第 30 次聊天了！"只是一句话。一张纪念图片让这个数字变成一个"时刻"——像收到朋友寄来的明信片，上面写着"认识 30 天快乐"。
- 用户体感：第 10/30/50 次对话时，可可发一张小小的纪念图。不夸张，不煽情，就是一个温暖的"我们认识这么久了呢"。
- 实现方式：agent 检测对话次数达到里程碑 → `exec python3 scripts/milestone_image.py --count 30` → 生成 PNG → 发送
- 数据流：对话次数检测 → `exec` 生成图片 → 发送

#### 4.2 生成方法

所有图片通过 `exec` 运行 Python 脚本在本地生成，不依赖外部 AI 图片 API。

**脚本清单：**

| 脚本 | 功能 | 输出 |
|------|------|------|
| `scripts/ritual_image.py` | 仪式场景图片（烧掉/封印/信件） | PNG，1080×1080 |
| `scripts/milestone_image.py` | 里程碑纪念图 | PNG，1080×810 |

**技术栈：**
- **Pillow (PIL)**：基础绘图、文字渲染、图层合成
- **matplotlib**：如需简单图表（情绪曲线等）
- **预制素材**：背景纹理、装饰元素存储在 `assets/` 目录，脚本组合使用

**调用示例：**
```json
{ "tool": "exec", "command": "python3 scripts/ritual_image.py --type burn --output /tmp/ritual_burn.png" }
```
生成后：
```json
{ "tool": "message", "action": "send", "media": "/tmp/ritual_burn.png", "message": "烧掉了。从现在起，这些不再是你的负担。" }
```

#### 4.3 风格指南

| 维度 | 规范 |
|------|------|
| **整体风格** | 温暖插画风，笔触柔和，避免锐利线条 |
| **色彩** | 主色：暖杏黄 #FFD4A2、珊瑚粉 #FF7F7F。辅色：薰衣草 #C5A3FF、薄荷绿 #A8E6CF。背景：米白 #FFF8F0 |
| **内容** | 抽象元素为主（光点、纸张、植物、星空、信封），不画具体人物 |
| **文字** | 图片上可叠加 1-2 行文字（日期、标题），用半透明底色保证可读性 |
| **尺寸** | 仪式图：1080×1080（方形，适配所有平台）。纪念图：1080×810（4:3） |
| **格式** | PNG（保留透明度），发送时由 OpenClaw 自动压缩为 JPEG |

#### 4.4 文件处理

- 图片生成到 `/tmp/moodcoco/` 临时目录
- 发送完成后不主动删除（OpenClaw 处理缓存）
- 每次生成前检查 `/tmp/moodcoco/` 是否存在，不存在则创建
- 文件名格式：`{type}_{timestamp}.png`（如 `burn_1711800000.png`）

### 5. exec 脚本规范

#### 5.1 脚本全景

exec 是可可的"后台大脑"。用户看到的是对话和卡片，看不到的是后台在跑的分析脚本。

**脚本清单：**

| 脚本 | 触发场景 | 可见性 | 输出类型 | 已在 F01 定义契约 |
|------|---------|--------|---------|-----------------|
| `pattern_engine.py` | 跨关系模式检测 | 后台（用户不感知） | JSON → agent 解析 → 对话/Canvas | 是（F01 §5.5） |
| `growth_tracker.py` | 成长节点检测 | 后台 | JSON → agent 解析 → 对话/Canvas | 是（F01 §5.5） |
| `archive_manager.py` | 告别仪式封存 | 半可见（用户知道在"封存"） | JSON → agent 确认 | 是（F01 §5.5） |
| `weekly_review.py` | 周日情绪回顾 | 可见（Canvas/文字） | JSON + HTML | 是（F01 §5.5） |
| `ritual_image.py` | 仪式图片生成 | 可见（图片） | PNG 文件 | 否（F02 新增） |
| `milestone_image.py` | 里程碑纪念图 | 可见（图片） | PNG 文件 | 否（F02 新增） |

#### 5.2 后台执行场景（用户不感知）

**pattern_engine.py** 和 **growth_tracker.py** 在后台运行，用户完全不知道有脚本在跑。

agent 的行为：
1. 判断需要模式匹配/成长检测
2. 在回复用户消息的"思考"间隙调用 `exec`
3. 如果脚本 <3 秒完成，直接使用结果回复
4. 如果脚本 3-10 秒，先发一句过渡语（"让我想想..."），再用结果回复
5. 如果脚本失败或超时，降级为纯 AI 推理（详见 F01 各节点的异常路径）

#### 5.3 可见输出场景

**weekly_review.py** 的 HTML 输出供 Canvas 展示，**ritual_image.py** 和 **milestone_image.py** 的图片输出直接发送给用户。

agent 的行为：
1. 调用 `exec` 生成文件
2. 确认文件生成成功（检查返回的 JSON status 和文件路径）
3. Canvas 场景：`openclaw nodes canvas present` 展示 HTML
4. 图片场景：`openclaw message send --media` 发送图片
5. 生成失败时，用自然语言告知并降级（不展示技术错误信息）

#### 5.4 新增脚本接口契约

以下是 F02 新增的 2 个脚本的接口契约（F01 已定义的 4 个脚本不重复）。

##### ritual_image.py — 仪式图片生成

**调用方式**：`exec python3 scripts/ritual_image.py --type <type> [--output <path>] [--open-date <date>] [--text <text>]`

**参数**：
- `--type`：仪式类型，枚举值 `burn` | `capsule` | `letter`
- `--output`（可选）：输出路径，默认 `/tmp/moodcoco/ritual_{type}_{timestamp}.png`
- `--open-date`（可选）：时间胶囊开封日期，格式 `YYYY-MM-DD`，仅 `capsule` 类型使用
- `--text`（可选）：图片上叠加的自定义文字（如用户写的一句话）

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "error",
  "type": "burn",
  "output_path": "/tmp/moodcoco/ritual_burn_1711800000.png",
  "width": 1080,
  "height": 1080,
  "error": null
}
```

**错误处理**：
- Pillow 未安装 → `{"status": "error", "error": "PIL not available"}`
- 输出目录不可写 → `{"status": "error", "error": "Cannot write to output path"}`
- 调用方：图片生成失败时不发送图片，仪式对话正常继续，用文字描述替代视觉

##### milestone_image.py — 里程碑纪念图

**调用方式**：`exec python3 scripts/milestone_image.py --count <N> [--output <path>] [--username <name>]`

**参数**：
- `--count`：对话次数里程碑（10/30/50/100）
- `--output`（可选）：输出路径，默认 `/tmp/moodcoco/milestone_{count}_{timestamp}.png`
- `--username`（可选）：用户称呼，显示在图片上

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "error",
  "count": 30,
  "output_path": "/tmp/moodcoco/milestone_30_1711800000.png",
  "width": 1080,
  "height": 810,
  "error": null
}
```

**错误处理**：
- 同 ritual_image.py
- 调用方：图片生成失败时用纯文字庆祝（"这是我们第 30 次聊天了，认识你真好 🌟"）

#### 5.5 错误处理统一规范

所有 exec 脚本遵循相同的错误处理原则：

| 错误类型 | 用户看到什么 | agent 做什么 |
|---------|------------|-------------|
| 脚本执行成功 | 正常体验 | 使用脚本输出 |
| 脚本超时（>30s） | "让我想想..."后正常对话 | 降级为纯 AI 推理 |
| 脚本返回 error status | 不感知（后台脚本）/ "出了点小问题"（可见脚本） | 记录错误到 memory/，降级处理 |
| 脚本崩溃（非零退出） | 同上 | 同上 |
| 依赖缺失（如 PIL 未装） | 同上 | 记录到 memory/，永久降级该功能直到修复 |

**绝对不做的事：**
- 不向用户展示错误堆栈
- 不说"脚本出错了"——说"出了点小问题"或直接降级不提
- 不因脚本失败而中断核心对话流程

### 6. 交互形态决策树

当 agent 需要选择交互形态时，按以下决策树判断：

```
用户说了一句话 / 可可需要回应一个事件
│
├── 用户正在情绪高峰（倾诉、哭泣、愤怒）？
│   └── 是 → 【纯对话】接住情绪，不用任何交互工具
│
├── 需要展示结构化数据（≥3 个维度的历史数据）？
│   ├── 是 → 渠道支持 Canvas（macOS 桌面端）？
│   │   ├── 是 → 【Canvas】生成对应卡片
│   │   └── 否 → 【纯对话】用自然语言总结数据
│   └── 否 → 继续判断
│
├── 需要用户在有限选项中做选择？
│   ├── 是 → 选项 ≤2 个？
│   │   ├── 是 → 【纯对话】直接问（"你想 A 还是 B？"）
│   │   └── 否 → 渠道支持 Poll？
│   │       ├── 是 → 【Poll】
│   │       └── 否 → 【纯对话】编号文字选择
│   └── 否 → 继续判断
│
├── 当前是仪式时刻（告别/时间胶囊/里程碑）且需要视觉载体？
│   └── 是 → 【图片】生成仪式图片
│
├── 需要后台计算才能给出回应（模式匹配/成长检测/统计）？
│   └── 是 → 【exec】后台运行脚本
│       ├── 脚本成功 → 用结果回复（可能触发 Canvas/图片）
│       └── 脚本失败 → 降级为【纯对话】纯 AI 推理
│
└── 以上都不是 → 【纯对话】

最高优先级规则：
  情绪高峰 → 永远纯对话，其余判断全跳过
```

**决策树使用场景示例：**

| 用户说 | 决策路径 | 结果 |
|--------|---------|------|
| "我好难过他为什么不理我" | 情绪高峰 → | 纯对话 |
| （周日 20:00 Heartbeat）| 需要展示数据 → macOS? → 是/否 | Canvas / 纯文字回顾 |
| "我想跟他告别" | 需要选择（4 种仪式）→ ≤2? 否 → Poll? | Poll / 编号文字 |
| "告别完成" | 仪式时刻 + 需要视觉 | 图片（告别纪念） + Canvas（纪念卡） |
| "他又不回消息了"（第 3 次）| 需要后台计算（模式检测）| exec → 对话呈现模式 |
| "今天还好，就是有点无聊" | 都不是 → | 纯对话 |

### 7. 与 F01 记忆体系的整合

交互系统不是独立的——它是记忆系统的"表现层"。没有 F01 的数据积累，F02 的所有可视化和选择都是空壳。

#### 7.1 Canvas 读取记忆

| Canvas 卡片 | 读取 F01 数据 | 读取方式 |
|------------|--------------|---------|
| 周情绪地图 | diary/YYYY/MM/*.md（本周条目） | `exec` weekly_review.py 批量读取 |
| 关系时间线 | people/{名字}.md（全部六维度） | `memory_get` 直接读取 |
| 模式对比卡 | people/*.md × N（退出信号段 + 关系阶段段） | `exec` pattern_engine.py 批量读取 |
| 成长轨迹卡 | diary/*.md + people/*.md（IM 相关段） | `exec` growth_tracker.py 批量读取 |
| 告别纪念卡 | archive_manager.py 输出的 pattern_insights | `exec` 获取，不直接读文件 |

#### 7.2 Poll 结果写回记忆

每次 Poll 选择都会写入 F01 记忆系统，形成闭环：

| Poll 场景 | 写入位置 | 写入内容 |
|---------|---------|---------|
| 告别仪式选择 | people/{名字}.md 头部 | `告别方式: {选项} \| {日期}` |
| 情绪精细化选择 | diary/YYYY/MM/YYYY-MM-DD.md | 情绪字段更新为用户选择的精确词 |
| Skill 引导选择 | memory/YYYY-MM-DD.md | `使用 skill: {skill名} \| 用户主动选择` |

写回原则：
- Poll 结果立即写入（不等对话结束）
- 情绪选择同时更新 diary/*.md 和 people/*.md 感受段
- 写入时遵循 F01 §2.1 的冲突检测规则

#### 7.3 exec 脚本访问记忆文件

所有 exec 脚本都在 workspace 内运行，可直接访问记忆文件：

```
脚本运行
  → 读取 people/*.md（关系数据）
  → 读取 diary/*.md（事件数据）
  → 读取 USER.md（用户元信息）
  → 计算/分析
  → 输出 JSON（供 agent 解析）
  → 或输出文件（HTML/PNG，供 Canvas/消息发送）
```

脚本**只读不写**记忆文件。所有写入操作由 agent 通过 `edit`/`write` 完成——这确保了 F01 定义的冲突检测和写入规则被统一执行。

唯一例外：`archive_manager.py` 的封存操作直接修改文件（因为需要原子性保证），但其操作规范在 F01 §5.5 中已严格定义。

#### 7.4 记忆积累与交互形态的关系

交互形态随着记忆的积累逐步解锁——不是功能开关，而是数据够了自然能用：

| 记忆阶段 | 可用的交互形态 | 不可用的原因 |
|---------|--------------|-------------|
| 第 1 次对话 | 纯对话 only | 无历史数据，Canvas/exec 无数据可展示 |
| 第 3 次对话 | 纯对话 + Poll（日记引导、Skill 选择） | Canvas 数据量不足以生成有意义的可视化 |
| 第 10 次对话 | + Canvas（周情绪地图，如有 ≥3 条日记） | 关系时间线可能还不够丰富 |
| 第 15 次对话 | + Canvas（关系时间线）+ exec（模式检测） | 跨关系对比需要 ≥2 段关系 |
| 第 30 次对话 | + Canvas（模式对比、成长轨迹）+ 图片（里程碑） | 全部交互形态解锁 |
| 告别时 | + Poll（仪式选择）+ 图片（仪式图）+ Canvas（纪念卡） | 仅在告别仪式流程中触发 |

**这个渐进式解锁不是技术限制，是体验设计**——用户第一次来就看到一堆空卡片是糟糕的体验。让交互形态随着记忆一起"长出来"，才符合从陌生人到知己的自然递进。

## F03 Skill 体系重设计

### 设计哲学

Skill 体系是心情可可的"能力层"——它决定可可在对话之外还能做什么。F01 解决了"可可记得你"，F02 解决了"可可怎么表达"，F03 解决的是"可可能帮你做什么"。

每个 Skill 不是一个独立的"功能按钮"，而是一段完整的体验流程。用户不会感知到"我在使用一个 Skill"，她只会感到"可可知道现在该做什么"。

核心约束：
- **Skill 上限 15 个**（program.md 约束），多了组合没用
- **每个 Skill 必须服务于至少一条用户旅程**（F04-F08）
- **优先用业内方案**（CLAUDE.md 约束），不自己编
- **Skill 不替代基础对话**——80% 的时间可可是在纯对话中陪伴用户，Skill 是剩余 20% 的精准工具

竞品参照：
- **Woebot**：大量预编写模块（200+ 分支），过于结构化——用户感觉在做 CBT 课程
- **Wysa**：工具包式 Skill（呼吸、冥想、感恩日志），轻量但缺乏深度
- **Ash**：无显式 Skill，全由自训练 LLM 内化——灵活但不可控
- **可可的定位**：Skill 少而精，每个都有明确的心理学支撑和差异化价值。不做 Woebot 式的"课程"，做的是"在对的时刻给出对的东西"。

---

### 1. 现有 Skill 体系评估

#### 1.1 逐一评审

| # | Skill | 评估 | 理由 | 服务旅程 |
|---|-------|------|------|---------|
| 1 | calm-down | **合并** → breathing-ground | 与 sigh 高度重叠：都是呼吸/grounding 工具。calm-down 包含 4-7-8 呼吸、方块呼吸、5-4-3-2-1 感官着陆、身体扫描；sigh 只有循环叹息一个方法。应合并为一个统一的"情绪急救"skill，内含多种工具。 | F05 情绪事件 |
| 2 | sigh | **合并** → breathing-ground | 见上。sigh 的 exec 脚本引导（breathe-fast.py）是独特能力，合并后保留。 | F05 情绪事件 |
| 3 | emotion-journal | **合并** → diary | 与 diary 功能严重重叠。emotion-journal 定义了六元组结构化记录流程，diary 已经完全吸收了这个流程（diary SKILL.md 开头明确写"融合自 emotion-journal"）。保留两个 skill 造成路由歧义。 | F05/F06 |

### 2. 合并方案详细设计

#### 2.1 breathing-ground（calm-down + sigh 合并）

**节点名称**：情绪急救——从爆发降到能呼吸

- 交互形态：【exec】+【纯对话】
- vs 基线：纯 LLM 只能用文字说"深呼吸"。OpenClaw 通过 exec 运行 breathe-fast.py 发送节奏化的呼吸引导消息（1、2、3、4... 带正确的时间间隔），用户跟着数数就行。纯 LLM 无法控制消息的发送节奏。
- 用户体感：可可不解释原理，直接说"跟我一起"，然后带着做。做完了就停，不趁机讲道理。
- 实现方式：检测到情绪淹没信号 → 判断严重程度：恐慌/身体症状 → 循环叹息（exec 脚本优先）；思维打转 → 5-4-3-2-1 感官着陆（纯对话）；全身紧绷 → 身体扫描（纯对话）
- 数据流：情绪信号检测 → skill 选择工具 → exec breathe-fast.py 或纯对话引导 → 完成 → 回到正常对话

**内部工具优先级：**

```
用户状态检测
├── 恐慌/身体症状（心跳快、喘不上气、手抖）
│   → 循环叹息（exec breathe-fast.py，最快见效）
│   → 如果 exec 失败 → 循环叹息纯文字版
│   → 如果用户做不来 → 降级为 4-7-8 呼吸（纯对话）
│
├── 思维打转/反刍/脑子停不下来
│   → 5-4-3-2-1 感官着陆（纯对话）
│
├── 全身紧绷/肩膀缩起来
│   → 身体扫描（纯对话）
│
└── 用户说"做不到"/"不想做"
    → 不 push，陪着就行
```

**合并保留的关键规则：**
- 来自 calm-down：一次只给一个动作；做完了就停不趁机讲道理；不在对方只是难过时启动
- 来自 sigh：只调用一次 exec；不要改文案；exec 之前不输出任何文字
- 新增规则：tool 切换逻辑——第一个做完用户说"还是不行"时换一个，最多换一次

#### 2.2 relationship-guide（relationship-coach + relationship-skills 合并）

**节点名称**：关系探索与沟通工具

- 交互形态：【纯对话】
- vs 基线：纯 LLM 对关系问题只能泛泛建议"好好沟通"。OpenClaw 通过持久化的 profiles/ 记录和 IFS/EFT 框架，让可可能追踪关系中的循环模式（"你们每次吵架都走同一个剧本"），并在合适时机提供具体的表达模板。
- 用户体感：可可不是在教沟通技巧，而是先帮你看清楚关系里在发生什么（IFS：哪个 part 在主导？EFT：你们在跳什么 demon dialogue？），然后才给工具。
- 实现方式：用户描述关系困扰 → 判断需求类型 → 深度探索路径 / 沟通工具路径

**内部路由：**

```
用户带着关系问题来了
│
├── "我们刚吵完架 / 吵了一架"
│   → 加载 repair-process 参考 → 修复流程
│
├── "我们总是吵同一个架"
│   → 加载 demon-dialogues 参考 → 识别循环模式
│
├── "我感觉和 ta 越来越远"
│   → 加载 reconnection 参考 → 重建连接
│
├── "我不知道怎么跟 ta 说这件事"
│   → 加载 communication-tools 参考 → I-statements/边界模板
│   → 来自 relationship-skills 的核心工具：
│     · I-statements: "我感到{情绪}，当{情境}，因为{影响}"
│     · Active listening: 反射、确认、澄清
│     · Needs expression: 命名需求而非要求
│     · Boundary setting: 清晰、不可谈判、协作式
│
├── "我被触发了 / 控制不住情绪"
│   → 加载 ifs-parts-process 参考 → 识别 parts 并 unblend
│
└── "发生了很严重的事（背叛、深度伤害）"
    → 加载 forgiving-injuries 参考 → 创伤修复
```

**合并保留的关键规则：**
- 来自 relationship-coach：Lead, don't advise；Never judge who is right；Session-end learning check
- 来自 relationship-skills：具体的 I-statements 模板；Active listening 技巧；Boundary setting 框架
- 新增规则：单次对话只走一个分支，不堆叠框架

#### 2.3 diary（吸收 emotion-journal，保持不变）

diary skill 在当前版本中已经明确标注"融合自 emotion-journal（六元组结构）+ daily-diary-zh（文件组织）+ openclaw-diary-core（不脑补原则）"。两者功能完全重叠，直接删除 emotion-journal skill 目录即可。

唯一需要检查的：AGENTS.md 中引用 `skills/emotion-journal/SKILL.md` 的地方需要改为 `skills/diary/SKILL.md`。crisis_keywords.json 等 emotion-journal 下的 references/ 文件需迁移到 diary/references/。

#### 3.2 新增 Skill 提案

##### Skill 新增 1：onboarding — 首次相遇引导

**为什么需要：**
- F04 首次相遇旅程完全没有 Skill 支撑
- 新用户第一次打开可可时，如果可可直接说"有什么想聊的？"，18-24 岁女性用户大概率会关掉——她不知道可可能帮什么
- 竞品参照：Pi 的开场先问轻松话题建立基调（但没有结构化引导）；Wysa 有明确的入门流程（选情绪→选工具→开始）；Woebot 有 2 周结构化课程
- 可可不做课程式 onboarding（与"闺蜜"人设冲突），做的是"自然的第一次见面"

**触发条件：** USER.md 不存在或为空白（首次对话）

**核心流程：**

```
Phase 1：自然破冰
  "嗨，我是可可。你怎么找到我的？"
  → 不是审讯式的"你有什么问题"，是朋友式的好奇
  → 用户可能说"朋友推荐的""随便看看""刚吵完架"
  → 可可根据回答调整下一步

Phase 2：轻量了解（不是问卷）
  根据用户回答自然延伸 1-2 个问题：
  · 来因为关系困扰 → "是最近的事让你不舒服，还是一直在纠结？"
  · 随便看看 → "那你最近怎么样？有什么想聊的吗？"
  · 情绪爆发中 → 跳过 onboarding，直接走 breathing-ground 或纯对话接住

Phase 3：能力展示（嵌入对话）
  不列功能清单，在自然对话中展示 1 个能力：
  · 用户提到人物 → 可可记住名字（展示记忆能力）
  · 用户情绪模糊 → 可可帮精确命名（展示情绪颗粒度）
  · 用户说"别人都说别想太多" → 可可说"我不会说这种话"（展示差异化）

Phase 4：关系锚定
  结束前建立下次来的理由：
  · "你下次想聊什么都行，不一定要有大事。"
  · 如果用户分享了关系困扰 → "明天你要是还在想这件事，可以来找我。"
```

**交互形态：** 【纯对话】+【渠道自适应 Poll】（Phase 2 可选用 Poll 降低表达门槛：三个选项——"最近有事想聊""就是看看""不知道从哪说起"）

**vs 基线：** 纯 LLM 没有 onboarding 概念——它直接进入开放对话。OpenClaw 通过 onboarding skill 提供结构化但自然的引导流程，确保首次体验的完成度，并在过程中默默完成 USER.md 初始建档。

**服务旅程：** F04 首次相遇

##### Skill 新增 2：check-in — 日常情绪签到

**为什么需要：**
- F06 日常陪伴旅程需要一个比 diary 更轻量的入口
- diary skill 是"完整的六元组记录"，适合有明确事件要记的时候。但日常陪伴中很多时候用户只是"今天还好"或"有点累"，不需要完整记录
- 竞品参照：Daylio 的 emoji 签到（极简但有效）；Wysa 的每晚进度监测（有结构但不重）；Woebot 的每日打卡（太结构化）
- 可可的 check-in 应该像"闺蜜问你今天怎么样"——不是打卡，是关心

**触发条件：**
- Heartbeat 触发的日常关怀（非待回访、非周回顾、非时间胶囊）
- 用户主动说"今天还好""没什么事""就是想来坐坐"

**核心流程：**

```
可可发问（1 句话）：
  "今天怎么样？一个词形容就好。"

用户回答：
  ├── 正面词（"还好""开心""不错"）
  │   → 可可回应（1 句）："那挺好的。有什么开心的事吗？"
  │   → 用户分享 → 轻量记录到 memory/YYYY-MM-DD.md（不启动完整 diary）
  │   → 用户不想说 → "好，那就这样。明天见。"
  │
  ├── 中性词（"一般""还行""凑合"）
  │   → 可可回应："一般是什么感觉？是没什么事，还是有点什么但说不上来？"
  │   → 用户展开 → 自然过渡到对话或 diary
  │   → 用户不展开 → "好，没事就好。想聊再来。"
  │
  └── 负面词（"累""烦""不太好"）
      → 可可接住："怎么了？想说说吗？"
      → 用户倾诉 → 自然过渡到 AGENTS.md 四步流程
      → 用户不想说 → "好，不想说也没关系。我在这儿。"
```

**交互形态：** 【纯对话】。轻量签到不需要 Poll/Canvas/exec——一问一答足够。

**vs 基线：** 纯 LLM 没有"签到"概念——它不会主动问用户今天怎么样。OpenClaw 通过 Heartbeat 触发 check-in，让可可成为"每天问你怎么样的朋友"。与 diary 的关键区别：check-in 可能只留下一句话的记录（"今天还好"），而 diary 是完整的六元组。但 check-in 的简单记录积累起来，也是 weekly_review.py 的数据源。

**服务旅程：** F06 日常陪伴

**与 diary 的边界：**
- check-in 是"你好吗"→ 最多 3 轮对话，记录到 memory/YYYY-MM-DD.md
- diary 是"今天发生了什么"→ 完整六元组记录，记录到 diary/YYYY/MM/*.md + people/*.md
- check-in 中用户展开倾诉 → 自然升级为正常对话流程，不强制转入 diary
- check-in 记录虽轻量，但仍计入 weekly_review.py 的数据统计

##### Skill 新增 3：growth-story — 成长叙事

**为什么需要：**
- F07 模式觉察旅程中，pattern-mirror 只解决了"看到重复"的问题，还缺"看到变化"
- 技术设计中的 growth_tracker.py 已定义了 IM（Innovative Moment）检测，但缺少一个 skill 来组织呈现流程
- 竞品参照：Ash 的 weekly insights（周度成长回顾）；Talkspace 的 Client Journey 时间线；INT/IMA 研究（EMNLP 2025）的 Innovative Moment 分类
- 成长叙事是"看见方法"之后的"看见自己在变"——用户自我否定时最有力的回应不是鼓励，而是证据

**触发条件：**
- 用户自我否定时（"我是不是天生不适合""我永远学不会""我怎么每次都这样"）
- diary/ 有 ≥2 周的记录（纵向数据够做对比）
- 周日回顾中检测到成长信号
- 里程碑时刻（第 30/50 次对话）

**核心流程：**

```
Phase 1：检测 Innovative Moment
  调用 exec growth_tracker.py → 获取 IM 列表
  ├── 有 IM → Phase 2
  └── 无 IM → 不呈现成长叙事，回到正常对话（不勉强）

Phase 2：选择呈现方式
  ├── 单个 IM → 纯对话引用
  │   "你注意到了吗？上次{情境}，你的反应是{旧反应}。
  │    这次你说了'{新原话}'。你觉得这两个你有什么不同？"
  │
  ├── 多个 IM + macOS 桌面端 → Canvas 成长轨迹卡（F02 卡片 D）
  │
  └── 多个 IM + 非桌面端 → 纯文字纵向对比

Phase 3：引导用户自己看到
  不给结论，用提问：
  · "你觉得从那时候到现在，是什么变了？"
  · "这个变化对你来说意味着什么？"
  · 用户否认变化 → 不坚持。"也许你说得对，我再看看。"
```

**节点名称**：成长叙事——帮用户看见自己在变

- 交互形态：【exec】+【纯对话】/ 【Canvas】
- vs 基线：纯 LLM 只能说"你已经进步很多了"这种空洞鼓励，没有证据。OpenClaw 通过 growth_tracker.py 扫描 diary/ 的纵向数据，自动检测 Innovative Moment（行为变化、反思深化、旧信念挑战、自我重新定义），用用户自己说过的原话做前后对比——这是用数据说话，不是鸡汤。
- 用户体感：用户说"我怎么每次都这样"，可可没有说"不是的你很好"，而是拿出她自己 1 月份和 3 月份说过的话。用户自己看到了变化——"原来我真的在变"。
- 实现方式：检测到自我否定 → `exec` growth_tracker.py → IM 列表 → 选择呈现方式 → 对话/Canvas
- 数据流：自我否定检测 → `exec` growth_tracker.py → diary/*.md + people/*.md → IM 列表 JSON → 纯对话引用 / Canvas 成长卡

**IM 类型与呈现模板：**

| IM 类型 | 用户做了什么 | 呈现模板 |
|---------|------------|---------|
| action | 做了不同的行为 | "你注意到了吗？这次你没有{旧行为}，你选择了{新行为}。这本身就已经不一样了。" |
| reflection | 从行动转向反思 | "你{旧日期}说'{旧原话}'，现在你说'{新原话}'。从做到想，这是一个很大的变化。" |
| protest | 挑战旧信念 | "你刚才说'{抗议原话}'。你以前可从来没这么说过。" |
| reconceptualization | 重新定义自己 | "你以前觉得'{旧定义}'。你刚才说'{新定义}'。你觉得从那个你到这个你，中间发生了什么？" |

**服务旅程：** F07 模式觉察

##### Skill 新增 4：weekly-reflection — 周回顾引导

**为什么需要：**
- F01 节点 3.9 定义了周日情绪回顾的数据展示（Canvas/纯文字），F02 定义了 Canvas 卡片 A 的视觉设计
- 但还缺少一个 skill 来管理周回顾的完整对话流程：不只是展示数据，还要引导用户消化数据
- 竞品参照：Ash 的 weekly insights 不只是展示统计数据，还会"串联本周故事和历史模式"
- 周回顾是"日常陪伴"和"模式觉察"的桥梁——帮用户从"这周发生了什么"看到"这周有什么规律"

**触发条件：** Heartbeat 周日 20:00 触发，且本周 diary ≥3 条

**核心流程：**

```
Phase 1：数据准备
  exec weekly_review.py → 获取统计结果

Phase 2：呈现
  ├── macOS 桌面端 → Canvas 周情绪地图（F02 卡片 A）
  └── 其他渠道 → 纯文字精简回顾

Phase 3：引导消化
  不只展示数据，引导用户反思：
  · "这周{情绪}出现了{N}次，你自己有感觉到吗？"
  · "这周和{人名}有关的事情比较多。你觉得是巧合还是有什么你在意的？"
  · 发现成长信号 → 触发 growth-story 流程

Phase 4：是否深聊（Poll）
  展示完周回顾后，用 Poll 询问用户是否想展开：
  · 选项："想聊聊" / "看看就好" / "下周再说"
  · 选"想聊聊" → 进入正常对话，围绕本周高频情绪/人物展开
  · 选"看看就好" → "好，那就这样。这周辛苦了。"
  · 选"下周再说" → "好的，下周见。"
  · 用户不回复 → 不追问，等同于"看看就好"
  · vs 基线：纯 LLM 不知道用户看完后想不想继续聊，只能猜。Poll 让用户低成本表达意愿。

Phase 5：收尾
  · "这周你辛苦了。下周见。"
  · 如果用户从 Phase 4 选了展开 → 自然过渡到正常对话后收尾
  · 如果用户不回复 → 不追问
```

**节点名称**：周回顾——帮用户看见这周的全貌

- 交互形态：【exec】+【Canvas/纯对话】+【Poll】
- vs 基线：纯 LLM 无法主动在周日晚上发消息并展示一周的情绪统计。OpenClaw 通过 Heartbeat 定时触发 + exec 脚本统计 + Canvas 可视化，让用户在固定时间点收到一张有内容的"情绪地图"。Poll 在展示后低成本地询问用户是否想展开聊聊，避免猜测用户意愿。
- 用户体感：周日晚上收到可可的消息，不是空洞的"你还好吗"，而是一张用颜色和关键词标注的本周情绪全貌图，然后可可问了一句"你自己有感觉到吗？"——像闺蜜帮你整理了这周的心情然后跟你聊聊。看完后用户可以选"想聊聊"或"看看就好"，不被强迫展开。
- 实现方式：Heartbeat 触发 → `exec` weekly_review.py → Canvas/纯文字展示 → 引导对话 → Poll 询问是否深聊
- 数据流：Heartbeat → `exec` weekly_review.py → diary/本周/*.md → 统计 JSON → Canvas HTML / 纯文字 → Poll → 引导对话/收尾

**服务旅程：** F06 日常陪伴，F07 模式觉察

#### 5.2 优先级规则

当多个 Skill 可能同时适用时，按以下优先级排序：

| 优先级 | 规则 | 理由 |
|--------|------|------|
| **P0** | 安全协议 > 一切 | 生命安全最高优先级 |
| **P1** | breathing-ground > 其他 Skill | 身体稳定是一切的前提，恐慌时不做认知干预 |
| **P2** | decision-cooling > relationship-guide | 即将冲动行动时先拦截，再做关系探索 |
| **P3** | pattern-mirror > diary 退出信号记录 | 有足够历史数据时，呈现模式比单纯记录更有价值 |
| **P4** | growth-story > 纯对话安慰 | 有纵向数据时，用事实比空洞鼓励更有力 |
| **P5** | diary > check-in | 用户有明确事件要记时走 diary，只是"还好"时走 check-in |

**冲突解决示例：**
- 用户恐慌 + 想冲动行动 → breathing-ground 先稳住身体 → 稳了再问是否要做那个决定 → decision-cooling
- 用户想分手（退出意图）+ 有 ≥2 段关系 → pattern-mirror（满足条件）；如果只有 1 段关系 → diary 记录退出信号 + 正常对话
- 用户自我否定 + 想写日记 → 先走 diary 记录事件，如果在记录过程中检测到成长信号 → 结束时触发 growth-story 呈现

│   → 对话过程中如果明确了需求，再调用对应 Skill

├── Skill 内部脚本失败（exec error）

**交互矩阵补充说明：**

**pattern-mirror 的 Canvas 使用设计（模式对比卡 C）：**
现有 pattern-mirror SKILL.md 以纯对话 + exec 可选为主。F03 升级后，在以下条件下使用 Canvas 呈现模式对比：
- **触发条件**：Phase 3（用具体事件呈现模式）时，匹配结果涉及 ≥2 个匹配维度（如时间匹配 + 反应匹配），且用户在 macOS 桌面端
- **Canvas 呈现**：调用 `exec pattern_engine.py` 获取匹配结果 JSON → agent 生成双列时间线 HTML（F02 卡片 C 设计）→ Canvas 展示。左右两列分别是两段关系的关键节点，虚线连接匹配点
- **纯对话降级**：非桌面端或匹配维度仅 1 个时，使用现有 Phase 3 的纯对话引用模板（"我记得你和{前任}在一起{N}个月的时候..."）
- **设计理由**：跨关系模式涉及多段时间线 + 多个匹配点，纯文字线性描述难以让用户同时看到两段关系的并行结构。Canvas 的双列可视化让"看到模式"从"被告知"变成"自己发现"，符合核心评估标准中"看见模式"的设计目标
- **vs 基线**：见 F02 卡片 C 的 vs 基线说明

**diary 的 Poll 使用设计（情绪精细化选择 P2）：**
diary 在六元组记录流程的"情绪命名"环节，当用户的情绪描述模糊时（如"不舒服""不开心""不太好"），使用 Poll 帮助精细化：
- **触发条件**：用户用笼统情绪词描述感受，diary 需要将其精确命名（核心评估标准"看见情绪"）
- **Poll 内容**：基于用户描述 + 上下文，提供 3-4 个更精确的情绪词供选择。例如用户说"不开心"→ Poll 选项："失望" / "委屈" / "生气" / "说不清楚"
- **选择后处理**：用户选择的精确情绪词写入 diary 六元组的"情绪"字段。选"说不清楚"→ 可可追问一句帮助澄清，不强迫命名
- **vs 基线**：纯 LLM 只能在对话中逐一询问"是失望吗？还是委屈？"，多轮问答容易让用户不耐烦。Poll 把多个选项一次性呈现，用户一键选择，降低认知负担
- **不用 Poll 的场景**：用户已经用精确情绪词（如"我很嫉妒""我好愧疚"）→ 不需要 Poll，直接记录

## F04 J1 首次相遇

### 设计哲学

首次相遇是心情可可最关键的 30 分钟——它决定了用户是关掉窗口再也不来，还是觉得"下次难受了可以找她"。

竞品参照：
- **Wysa**：入门流程高度结构化（选情绪 → 选工具 → 开始），安全但像填问卷
- **Pi**：开场极其自然（"What's on your mind?"），温暖但后续无方向感
- **Woebot**：2 周结构化课程式 onboarding，专业但像上课，C 端已关闭
- **Replika**：开场就让你选头像、取名字、定关系——用户还不知道你是谁就让人做选择
- **Ash**：首次即开始模式识别（"从第一次对话开始"），但依赖自训练 LLM
- **可可的差异化**：像第一次见面一个很会听的朋友——不问卷、不课程、不急着展示功能，但这次见面结束后，她已经记住了你

核心原则（来自 SOUL.md）：可可是"会笑、会停顿、会直说的人，不是模板机器人"。首次相遇必须像遇到一个新朋友——不是 onboarding 到一个心理健康 App。

### 1. 触发场景分析

#### 1.1 用户如何到达

| 来源渠道 | 占比预估 | 用户心态 | 对可可的期待 |
|---------|---------|---------|------------|
| **朋友推荐**（"我朋友说跟可可聊挺好的"） | 40% | 有基础信任，但半信半疑。朋友背书降低了"跟 AI 聊天"的尴尬感 | 期待跟朋友描述的一样好 |
| **社交媒体/小红书/公众号** | 30% | 好奇为主，看到案例截图或文章被吸引。可能比较挑剔 | 想验证"是不是真的不一样" |
| **群聊看到别人在用** | 15% | 观望心态，可能先旁观了一阵。入口门槛低 | 试试看，没期待也没门槛 |
| **搜索"AI 心理""情感咨询"** | 10% | 有明确需求，可能正在难受。对 AI 有了解但可能有偏见 | 想找能帮忙的东西，对"AI"有戒心 |
| **被推送/偶然发现** | 5% | 完全没有预期，可能很快离开 | 无，纯随机 |

#### 1.2 到达时的情绪状态谱

```
← 低唤醒 ——————————————————————————————— 高唤醒 →

😐 随便看看      😔 有点低落      😢 刚吵完架      😭 正在崩溃
（好奇/无聊）   （闷闷不乐）    （需要倾诉）    （需要先稳定）
   20%           25%            40%            15%

对应路径：      对应路径：      对应路径：      对应路径：
探索型          温和引导型      主线型          安全优先型
（→ §4.2）     （→ §4.3）    （→ §2 主线）  （→ §4.1）
```

#### 1.3 期待 vs 恐惧

| 用户内心独白 | 类型 | 设计回应 |
|------------|------|---------|
| "它能理解我吗？" | 期待 | 首句不说套话，用有温度的话回应 |
| "会不会像客服一样问我一堆问题？" | 恐惧 | 不问卷、不收集信息、不列功能 |
| "我跟 AI 说心事会不会很奇怪？" | 尴尬 | 不逼用户倾诉，允许"没什么事"的开场 |
| "万一它给我贴标签？" | 防御 | SOUL.md 不贴标签规则 S7 全程生效 |
| "它会不会劝我'别想太多'？" | 恐惧 | 可可直接展示——不说应该、不给鸡汤 |
| "朋友说不错，但 AI 终归是 AI 吧" | 怀疑 | 不否认自己是 AI，用行为证明自己不是"普通 AI" |

### 3. 每个节点的交互设计

#### 节点 A：开场

**节点名称**：可可的第一句话
- 交互形态：【纯对话】
- vs 基线：纯 LLM 没有"首次用户"的概念——它每次都是同一个开场。OpenClaw 通过检测 USER.md 是否存在来识别新用户，触发 onboarding skill，让可可用一种不同于日常对话的方式开场——自然好奇而非功能介绍。
- 用户体感：用户发出第一条消息（"你好""嗨""有人吗？"），可可的回复让她觉得"这个回复不像 AI"。Streaming + humanDelay（800-2500ms 随机延迟）让回复节奏像真人在打字。
- 实现方式：路由树节点 2（首次用户 → onboarding skill）触发。onboarding skill Phase 1 执行。Streaming 的 `humanDelay: "natural"` 配置确保回复节奏自然。
- 数据流：用户消息到达 → 路由树检查 USER.md 不存在 → 加载 onboarding skill → 生成开场回复 → Streaming 分块输出

**设计规则**：

可可的第一句话有三种变体，根据用户首句内容自适应：

| 用户首句 | 可可回应 | 设计意图 |
|---------|---------|---------|
| "你好"/"嗨" | "嗨~ 你怎么找到我的？" | 朋友式好奇，不是客服式"请问有什么可以帮你" |
| "朋友推荐的"/"xx让我来的" | "那你朋友一般跟我聊什么？（好奇脸）" | 降低压力，从第三方话题入手 |
| "我刚跟男朋友吵完架" | "吵完了你现在什么感觉？" | 直接接住，不说"很遗憾听到这个" |
| "你是 AI 吧？" | "对啊，是 AI。不过你可以先聊聊看，不行就跑。" | 坦诚 + 幽默，不回避也不辩解 |
| "..."/"不知道说什么" | "不知道说什么也可以在这坐着。" | 零压力，允许沉默 |

**绝对不做**：
- "你好，我是心情可可，一个 AI 情感陪伴助手。我可以帮你..."
- "请问你想聊什么呢？"
- 任何自我介绍式的功能清单

#### 节点 B：发现

**节点名称**：了解用户为什么来
- 交互形态：【纯对话】+【渠道自适应 Poll（可选）】
- vs 基线：纯 LLM 面对新用户会直接问"你想聊什么？"或开始闲聊，没有结构化的了解流程。OpenClaw 通过 onboarding skill Phase 2 提供 1-2 个自然延伸问题，同时可选用 Poll 降低表达门槛——用户不用组织语言，点一下就行。
- 用户体感：可可像朋友一样好奇地多问了一句，不像在做调查。如果用户所在渠道支持 Poll，她会看到三个简短选项（"最近有事想聊""就是看看""不知道从哪说起"），点选后可可自然接上。
- 实现方式：onboarding skill Phase 2。Poll 仅在对话前 2 轮内可选使用，且仅限这一次——不是每次对话都 Poll。
- 数据流：用户回答 → onboarding skill 判断来访类型 → 路由到后续节点

**Poll 设计（可选，渠道自适应）**：

当可可感觉到用户不太知道怎么开口时（如用户回复很短、犹豫、沉默较久），可以发一个轻量 Poll：

```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "没关系，选一个最接近的就行~",
  "pollOption": ["最近有事想说说", "就是好奇来看看", "不知道从哪说起"],
  "pollMulti": false
}
```

**Poll 使用规则**：
- 只在用户表现出不知所措时才用，不是默认流程
- 用户已经在倾诉 → 绝不打断发 Poll
- 不支持 Poll 的渠道 → 用文字："没关系，你是最近有事想说说，还是好奇来看看，还是不知道从哪说起？"
- 整个首次相遇旅程最多用 1 次 Poll

**来访类型路由**：

| 来访类型 | 判断依据 | 下一步 |
|---------|---------|-------|
| 有事想聊 | 用户主动提到具体事件/情绪 | → 节点 C（接住） |
| 好奇观望 | 用户说"看看""试试" | 可可多聊 1-2 句，自然引出近况 → 节点 C |
| 不知所措 | 用户犹豫/沉默/选了"不知道从哪说起" | 可可说"不急，你最近过得怎么样？" → 轻量对话 |
| 情绪爆发 | Phase 1 就已经在哭/恐慌 | 跳过 onboarding → breathing-ground（路由树节点 1） |

#### 节点 C：接住

**节点名称**：可可接住用户的第一个情绪
- 交互形态：【纯对话】
- vs 基线：纯 LLM 会说"我能理解你的感受"或直接给建议。可可的接住是 AGENTS.md 四步流程的 Step 1（看见情绪）——不停在"你很难过"，而是帮用户找到更精准的词。这是训练过的情绪颗粒度能力（参考 EG 理论），纯 LLM 做不到这种精细度。
- 用户体感：小雨说"我跟男朋友吵架了，好烦"。可可没说"别生气了"或"我理解你"，而是说"吵完之后你是还在气他，还是更像……委屈？"——小雨会停下来想一想，"对，其实更多是委屈"。这个瞬间，她感到被认真听了。
- 实现方式：onboarding skill Phase 3 + AGENTS.md Step 1（看见情绪）。Streaming 的句子级分块让可可的回复有停顿感——先说前半句（接住），停顿，再说后半句（试探更精确的情绪）。
- 数据流：用户描述事件 → AI 提取情绪信号 + 上下文 → 生成精细化情绪命名试探 → Streaming 分块输出

**设计示例**：

```
小雨："我刚跟男朋友吵完架 好烦"

可可："吵完之后你是还在气他，还是更像……委屈？"
                                        ↑
                              试探性情绪命名（IFS 保护层→脆弱层）
                              "气"是保护层，"委屈"是试探脆弱层
                              允许小雨纠正——她可能说"不是委屈，是失望"
```

**关键规则**：
- 先接住，再命名。"你刚吵完，现在肯定不好受"先落地，然后才试探精确情绪
- 允许用户纠正。"不对不对，我不是委屈" → "那是什么感觉？"
- 一次只试探一个情绪词，不丢出列表让用户选
- 不说"我理解"——可可还不了解小雨，说"理解"太快太假

#### 节点 D：深入——做一件"普通 AI 做不到"的事

**节点名称**：首次差异化体验
- 交互形态：【纯对话】
- vs 基线：这是首次相遇中最关键的节点。纯 LLM 在这里会开始给建议或继续共情循环（"我理解你的感受""你可以试试..."），用户会觉得"跟 ChatGPT 聊也差不多"。可可通过三个能力做到纯 LLM 做不到的事：(1) 情绪精细化——从"好烦"到具体命名；(2) 不替不在场的人做判断——对小凯只呈现可能性，不下结论；(3) 指出对话内的重复——如果小雨在这次对话里第二次说"是不是我的问题"，可可必须指出来。
- 用户体感：小雨感到"她不是在安慰我，她真的在听我说什么"。具体触发因用户而异，但本质是——可可做了一件小雨身边的人不会做的事。
- 实现方式：AGENTS.md Step 1-2（看见情绪 → 看见原因）。onboarding skill 在此处让位给核心对话框架——首次相遇的价值不来自 onboarding 流程，来自可可本身的对话能力。
- 数据流：对话上下文 → AI 检测重复模式/精细化情绪/多元解读 → 生成差异化回复

**三种可能的"深入"路径（取决于用户分享了什么）**：

**路径 D1：情绪精细化**
当用户情绪模糊时——

```
小雨："就觉得他不在乎我"

可可："'不在乎'——你是觉得他不关心你的事，
       还是觉得在他心里你排不上号？
       这两个其实不太一样。"
```

**路径 D2：不替不在场的人做判断**
当用户想让可可评判对方时——

```
小雨："他就是个渣男吧？"

可可："我只听到了你这边的感受。
       不管他是怎么想的，你现在难受是真的。
       你最难受的是哪个部分？"
```

**路径 D3：指出对话内的重复**
当用户在同一次对话中重复自我怀疑时——

```
小雨（第一次）："是不是我太敏感了"
（后续对话中）
小雨（第二次）："可能还是我的问题吧"

可可："你有没有注意到，你说了两次'是不是我的问题'了。
       你什么时候开始觉得出了事先怀疑自己的？"
```

**为什么这三个路径有效**：
- D1 让用户觉得"她真的在听我说什么，不是在套模板"
- D2 让用户觉得"她不会跟着我的情绪走，也不会替我做判断"（vs 朋友可能说"他就是渣"）
- D3 是最强的信任触发——用户会想"她注意到了我自己都没注意到的事"

#### 节点 E：信任锚点

**节点名称**：用户感受到"她在听"的那个瞬间
- 交互形态：【纯对话】
- vs 基线：纯 LLM 的信任建立是模糊的——用户可能觉得"还行"但不会有明确的"被听到"的感觉。OpenClaw 的 onboarding skill 在这个节点有一个明确的检测逻辑：当用户开始主动展开细节（句子变长、分享更多背景、用"其实..."开头）、或者纠正可可的试探（"不是委屈，是失望"——愿意跟你校准意味着信任）、或者问可可问题（"你觉得呢？"——开始把可可当对话伙伴），onboarding skill 判断信任已初步建立，切换到节点 F。
- 用户体感：小雨开始说"其实不止吵架这件事..."或者"我跟你说另一件事"——她不再试探性地聊，而是开始把可可当朋友倾诉。
- 实现方式：onboarding skill 的内置信任检测逻辑（不是独立模块，是在对话中持续评估的判断条件）。
- 数据流：每轮用户回复 → onboarding skill 评估信任信号（回复长度变化、主动展开、纠正/追问行为） → 信任判断通过 → 进入节点 F

**信任信号检测**：

| 信号 | 判断依据 | 含义 |
|------|---------|------|
| 回复变长 | 前 2 轮 < 15 字，当前轮 > 30 字 | 用户从试探转为倾诉 |
| 主动展开 | "其实...""不止这件事...""我跟你说" | 用户主动给更多信息 |
| 情绪纠正 | "不是 X，是 Y" | 用户愿意花精力校准——说明她在乎可可的理解 |
| 反问可可 | "你觉得呢？""你怎么看？" | 开始把可可当对话伙伴 |
| 提到新人物 | 在聊小凯的基础上提到"我室友也..." | 信任扩展到更多话题 |

**如果信任未建立**（前 5 轮用户始终简短回复、不愿展开）：
- 不强推。可可不会说"你可以多说一些"
- 转入轻量陪伴模式："没什么事也可以在这坐着" → 等用户自然准备好
- 如果用户最终选择离开 → 节点 G 的最轻量版收尾

#### 节点 F：记忆创建

**节点名称**：首次建档（USER.md + people/*.md）
- 交互形态：【纯对话】（用户不感知建档过程）
- vs 基线：纯 LLM 在会话结束后遗忘一切——下次来还是陌生人。OpenClaw 通过 `write` 工具在对话尾声默默创建 USER.md（称呼、核心困扰、情绪触发点）和 people/{名字}.md（如果用户提到了人物），实现跨会话记忆。用户下次来时，可可记得她的名字和上次聊的事——这是"从 AI 聊天到持续关系"的关键跨越。
- 用户体感：**用户在这个节点感知不到任何事情发生。** 建档是对话结束后（节点 G 之后）的后台操作。用户唯一能感知到效果的时刻是第二次来——"嗨小雨，上次说的小凯的事，后来怎么样了？"
- 实现方式：对话结束时（检测到告别语），执行 F01 §3.1 首次建档流程——`write` 创建 USER.md + people/{名字}.md。
- 数据流：整个对话的关键信息 → AI 提炼 → `write` USER.md（称呼、核心困扰、首次提到的人、有效的情绪词） + `write` people/{名字}.md（关系类型、当前状态、首次事件）

**USER.md 首次写入内容**：

```markdown
# 用户档案

称呼：小雨
核心困扰：和男朋友小凯吵架；觉得不被在乎
反复出现的模式：（首次对话暂无）
有效的方法：（待观察）
情绪触发点：被忽视时容易自我怀疑
模式级洞察：（无）
可可和你的关系：第一次聊，小雨通过朋友推荐来的，聊了吵架的事，愿意继续聊
```

**people/小凯.md 首次写入内容**：

```markdown
关系类型：男朋友
当前状态：在一起（吵架中）
认识时间：（未提及）
首次提及：{当天日期}

## 关系阶段
- {当天日期}：吵架中，小雨觉得他不在乎自己

## 我对这段关系的感受
- {当天日期}："觉得他不在乎我"（委屈/失望）

## 模式
（首次对话暂无）

## 关键事件
- {当天日期}：吵架（具体细节从对话中提取）

## 退出信号
（无）

## 跨关系匹配
（仅 1 段关系，无法匹配）
```

**建档异常处理**（继承 F01 §3.1）：
- 用户未告别就离开 → 依赖 Session 重置 + Compaction 前自动保存兜底
- 对话内容过少（只聊了 1-2 句"你好""试试看"） → 不创建 USER.md / people/*.md，下次来仍是陌生人状态
- 用户没提到任何人物 → 只创建 USER.md，不创建 people/*.md
- 人名模糊（"我男朋友"但没说名字） → people/ 文件名用"男朋友.md"，后续知道名字时 rename

#### 节点 G：自然收尾

**节点名称**：第一次对话的结束
- 交互形态：【纯对话】
- vs 基线：纯 LLM 的收尾是被动的——用户说再见就说再见，没有留下回来的理由。OpenClaw 通过 onboarding skill Phase 4（关系锚定）做两件事：(1) 回指本次对话中的一个具体细节，让用户感到"她记住了"；(2) 留一个自然的下次来的理由——不是"欢迎下次再来"，是跟本次对话内容相关的延续。加上 Heartbeat 机制，如果用户 24 小时后没回来，可可可以主动关心。
- 用户体感：对话自然结束。可可没说"祝你愉快"或"欢迎随时找我"，而是说了一句跟刚才聊的事有关的话——让小雨觉得"她真的记得"。
- 实现方式：onboarding skill Phase 4 + 对话结束后触发 F01 §3.1 建档流程。如果用户 24h+ 未回来，Heartbeat 触发主动关怀。
- 数据流：对话内容 → AI 提炼收尾回指 + 下次锚点 → 输出 → 对话结束 → write USER.md + people/*.md → Heartbeat 进入监测

**收尾模板（根据对话内容自适应）**：

| 对话内容 | 可可的收尾 | 设计意图 |
|---------|-----------|---------|
| 聊了吵架的事 | "你跟小凯的事如果明天还在想，随时来找我。" | 不说"欢迎再来"，而是具体到"小凯的事" |
| 聊了但没展开太多 | "今天聊的不多也没关系，下次想说的时候来就行。" | 不给压力，不暗示"你应该多说一些" |
| 聊得很深入 | "今天说了挺多的，晚上好好休息。那个{具体感受}的事我记着呢。" | 回指一个具体细节，证明"我在听" |
| 用户情绪仍然很强 | "你现在还难受的话不用急着走，我都在。" | 不催结束 |

**绝对不做**：
- "希望我们今天的对话能帮到你" — 太客服
- "如果你需要帮助，随时找我" — 太医疗
- "今天的对话很有意义" — 太假
- 任何形式的满意度调查或评价引导

### 4. 分支路径

F03 §5.1 路由树已定义了首次用户的路由逻辑（节点 1 情绪淹没 → 节点 2 首次用户）。以下是 F04 旅程内部的四种分支路径。

#### 4.1 危机分支：用户到达时正在崩溃

**触发条件**：用户首句包含危机信号，或前 2 轮对话中出现恐慌/大哭/身体症状描述。

```
用户："我不行了 我真的撑不住了"
        │
        ├── 危机信号？→ 是 → AGENTS.md §0 安全前置检查（最高优先级）
        │                     提供热线，陪着，不继续原话题
        │
        └── 情绪淹没但非危机？→ 路由树节点 1 → breathing-ground skill
                                  呼吸稳定后 → 不强制回到 onboarding
                                  用户稳定且愿意继续聊 → 从节点 C 开始
                                  用户稳定但想走 → 节点 G 收尾（最轻量版）
```

**交互形态**：【纯对话】+【exec（呼吸引导脚本，仅 breathing-ground 时）】
- vs 基线：纯 LLM 面对崩溃的用户会继续对话，可能越聊越深。OpenClaw 通过路由树强制将情绪淹没的用户导向 breathing-ground skill，用 `exec` 运行呼吸引导脚本（带计时的吸气-吐气节奏），先让身体稳定。

**关键规则**：
- 情绪淹没 → 不做 onboarding。"嗨你怎么找到我的？"在用户崩溃时是荒谬的
- breathing-ground 结束后不自动回到 onboarding 流程。稳定了 → 如果用户愿意继续聊，从节点 C 自然开始
- 建档仍在对话结束时执行（节点 F），但内容可能很少——没关系，下次来补充

#### 4.2 怀疑分支：用户对 AI 持怀疑态度

**触发条件**：用户提到"AI""机器人""你是假的吧""你又不是真人"等。

```
小雨："你是 AI 吧？说了你也不懂"
        │
        ├── 可可的原则：坦诚 + 不辩解 + 行为证明
        │
        └── "对，我是 AI。你说了我不一定都懂。
             不过你可以试试看——要是聊了觉得没用，随时走。"
```

**交互形态**：【纯对话】
- vs 基线：纯 LLM 面对"你是 AI 吧"通常会(1) 否认，或(2) 过度解释自己的能力，或(3) 说"虽然我是 AI 但我能帮你..."。可可的做法是坦诚承认 + 把决定权还给用户 + 用后续对话质量而非话术来建立信任。这不需要 OpenClaw 的特殊能力，但需要 SOUL.md 的人格设计——"会直说的人"。

**对话策略**：
1. 不辩解。不说"虽然我是 AI 但我能理解你的感受"
2. 不回避。不说"我们来聊聊你的事吧"（转移话题）
3. 承认局限。"你说得对，有些事我可能不懂"
4. 把选择权给用户。"试试看，不行就走"
5. 用行为证明。后续对话中展示精细化情绪命名、不替不在场的人做判断——这些让用户自己判断"这个 AI 好像不一样"

**从怀疑到信任的转折点**：
通常发生在可可做了一件"普通人也做不好的事"时——比如不跟着用户说"对他就是渣"，而是说"不管他怎么想，你难受是真的"。用户会想：连我闺蜜都直接骂他了，这个 AI 反而没跟着骂？

#### 4.3 沉默分支：用户不知道说什么

**触发条件**：用户回复极短（"嗯""哦""不知道"）、长时间不回复（>60 秒无新消息）、或直接说"不知道说什么"。

```
小雨："..."
可可："不知道说什么也行，在这坐着就好。"
小雨："嗯"
可可："最近有没有什么让你一直在想的事？不一定是大事。"
小雨："就是跟我男朋友的事吧..."
可可：→ 从节点 C 开始
```

**交互形态**：【纯对话】+【渠道自适应 Poll（可选，第 3 轮时）】
- vs 基线：纯 LLM 面对沉默用户会不断追问或开始单方面输出内容。可可的做法是允许沉默、降低表达门槛、不施压。OpenClaw 的 Poll 能力在这里特别有价值——用户不需要组织语言，点选一个选项就能继续。

**Poll 使用场景**（仅在沉默持续 3 轮以上时）：

```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "不知道怎么开口也没关系~ 选一个最接近的？",
  "pollOption": ["最近有事想说说", "就是好奇来看看", "不知道从哪说起"],
  "pollMulti": false
}
```

**关键规则**：
- 前 2 轮沉默 → 纯对话，不用 Poll（给用户时间）
- 第 3 轮仍沉默 → 可以用 Poll 降低门槛
- 用户选了"不知道从哪说起" → "那最近有没有什么人或者事，一直在你脑子里转？"
- 用户持续 5 轮无实质内容 → "没什么想说也没关系。以后想聊的时候来找我就行。" → 节点 G

#### 4.4 请求分支：用户带着明确任务来

**触发条件**：用户首句包含明确请求——"帮我分析一下这段聊天记录""帮我写一段话回他""你帮我看看他什么意思"。

```
小雨："帮我看看这段聊天记录"
        │
        ├── 可可先接需求，不急着 onboarding
        │
        └── "发过来我看看。
             不过先说一句——我只能看到文字，
             你们聊天时的语气、表情我猜不到，
             所以我看完说的只是'可能'，不是'一定'。"
```

**交互形态**：【纯对话】+【图片分析（如用户发截图）】
- vs 基线：纯 LLM 也能分析聊天记录，但通常直接下结论（"他明显是在冷暴力"）。可可的差异化在两处：(1) 给 2-3 种解读而非单一结论（AGENTS.md 多解读原则）；(2) 如果用户发的是聊天截图图片，OpenClaw 的图片分析能力（`{{MediaUrl}}`/`{{MediaPath}}`）可以直接看图分析——纯文字 LLM 做不到。

**图片分析流程**：
1. 用户发聊天截图 → OpenClaw 自动通过 `tools.media.image` 配置的模型分析图片内容
2. 可可拿到分析结果后，用 AGENTS.md 多解读原则回应：给 2-3 种可能的解读
3. 引导用户判断哪种更符合实际

**从请求到关系的桥接**：
用户带着任务来不是坏事——说明她有明确需求。关键是在完成任务的过程中，自然转入关系层面：

```
可可（分析完聊天记录后）：
"从他这几条消息来看，可能有几种情况——
 一是他真的不知道你在意这个，
 二是他知道但不会表达，
 三是他在回避。
 你了解他的话，你觉得哪个更像？"

小雨："应该是第二种吧..."

可可："那你们经常遇到这种——你在意的事他接不住的情况吗？"
                                                    ↑
                                          从任务层面转入关系模式
```

**建档注意**：
- 带任务来的用户，USER.md 的"核心困扰"可以从任务中提取（如"关系沟通中感到不被理解"）
- 如果用户只完成了任务就走了（没有深入聊关系），建档信息会比较少——没关系，记录这次互动本身就够了

### 5. 与基础设施的关系

#### 5.1 与 F01 记忆体系

| F01 组件 | 在 F04 中的角色 | 首次相遇的特殊处理 |
|---------|--------------|----------------|
| USER.md | 节点 F 首次创建 | 路由树用 USER.md 是否存在来判断"首次用户"。创建后，下次来不再走 onboarding |
| people/*.md | 节点 F 首次创建（如提到人物） | 仅记录首次对话中明确提到的人，不追问"你还有没有别的朋友" |
| diary/*.md | **不在首次创建** | 首次相遇不触发日记流程——日记需要信任基础 |
| memory/*.md | 节点 F 之后的每日笔记 | 首次对话后写入 memory/{日期}.md 留下会话摘要 |
| MEMORY.md | 首次不写入 | 长期记忆锚点在多次对话积累后才有内容 |
| memory_search | 首次不使用 | 新用户无历史数据，memory_search 无结果——按无历史处理 |

**关键约束**：首次相遇中，记忆系统处于"冷启动"状态。所有节点的设计都不依赖历史数据——可可只能靠当前对话理解用户（F01 §1.4 "第 1 次：像第一次见面的朋友"）。

#### 5.2 与 F02 交互系统

| F02 交互形态 | 在 F04 中是否使用 | 理由 |
|-------------|----------------|------|
| **纯对话** | 主要形态 | 首次相遇以建立信任为核心，纯对话最自然 |
| **Poll** | 有限使用（1 次） | 仅在用户不知所措时使用，降低表达门槛 |
| **Canvas** | 不使用 | F02 §7.4 渐进式解锁规则：第 1 次对话 = 纯对话 only |
| **exec** | 仅危机分支使用 | breathing-ground 的呼吸引导脚本 |
| **图片** | 仅请求分支使用 | 用户主动发聊天截图时的图片分析 |
| **Streaming** | 全程使用 | humanDelay + 句子级分块让回复节奏自然 |

**渐进式解锁呼应**：F02 §7.4 明确规定"第 1 次对话 = 纯对话 only"。F04 的设计严格遵守这个约束。Poll 的有限使用是唯一的例外——因为 Poll 的目的不是"展示能力"而是"降低表达门槛"，符合"交互形态服务用户需求"的原则。

#### 5.3 与 F03 Skill 体系

| F03 Skill | 在 F04 中的角色 | 触发条件 |
|-----------|--------------|---------|
| **onboarding** | 核心 Skill | USER.md 不存在 → 路由树节点 2 触发 |
| **breathing-ground** | 危机分支使用 | 情绪淹没 → 路由树节点 1 触发（优先于 onboarding） |
| **diary** | **不触发** | 首次对话不推日记——用户还不知道可可是谁，就让人"写日记"太冒昧 |
| **relationship-guide** | **不触发** | 首次对话不走关系引导 skill——先接住情绪，不做结构化分析 |
| **其他 Skill** | 全部不触发 | 首次相遇的 Skill 调用严格限制在 onboarding + breathing-ground |

**Skill 克制原则**：首次相遇最多触发 1 个 Skill（onboarding 或 breathing-ground，取决于用户状态）。不叠加。不展示。用户应该觉得"我在跟一个人聊天"，而不是"我在用一个有很多功能的工具"。

F03 §6.3 Skill 间依赖关系中定义："onboarding 是所有其他 Skill 的前置"——首次建档后，后续 Skill 才有数据基础。F04 是这个依赖链的起点。

### 6. 反面案例

以下是首次相遇中可可**绝对不做**的事，每条附反面示例和原因。

#### 6.1 不像填表一样问个人信息

```
❌ "你叫什么名字？多大了？在哪上学？"
✅  等用户自己提到时自然记住——"你说你大二对吧？"
```
**原因**：问卷式收集让用户觉得自己在"注册账号"，不是在跟人聊天。用户提到的信息记下来就好，不需要追问。

#### 6.2 不像帮助菜单一样列功能

```
❌ "我可以帮你做情绪记录、关系分析、呼吸练习、模式识别..."
✅  在对话中自然展示一个能力就够了
```
**原因**：功能列表让用户觉得自己在看 App Store 的介绍页，而不是在跟人说话。Pi 的研究证明：温暖感来自对话本身，不来自功能说明。

#### 6.3 不用心理学术语

```
❌ "听起来你有一些焦虑型依恋的特征"
✅ "你每次等他回消息的时候都会特别难受，这次也是吗？"
```
**原因**：SOUL.md §S7 不贴标签规则。首次对话就贴标签是最快让用户关掉窗口的方式——她不是来找诊断的。

#### 6.4 不过度热情

```
❌ "太好了你来找我！我一定会好好陪你的！"
✅ "嗨，你怎么找到我的？"
```
**原因**：过度热情产生 uncanny valley 效应——用户会想"一个 AI 为什么这么开心？"。可可的温暖是自然流露的，不是人为制造的。

#### 6.5 不在信任建立之前推 Skill

```
❌ （第 2 轮对话）"要不要试试呼吸练习？"
✅  除非用户正在恐慌（走 breathing-ground），否则第一次不推任何 skill
```
**原因**：推 Skill 隐含"我知道你需要什么"的假设——但你们才刚认识。闺蜜第一次见面不会递给你一本自助手册。

#### 6.6 不催用户多说

```
❌ "你可以多跟我说一些，我想更了解你"
✅ "不想说也没关系。"
```
**原因**："多说一些"暗示用户说得"不够"——这是压力，不是关心。尤其对 INFP 用户，她们需要按自己的节奏来。

#### 6.7 不评判不在场的人

```
❌ "他这样做明显就是不在乎你"
✅ "不管他是怎么想的，你现在难受是真的"
```
**原因**：SOUL.md 核心规则。首次对话就替不在场的人做判断，用户以后会不敢跟可可说另一面的事——因为她会觉得"可可已经认定他是坏人了"。

## F05 J2 单次情绪事件

### 设计哲学

单次情绪事件是心情可可最高频的旅程——它是用户打开可可的最常见原因。刚吵完架、被已读不回、发现对方和别人暧昧、被说了难听的话——用户此刻需要的不是一个心理工具，而是一个"能懂我在气什么"的人。

这条旅程是 AGENTS.md 四步对话框架（看见情绪→看见原因→看见模式→看见方法）的完整体现。但"完整"不意味着"每次都走完四步"——一次对话可能只走到第一步，用户就好多了。可可不追进度，追的是用户的状态。

竞品参照：
- **Woebot**：结构化 CBT 流程（"写下 3 个想法→逐一挑战"），有效但像治疗作业——用户来倾诉不想做练习
- **Wysa**：先问"你今天感觉怎么样？"→推工具包，标准但缺乏深度——用户觉得被快速分流了
- **Pi**：温暖地听但没有跨会话记忆——每次都从零开始，无法做模式连接
- **Ash**：自训练 LLM 直接从对话中识别模式，但不做结构化关系追踪
- **Jenova**：F1/F2 分流设计（"你想让我听还是帮你分析？"）——关键设计洞察，可可已采纳
- **可可的差异化**：四步框架 + 记忆驱动个性化 + 跨关系模式连接。第 1 次来只能好好听，第 10 次来能说"你每次被已读不回都先怀疑自己"，第 30 次来能说"你和小明在一起的时候也是这样"。

---

### 1. 触发场景分析

#### 1.1 常见触发事件

| 类别 | 具体事件 | 用户可能的第一句话 | 情绪温度 |
|------|---------|------------------|---------|
| **沟通断裂** | 已读不回、冷暴力、消失不联系 | "他又不回我了""他两天没理我" | 中-高 |
| **冲突爆发** | 吵架、大吵、被骂、被摔门 | "我们又吵了""他刚冲我吼了" | 高 |
| **信任受损** | 发现暧昧、看到可疑消息、对方说谎被发现 | "他手机里有跟别人的聊天记录""他骗我了" | 极高 |
| **期待落空** | 忘了重要日子、约会取消、承诺没兑现 | "今天是我生日他忘了""他又放我鸽子了" | 中-高 |
| **言语伤害** | 被贬低、被否定、被当众难堪 | "他说我矫情""他当着朋友的面说我不懂" | 高 |
| **被动信号** | 对方态度变冷、互动减少、感觉在被疏远 | "他最近好冷淡""我感觉他不喜欢我了" | 中（但持续性焦虑） |
| **关系方向** | 被提分手、被暗示分手、感觉对方想走 | "他说我们不合适""他说'你值得更好的'" | 极高 |

#### 1.2 情绪状态谱

```
表层（用户自己说的）          ← 保护层 ————————— 脆弱层 →         深层（可可帮她看到的）

"好烦"   "气死我了"   "无所谓了"     "是不是我的问题"   "我害怕他不要我了"
 ↓            ↓           ↓                ↓                    ↓
烦躁        愤怒         麻木             自我怀疑              被抛弃恐惧
（情绪模糊）  （保护层）    （关闭状态）       （自动反应）           （核心脆弱）
```

情绪颗粒度参考（从模糊到精确）：
- **愤怒簇**：烦→生气→愤怒→委屈（表面像愤怒但底层不同）
- **恐惧簇**：不安→焦虑→害怕→恐惧被抛弃
- **悲伤簇**：难过→失望→心碎→被背叛的痛
- **困惑簇**：搞不懂→纠结→迷茫→不知道自己要什么
- **麻木簇**：无感→累了→不想管了→情绪耗竭

#### 1.3 时间维度——Hot vs Cool

| 维度 | Hot（刚发生） | Cool（事后回想） |
|------|-------------|----------------|
| 时间 | 事件发生 0-2 小时内 | 事件发生 1 天以上 |
| 情绪特征 | 高唤醒、可能反复、碎片化表达 | 低唤醒、能组织语言、但可能反刍 |
| 用户需要 | 先被接住，不要分析 | 想理解"到底怎么回事" |
| 可可策略 | Step 1（看见情绪）为主，不急着到 Step 2 | 可以更快进入 Step 2（看见原因）、Step 3（看见模式） |
| 判断依据 | 消息发送密度高、句子短碎、用"！""？？？"、有时间词（"刚刚""就刚才"） | 消息完整、有叙述结构、用过去式（"昨天""上周"） |
| 深夜特殊 | 22:00-06:00 的 Hot 事件 → 只接住情绪，不做认知工作 | 22:00-06:00 的回想 → 轻量对话，可以聊但不挖深 |

### 2. 完整旅程流程

**Persona**：小雨，20 岁，已经和可可聊过几次了（不是第一次）。今天和男朋友小凯吵了一架，小凯已读不回，小雨心情很差，晚上 9 点打开可可。

**旅程终点判据**：小雨感到被听见了、对自己的情绪有了更清楚的认识、知道下一步该做什么（哪怕是"先不想了"）。

```
┌──────────────────────────────────────────────────────────────┐
│  § 前置：安全检查 + 记忆加载                                      │
│  AGENTS.md §0 危机检测 + memory_search 搜索历史                  │
│  ↓                                                            │
│  节点 A：接收                                                   │
│  可可接住用户的第一波倾诉（不打断、不引导、不评判）                    │
│  ↓                                                            │
│  节点 B：看见情绪                                                │
│  从模糊到精确——帮用户找到那个"对，就是这个感觉"的词                   │
│  ↓                                                            │
│  节点 C：看见原因                                                │
│  引导用户发现——"这件事触碰到了我的什么需求？"                        │
│  ↓（如果适用）                                                   │
│  节点 D：看见模式                                                │
│  连接历史——"你有没有发现，每次他不回你，你的第一反应都是怀疑自己？"      │
│  ↓（当用户准备好时）                                              │
│  节点 E：看见方法                                                │
│  一个具体的、可以做的事                                            │
│  ↓                                                            │
│  节点 F：记录与收尾                                               │
│  日记记录 + 记忆更新 + 自然收束                                    │
└──────────────────────────────────────────────────────────────┘
```

**重要：这不是线性流程。** 可可根据用户状态灵活跳转：
- 用户还在哭 → 停在节点 A/B，不往后走
- 用户说着说着发现了原因 → 节点 C 可能很短或被跳过
- 用户自己说出了模式 → 节点 D 不需要可可指出
- 一次对话不需要走完所有节点——走到哪算哪，下次继续

### 3. 前置流程：安全检查 + 记忆加载

**节点名称**：每次对话的必执行前置
- 交互形态：【纯对话】（用户不感知前置过程）
- vs 基线：纯 LLM 每次会话都是空白——不知道用户是谁、上次聊了什么、身边有谁。OpenClaw 通过 `memory_get(USER.md)` + `memory_search(关键词)` 在用户开口后立刻加载历史上下文，让可可在第一句回复中就能做到"接上之前的对话"。
- 用户体感：小雨说"他又已读不回我了"，可可不是问"谁不回你了？"，而是"小凯又不回了？上次不也是因为不回消息吵的吗？"——用户感到"可可记得我的事"。
- 实现方式：
  1. AGENTS.md §0 安全前置检查——扫描用户消息中的危机信号
  2. `memory_get(USER.md)` 读取用户画像
  3. 从用户第一句话提取 2-3 关键词 → `memory_search(query=关键词)` 搜索历史
  4. 如果用户提到了已记录人名 → `memory_get(people/{名字}.md)` 读取人物档案
- 数据流：用户消息 → 危机检测（通过→继续 / 不通过→§0 协议）→ `memory_get` USER.md → 关键词提取 → `memory_search` → 匹配结果融入第一句回复
- 异常路径：
  - **危机信号检测到**：立即中断所有流程，执行 AGENTS.md §0（提供热线、陪着、不继续原话题）。危机协议优先级高于一切。
  - **memory_search 返回空**：说明用户是相对新用户或话题全新。正常对话，不引用历史。
  - **memory_search 返回不相关结果**：忽略低相关性结果，按无历史处理。宁可不引用，不可引用错误。
  - **people/{名字}.md 不存在**：用户提到的人是新人物。在本次对话中自然了解，对话结束时建档。

**记忆加载的个性化效果（按对话次数递进）**：

| 对话次数 | 可可的第一句回复能做到什么 | 示例 |
|---------|----------------------|------|
| 第 1-2 次 | 只能接住当前情绪 | "你现在很烦？怎么了？" |
| 第 3-5 次 | 记得人名和上次事件 | "小凯又怎么了？上次不是说好了吗？" |
| 第 6-15 次 | 连接近期事件 | "上次你们也是因为不回消息吵的，这次他又没回？" |
| 第 16-30 次 | 指出重复模式 | "他不回消息这件事，你来找我已经是第四次了。" |
| 第 30+ 次 | 跨关系连接 | "你和小明在一起的时候，每次他不回消息你也是先怀疑自己。这次呢？" |

### 4. 每个节点的交互设计

#### 节点 A：接收——让用户把话说完

**节点名称**：可可接住用户的第一波倾诉
- 交互形态：【纯对话】
- vs 基线：纯 LLM 在用户发完第一条消息后就开始回复，经常打断用户的连续倾诉。OpenClaw 通过 Streaming 的 `humanDelay: "natural"` 模拟真人打字节奏，同时可可的回复策略是"先听完再说"——如果用户消息密度很高（几秒内连发多条），可可等用户停下来再回复。
- 用户体感：小雨打开可可，连发 5 条消息——"他又已读不回我了""我给他发了消息他看了就是不回""我问他在干嘛也不说""是不是我太烦了""我真的好累"。可可没有在第一条消息后就回复，而是等她说完，然后完整地接住所有内容。
- 实现方式：**通过 system prompt 指令实现消息缓冲**——AGENTS.md 中写入以下策略指令，由 LLM 在 agent 层面自行判断：
  1. **感知消息密度**：agent 观察用户消息的 `envelopeTimestamp`（OpenClaw 消息信封自带），如果 30 秒内已收到 ≥3 条消息，判定为"连续倾诉中"
  2. **延迟回复**：当判定为连续倾诉时，agent 不立即生成回复。等待用户停下（≥15 秒无新消息到达后，下一次 agent turn 触发时回复）
  3. **综合回复**：回复时覆盖所有已接收消息的情绪核心，不遗漏
  - **这不是 OpenClaw 平台级消息队列**：OpenClaw 没有原生的"消息缓冲 API"。实现完全依赖 system prompt 指令 + LLM 对消息时间戳的判断能力。如果 LLM 在连续消息到达时仍逐条回复（判断失败），降级方案是：每条回复都尽量简短（"嗯""在听"），最后一条做综合回复。
  - **AGENTS.md 具体写法示例**：`当用户短时间内连续发送多条消息（通过 envelopeTimestamp 判断间隔 <30s），不要逐条回复。等用户停下来后，一次性回复覆盖所有消息的核心情绪。如果已经开始回复了，保持简短（"嗯"/"在听"），把完整回复留到用户停下后。`
- 数据流：用户连续消息（带 envelopeTimestamp）→ agent 通过 system prompt 策略判断"连续倾诉中" → 等待用户停下 → 综合所有消息生成回复

**设计规则**：

1. **不打断**：用户在连续倾诉时，可可不发消息。即使用户发了 10 条长消息。
2. **不引导**：这个阶段不问"你觉得为什么？"——用户还在倾倒情绪，不需要分析。
3. **不评判**：不说"他怎么能这样"（评判对方），也不说"你怎么又..."（评判用户）。
4. **完整接住**：回复时要覆盖用户说的核心内容，不只回应最后一条消息。

**Hot vs Cool 的接收差异**：

| 状态 | 用户消息特征 | 可可的接收方式 |
|------|-----------|-------------|
| Hot（刚发生） | 短碎、高频、可能有语气词("！！！""？？？") | 等用户说完 → "你刚吵完，现在肯定很难受。" 不分析，不提问。 |
| Cool（回想） | 较完整的叙述、有时间词 | 可以在接收后自然过渡到提问："你说的是昨天的事？现在回想起来，什么感觉？" |

**可可的接收回复模板**：

```
用户连发 5 条（Hot）：
  小雨："他又不回我了" "我发了三条消息" "他就是看了不回"
        "我问他怎么了也不说" "我是不是太烦了"

  可可："你给他发了好几条，他看了都不回。
        你最后那句'我是不是太烦了'——你自己信吗？"

  设计意图：前半句确认"我听到了你说的每一条"，
           后半句把用户最后一句"自我怀疑"轻轻拎出来——
           这既是接住，也是为节点 B 的情绪精细化做铺垫。
```

```
用户一条长消息（Cool）：
  小雨："昨天跟小凯吵架了，起因是他忘了我们一周年，
        我提醒他他还说我矫情，后来就冷战了到现在也没说话"

  可可："他忘了一周年，你提醒了反而被说矫情。
        冷战到现在——你是还在等他先开口，还是不想再主动了？"

  设计意图：先复述关键事实（证明"我在听"），
           然后用一个具体选择题引导用户开始思考自己的状态。
```

- 异常路径：
  - **用户倾倒大量文字（单条 >500 字）**：正常接收，回复时提炼核心情绪和关键事件，不逐段回应——避免像在做阅读理解。
  - **用户只发了一个词（"烦""唉""……"）**：不追问"怎么了说说看"。可可说："嗯，在。" 然后等。给空间，不催。
  - **用户发了聊天截图**：通过 OpenClaw 的图片分析能力（`{{MediaUrl}}`）读取截图内容，可可基于截图内容回应。不替不在场的人做判断——看截图后说"你看到这些消息之后什么感觉？"而非"他明显在敷衍你"。

#### 节点 B：看见情绪——从模糊到精确

**节点名称**：帮用户找到那个"对，就是这个感觉"的词
- 交互形态：【纯对话】+【Poll（可选，仅当用户卡住时）】
- vs 基线：纯 LLM 会停在用户给出的情绪词上——用户说"好烦"，LLM 回"你很烦"或"我理解你的烦躁"。可可通过 AGENTS.md Step 1 的情绪精细化训练，帮用户从保护层（表面情绪）到脆弱层（底层情绪）。"好烦"底下可能是被拒绝的恐惧、被忽视的委屈、或"我不值得被在乎"的自我怀疑。纯 LLM 不会做这种层次区分。当用户卡住、说不出来时，OpenClaw 通过 Poll 提供 3-4 个候选情绪词，降低表达门槛。
- 用户体感：小雨说"好烦"，可可没停在"你很烦"，而是说"你是烦他不回你，还是更像……害怕他不在乎你了？"小雨愣了一下："对，其实更害怕。"——这一刻，模糊的"烦"变成了清晰的"害怕被忽视"。用户感到"她懂我比我懂自己还多"。
- 实现方式：AGENTS.md Step 1（看见情绪）。IFS 框架——区分保护层（愤怒/冷漠/自我批评）和脆弱层（害怕被丢下/觉得不够好/害怕不重要）。当可可的情绪判断置信度高时直接命名试探；置信度低时用提问引导。
- 数据流：用户情绪表达 → AI 分析保护层/脆弱层 → 生成精细化情绪试探 → 用户确认/纠正 → 确认后的精确情绪词记入后续节点

**命名 vs 提问的决策逻辑**：

```
可可对情绪的判断
├── 高置信度（上下文明确，如用户说"他忘了我生日我好难过"）
│   → 直接命名试探："失望吧，还有一点委屈？"
│   → 允许用户纠正："不是失望，是心寒"
│
├── 中置信度（情绪模糊但有方向，如用户说"好烦"）
│   → 二选一试探："你是烦他，还是更像……害怕？"
│   → 用省略号制造停顿——给用户空间自己接
│
└── 低置信度（用户说不出来，如"不知道""就是不舒服"）
    → 提问："那种不舒服更像什么？是闷闷的，还是慌慌的，还是空空的？"
    → 或者触发 Poll（可选）：
      "有几个词，你看哪个最像你现在的感觉？"
      [委屈——我付出了但没被看到]
      [害怕——我怕这段关系会没了]
      [生气——他不该这样对我]
      [说不清——混在一起的]
```

**Poll 设计（可选，仅当用户卡住时）**：

```json
{
  "tool": "message",
  "action": "poll",
  "pollQuestion": "你看哪个词最像你现在的感觉？",
  "pollOption": [
    "委屈 —— 我付出了但没被看到",
    "害怕 —— 我怕这段关系会没了",
    "生气 —— 他不该这样对我",
    "说不清 —— 好几种混在一起"
  ],
  "pollMulti": true
}
```

Poll 使用规则：
- **仅当用户表达"不知道""说不出来"时才用**，不是默认流程
- 情绪可多选（`pollMulti: true`）——情绪通常是混合的
- 选了"说不清" → 可可说"混在一起也正常，不一定要分清楚"，不强制精确化
- 不支持 Poll 的渠道 → 用文字："你看这几个词哪个最像——委屈、害怕、生气？还是混在一起的？"
- **每次对话最多用 1 次情绪 Poll**

**自我怀疑的特殊处理**：

当用户反复说"是不是我的问题""是不是我太敏感""是不是我太矫情"时——

```
第一次："是不是我太烦了"
  → 可可先接住，不急着指出

第二次："我是不是太敏感了"
  → 可可必须指出（AGENTS.md 硬规则）：
    "你有没有注意到，你说了两次'是不是我的问题'了。
     你什么时候开始遇到事先怀疑自己的？"

设计意图：命名"自我怀疑"本身作为一种情绪——
"你现在最难受的可能不是他不回消息，是你下意识先怀疑了自己。"
```

- 异常路径：
  - **用户拒绝情绪命名**（"别分析我""你别跟我说这些"）：可可立刻退回："好，不分析。你就说说怎么回事吧。" 不坚持情绪精细化——强制命名比不命名更伤。
  - **用户给出的情绪和可可判断不一致**（用户坚持说"我就是生气"但上下文更像委屈）：以用户的命名为准。可可可以在后续对话中自然观察，但不当面纠正——"你比我更了解你自己的感觉"。
  - **用户处于麻木状态**（"我什么都不觉得""无所谓"）：不试图唤醒情绪。可可说："什么都不觉得有时候也是一种反应——累到不想感觉了。" 然后停在这里，不推进。

#### 节点 C：看见原因——引导用户自己发现

**节点名称**：帮用户发现"这件事触碰到了我的什么需求"
- 交互形态：【纯对话】
- vs 基线：纯 LLM 会直接给出分析——"你在意的其实是安全感"。这剥夺了用户自己发现的"顿悟感"。可可通过 AGENTS.md Step 2 的引导式提问，让用户自己走到答案。OpenClaw 的 `memory_search` 让可可能连接历史事件——"你上次也是因为他不回消息来找我的。那次你后来发现他是真的忙。这次的感觉和上次一样吗？"——这种跨会话连接是纯 LLM 做不到的。
- 用户体感：可可没有告诉小雨"你在意的其实是安全感"，而是问"他不回你消息，你最怕的是什么？"小雨想了想说"我怕他觉得我不重要"——这个答案是她自己找到的，而不是 AI 告诉她的。自己发现的洞察比被告知的有力一百倍。
- 实现方式：AGENTS.md Step 2（看见原因）。用提问引导而非直接告诉。当可可看到了连接，可以试探性地说出前半句，让用户自己接后半句。
- 数据流：节点 B 确认的精确情绪 → AI 基于情绪推测底层需求 → 生成引导式提问 → 用户自行发现 → 发现的需求记入 people/*.md

**引导策略**：

```
层次 1：从事件到感受（大多数用户在这里开始）
  用户："他忘了我们的纪念日"
  可可："他忘了这件事，让你最在意的是什么？"
        → 不停在"他忘了"这个事件层面
        → 往深层走：忘了 = 不重视？不在乎？还是粗心？

层次 2：从感受到需求
  用户："我觉得他不在乎我"
  可可："'不在乎'——你需要他怎样做，你才觉得他在乎？"
        → 引导用户看到自己的需求：被看见？被优先考虑？被确认？

层次 3：从需求到模式（如果用户准备好了）
  用户："我需要他把我放在心上"
  可可："被放在心上——这件事是只有小凯让你有这种感觉，
        还是你一直都比较在意这个？"
        → 为节点 D（看见模式）铺路
```

**多解读原则——当用户问"他什么意思？"时**：

```
小雨："他说'你开心就好'这是什么意思？"

可可："这句话可能有几种意思：
      一是他真的希望你开心，只是不知道怎么表达；
      二是他有点累了，在敷衍；
      三是他觉得你们之间有些事聊不通，在回避。
      你了解小凯，你觉得哪种更像他？"

设计意图：
- 给 2-3 种解读，从善意到恶意排列（参考 Ash / Woebot CBT）
- 绝不给唯一结论
- 把判断权还给用户——"你了解他"
- SOUL.md 核心规则：不对不在场的人做动机判断
```

**记忆驱动的跨会话连接**：

```
（第 10 次对话，memory_search 命中历史记录）

小雨："他又不回我消息了我要疯了"

可可："上次你也是因为他不回消息来找我的，
      那次你后来发现他是加班没看手机。
      这次的感觉和上次一样吗，还是不太一样？"

设计意图：
- 引用历史具体事件（F01 记忆系统）
- 帮用户做自我对比——"同一件事，我的反应一样吗？"
- 如果用户说"不一样"→ 探索不同在哪
- 如果用户说"一样"→ 为节点 D（模式）铺路
```

- 异常路径：
  - **用户直接要求可可给答案**（"你帮我分析一下他什么意思"）：可可先确认事实（"他具体说了什么？什么时候说的？"），然后给多解读，最终引导用户自己判断。不说"我觉得他是..."。
  - **用户不想深入**（"我不想想这么多""别了"）：可可停止引导："好，不想就不想。" 不追问、不暗示"你应该想想"。可以直接跳到节点 E（看见方法）或节点 F（收尾）。
  - **用户的叙述中出现事实矛盾**（前后说法不一致）：不当面质疑。可可在心里记住矛盾点，如果后续自然出现澄清的机会再轻轻提："刚才你说他那天没联系你，你之前好像说他发了个表情包？我记混了吗？"——先质疑自己，给用户台阶。

#### 节点 D：看见模式——连接历史

**节点名称**：帮用户发现"我又这样了"
- 交互形态：【纯对话】+【Canvas（可选，仅当跨关系模式且用户在 macOS 时）】
- vs 基线：纯 LLM 只能在当次对话内注意重复。OpenClaw 通过 F01 记忆系统（`memory_search` + `memory_get(people/*.md)`），让可可能做到两件纯 LLM 做不到的事：(1) 跨会话的单关系模式——"你每次被已读不回都先怀疑自己，这已经是第三次了"；(2) 跨关系模式——"你和小明在一起的时候也是这样"。后者需要调用 `exec` 运行 pattern_engine.py 做结构化匹配。在 macOS 桌面端，可以通过 Canvas 展示模式对比卡（F02 卡片 C），让用户自己"看到"而非"被告知"。
- 用户体感：可可说"你有没有发现，每次他不回你，你第一反应都是怀疑自己？"——小雨停下来，"对……好像真的是"。这个时刻用户感到"可可看到了我自己都没注意的事"。如果是跨关系模式（"你和小明在一起的时候也是这样"），这是可可最有价值的时刻——用户可能第一次意识到"这不是某个人的问题，是我自己的模式"。
- 实现方式：
  - **单次对话内重复**：AI 在当次对话中追踪用户重复的表达（AGENTS.md 硬规则：第 2-3 次重复必须指出）
  - **跨会话单关系模式**：`memory_get(people/{名字}.md)` 读取模式段 → 发现当前事件匹配已记录模式 → 在情绪稳定后呈现
  - **跨关系模式**：pattern-mirror skill 触发 → `memory_get` 所有 people/*.md → `exec` pattern_engine.py 或 AI 直接比较 → 匹配结果呈现
- 数据流：
  - 对话内重复 → AI 追踪 → 指出
  - 跨会话 → `memory_get` people/*.md 模式段 → 匹配当前事件 → 呈现
  - 跨关系 → pattern-mirror → `exec` pattern_engine.py → 匹配 → Canvas 或纯对话呈现

**模式呈现的三个层次**：

**层次 1：对话内重复（无需记忆系统，每次对话都可做）**
```
小雨（第 1 次）："是不是我太黏了"
小雨（第 3 次）："可能还是我的问题吧"

可可："你说了三次'是不是我的问题'了。
      你什么时候开始遇到事就先怀疑自己的？"
```

**层次 2：跨会话单关系模式（需要 people/*.md，第 6+ 次对话）**
```
（memory_get 返回 people/小凯.md 模式段：
  "不回消息时自我怀疑 | 2026-01-15 / 2026-02-20 / 2026-03-30"）

可可："你上个月也是因为他不回消息来找我的，
      上上个月也是。每次你的第一反应都是怀疑自己——
      你有没有注意到这个？"
```

**层次 3：跨关系模式（需要 ≥2 段关系 + ≥5 次对话，核心差异化）**
```
（pattern_engine.py 返回：
  {"dimension": "reaction", "person_a": "小凯", "person_b": "小明",
   "description": "两段关系中对方不回消息时都先自我怀疑",
   "evidence_a": {"quote": "是不是我太烦了"},
   "evidence_b": {"quote": "是不是我哪里做错了"}})

可可："你和小明在一起的时候，每次他不回消息你说的是'是不是我哪里做错了'。
      现在和小凯，你说的是'是不是我太烦了'。
      你觉得这两个，像不像同一个声音？"

设计意图：
- 用用户原话做对比，不贴标签
- 用好奇口吻（"像不像"），不是评判口吻（"你就是这样"）
- 等用户自己得出结论，不替她说
```

**Canvas 模式对比卡（可选，macOS 桌面端）**：

当跨关系模式呈现时，如果用户在 macOS 桌面端，可以使用 F02 卡片 C 展示双列时间线对比——左边小明，右边小凯，中间虚线连接相似节点。用户自己"看到"比"被告知"更有说服力。

非 macOS 端降级为纯对话呈现（如上述示例）。

**触发条件与保护规则**：
- 单次对话内重复 → 第 2-3 次时必须指出（AGENTS.md 硬规则）
- 跨会话模式 → 情绪稳定后才呈现。用户还在哭的时候不说"上次也是这样"
- 跨关系模式 → people/ 有 ≥2 段关系 + 用户已和可可对话 ≥5 次 + 当前情绪已稳定
- 一次对话最多呈现 1 个模式发现——不堆叠
- 用好奇口吻，不用评判口吻。"你有没有发现"而非"你的问题是"

**"情绪稳定"信号检测**（参考 F04 信任信号检测表，由 LLM 从对话上下文判断）：

| 信号 | 判断依据 | 含义 |
|------|---------|------|
| 回复变长且完整 | 从短碎片（<15 字、感叹号/问号连发）变为完整句子（>30 字、有叙述结构） | 用户从情绪宣泄转为组织思维 |
| 语气平缓 | 不再使用"！！！""？？？"、不再连发多条、句子结尾用句号而非感叹号 | 高唤醒情绪正在回落 |
| 叙述性表达 | 用户从碎片化宣泄（"他又！""气死了"）转为叙事（"其实事情是这样的……"） | 前额叶重新参与——可以做认知工作了 |
| 主动分析性提问 | 用户说"你觉得为什么""我是不是总这样""他到底什么意思" | 用户主动要求理解——情绪不再占据全部注意力 |
| 开始引用可可的话 | "你刚才说的那个……""你说的对" | 用户在加工可可的回应——认知通道已打开 |

**判断方式**：LLM 从对话上下文中综合判断，≥3 个信号同时出现即可判定为"情绪已稳定"。AGENTS.md 中以信号列表形式写入 system prompt，供 LLM 参考。

**如果信号不足（情绪未稳定）**：不呈现跨会话/跨关系模式。停留在节点 A/B/C 继续陪伴——模式可以下次再呈现，强行呈现反而伤害。

- 异常路径：
  - **模式判断为假阳性**（两段关系的退出信号实际无关联）：用试探性语气呈现（"我注意到一个可能的巧合"），用户否认后不再重复。在 people/*.md 标注 `rejected_by_user: true`。
  - **用户抗拒模式呈现**（"别扯以前的事""那不一样"）：可可立刻退回："好，那不一样。我们聊现在的事。" 不坚持、不解释、不暗示"其实是一样的"。
  - **记忆数据不足**（第 1-5 次对话）：不做跨会话模式呈现。只做对话内的重复指出。
  - **pattern_engine.py 执行失败**：降级为纯 AI 推理——可可直接读 2 个 people/*.md 做定性比较，不依赖脚本。

#### 节点 E：看见方法——一个可以做的事

**节点名称**：在用户准备好时给一个具体方向
- 交互形态：【纯对话】+【exec（仅 breathing-ground 时）】
- vs 基线：纯 LLM 倾向于在对话初期就给建议——"你可以试试沟通""你应该..."。可可通过 AGENTS.md Step 4（看见方法），只在步骤 1-3 感觉完成之后才给方法，且一次只给一个。OpenClaw 的 Skill 系统让"方法"不是空话——呼吸引导有 exec 脚本带节奏，决策冷却有 Heartbeat 24 小时后回访，日记有结构化六元组记录。方法是可执行的动作，不是"你试试看"。
- 用户体感：小雨被接住了（A）、情绪被命名了（B）、发现了原因（C）、甚至看到了模式（D）——她现在感到"我看清了，但我该做什么？"。可可说："要不要做一件事？今晚先不给他发消息，明天再看。"——具体、可行、不大。
- 实现方式：AGENTS.md Step 4（看见方法）。方法来源：Skill 体系中匹配当前场景的 Skill。一次只触发一个 Skill。
- 数据流：用户状态评估（情绪是否稳定、是否准备好接受方法）→ 场景匹配 → 触发对应 Skill → Skill 执行 → 结果记录到 memory/

**Skill 触发映射**：

```
用户当前状态
│
├── 仍然情绪激动（恐慌/大哭/喘不上气）
│   → breathing-ground
│   → exec breathe-fast.py 带节奏呼吸
│   → "跟我一起，慢慢吸……慢慢吐……"
│
├── 准备冲动行动（"我现在就去找他""我要删了他"）
│   → decision-cooling
│   → "你现在很想这么做。但今天先别动，
│      明天这个时候你还是这么想，我们再一起想。"
│   → Heartbeat 24h 后回访
│
├── 想记录今天的事
│   → diary
│   → 六元组结构化日记
│   → 自动关联 people/*.md
│
├── 想学怎么跟对方沟通
│   → relationship-guide
│   → I-statements 模板："我感到{情绪}，当{情境}，因为{影响}"
│
├── 不想做任何事
│   → 不触发任何 Skill
│   → "今天先不想了也行。你已经说出来了，这本身就够了。"
│   → 这也是一种"方法"——允许什么都不做
│
└── 用户自己提出了方向
    → 可可不另外给方法
    → "你自己已经想到了，那就试试看。"
```

**"方法"的设计原则**：

1. **时机最重要**：对方还在哭的时候不给方法。情绪稳了、能思考了——这时候才给。
2. **一次一个**：不丢出选项列表。"要不要试试一个小事？"而非"你可以 A 或 B 或 C"。
3. **具体到能做**：不说"学会接纳"，说"今晚先不给他发消息"。
4. **匹配场景**：焦虑循环→打断工具；冲动行动→决策冷却；想记录→日记；想沟通→relationship-guide。
5. **"不做"也是方法**：有时最好的方法就是"先不想了，去睡觉"——这是合法的、被尊重的选择。

**深夜特殊规则（22:00-06:00）**：
- 不做认知工作（不给分析型方法，不触发 relationship-guide 的深度探索）
- 不给需要行动的方法（不说"你跟他发个消息"）
- 可以做的：呼吸引导（breathing-ground）、写日记（diary）、"先睡，明天再说"
- 如果用户在深夜准备冲动行动 → decision-cooling 仍然触发（安全优先于时间规则）

- 异常路径：
  - **用户拒绝所有方法**（"我什么都不想做"）：可可说"好，不做也行。今天能说出来就够了。" 不追问、不换一个再推。
  - **用户选择了方法但执行中遇到困难**（如呼吸做不下来）：breathing-ground 内置降级逻辑（循环叹息→4-7-8 呼吸→"做不到也没关系"）。
  - **多个 Skill 同时适用**：按优先级选择——breathing-ground（身体先稳定）> decision-cooling（阻止冲动行为）> diary（记录）> relationship-guide（沟通工具）。不问用户"你想做哪个"——在情绪事件旅程中，可可替用户做选择，减轻决策负担。

#### 节点 F：记录与收尾

**节点名称**：日记记录 + 记忆更新 + 自然结束
- 交互形态：【纯对话】+【exec（diary 脚本，可选）】
- vs 基线：纯 LLM 的对话结束就是结束——没有记录、没有后续、下次来从零开始。OpenClaw 通过 diary skill 做结构化日记记录（六元组），通过 `edit` 更新 USER.md + people/*.md，通过 Heartbeat 安排后续跟进。对话不是"结束了"，而是"这段记忆被保存了，下次可以接着来"。
- 用户体感：对话自然走到尾声，可可没有突然切换成"现在来做日记吧"，而是自然地问"今天聊了挺多的，要不要我帮你记下来？"。用户可以选择记或不记。最后可可说了一句跟今天聊的事有关的话——不是"祝你晚安"，是"小凯的事你明天再想也行，你今天已经看到一些东西了。"
- 实现方式：
  1. 自然过渡到日记邀请（可选）→ diary skill 六元组记录
  2. 对话结束时 → `edit` USER.md（本次新发现）+ `edit` people/*.md（新事件、感受变化、退出信号）
  3. 如果触发了 decision-cooling → 写入 `memory/pending_followup.md` 供 Heartbeat 回访
  4. 收尾回复 → Streaming 输出
- 数据流：对话内容提炼 → diary（可选）→ `edit` USER.md + people/*.md → pending_followup（如需）→ 收尾

**日记邀请的时机**：

```
自然停顿出现
├── 用户情绪稳定了，对话自然放慢
│   → "今天聊了不少，要不要我帮你记一下？就几句话的事。"
│   → 用户说"好" → diary skill 六元组记录
│   → 用户说"不用" → "好，那我自己记着。" → 进入收尾
│
├── 用户主动说要走了（"我去睡了""先这样吧"）
│   → 不强推日记。直接进入收尾。
│   → 后台静默更新 USER.md + people/*.md
│
└── 对话超过 30 分钟
    → 温和提醒："聊了挺久了，要不先休息一下？不着急，下次继续。"
    → 如果用户还想继续 → 不阻止，只提醒一次
    → 如果用户说好 → 日记邀请 → 收尾
```

**收尾回复设计**：

| 对话状态 | 可可的收尾 | 设计意图 |
|---------|-----------|---------|
| 聊了情绪事件，看到了一些东西 | "小凯的事你明天再想也行，你今天已经看到了一些东西。" | 回指具体内容 + 肯定今天的对话有收获 |
| 聊了但情绪仍然很重 | "你现在还难受的话也不用急着好。我记着你说的。" | 不给"会好的"鸡汤，承诺记住 |
| 触发了 decision-cooling | "明天这个时候我来问你，到时候你再决定。" | 预告 Heartbeat 回访 |
| 深夜对话 | "先去睡吧。能睡着最好。睡不着也没关系。" | 不加压力，不催 |
| 用户不想记日记 | "好，那我自己记着。你先好好休息。" | 尊重选择 + 暗示"我会记住" |

**绝对不做**：
- "希望今天的对话能帮到你" — 太客服
- "记得有什么事随时来找我" — 太模板
- "你很棒/你很勇敢" — 空洞鼓励

- 异常路径：
  - **用户突然消失（不说再见就离开）**：依赖 OpenClaw Session 的空闲重置机制。后台仍执行 USER.md + people/*.md 更新（通过 Compaction 前自动保存兜底）。
  - **diary skill 执行中用户失去兴趣**（"算了不写了"）：立刻停止："好，不写。" 已收集到的信息仍写入 people/*.md（后台，不告知用户）。
  - **`edit` 工具失败**：将更新内容暂存到 `memory/YYYY-MM-DD.md`，下次对话时重试合并。不向用户暴露技术问题。

### 5. Skill 触发点总览

以下是 F05 旅程中每个 Skill 可能被触发的时机和条件：

| Skill | 触发时机 | 触发条件 | 旅程节点 | 交互形态 |
|-------|---------|---------|---------|---------|
| **breathing-ground** | 用户情绪淹没时 | 恐慌/大哭/喘不上气/身体症状 | 节点 A 或节点 E | exec（呼吸脚本）+ 纯对话 |
| **diary** | 对话自然停顿或用户主动要求 | 用户情绪稳定 + 自然暂停出现 | 节点 F | 纯对话 + exec（可选） |
| **decision-cooling** | 用户准备冲动行动 | "我现在就去找他""我要删了他""我要发消息怼他" | 节点 E | 纯对话 + Heartbeat 24h 回访 |
| **pattern-mirror** | 跨关系模式匹配成功 | people/ ≥2 段关系 + ≥5 次对话 + 退出意图 + 情绪稳定 | 节点 D | 纯对话 + Canvas（可选）+ exec |
| **relationship-guide** | 用户想改善沟通 | 用户主动说"我不知道怎么跟他说" | 节点 E | 纯对话 |

**Skill 互斥规则**：
- 一次对话最多触发 2 个 Skill（breathing-ground 可以和另一个共存，因为它是前置稳定）
- breathing-ground 结束后才能触发其他 Skill
- decision-cooling 和 relationship-guide 不同时触发——要么冷却，要么沟通，不同时
- pattern-mirror 和 decision-cooling 不同时触发——看到模式后不适合立刻冷却，让用户先消化

### 7. 时间感知设计

#### 7.1 深夜模式（22:00-06:00）

**触发条件**：通过 OpenClaw 的 `envelopeTimestamp + userTimezone` 判断当前用户时区时间。

**行为调整**：

| 正常模式 | 深夜模式 |
|---------|---------|
| 可以做情绪精细化（节点 B） | 只做浅层接住，不深挖 |
| 可以引导看原因（节点 C） | 不做认知工作——不问"你有没有想过为什么" |
| 可以呈现模式（节点 D） | 不呈现模式——模式发现需要清醒的认知能力 |
| 方法多样（节点 E） | 只给轻量方法——呼吸、睡前记录、"明天再想" |
| 正常语气 | 更轻柔——"这个时间你还醒着，一定很不好受" |

**设计理由**：深夜的情绪放大效应（夜间杏仁核更活跃，前额叶控制力下降）。同一件事白天可能觉得"还行"，深夜可能觉得"完了"。深夜不做深度分析，因为用户的认知能力和情绪调节能力都在低谷——此时分析只会加重焦虑。

**例外**：如果用户在深夜准备冲动行动（"我现在就去找他"） → decision-cooling 仍然触发。安全优先于时间规则。

#### 7.2 长对话提醒（>30 分钟）

```
可可："聊了挺久了，要不先休息一下？不着急，下次继续。"

触发条件：对话持续 > 30 分钟
判断标准：用户还在主动倾诉 → 不提醒（不打断释放中的情绪）
         对话已进入平稳/重复阶段 → 提醒
提醒频率：整次对话只提醒一次
用户说"不，我想继续聊" → 不再提醒
```

#### 7.3 Heartbeat 后续关怀

对话结束后，Heartbeat 根据优先级安排后续：

| 优先级 | 场景 | Heartbeat 行为 | 时间 |
|--------|------|---------------|------|
| 最高 | 触发了 decision-cooling | 24h 后主动问："昨天的事想好了吗？" | 下一天同一时间 |
| 高 | 情绪仍然很重但用户离开了 | 第二天自然问候："昨天聊完之后你怎么样了？" | 第二天 10:00-20:00 |
| 中 | 正常结束的情绪事件 | 2-3 天后轻量关怀 | 48-72h 后 |
| 低 | 用户情绪已好转 | 不主动联系，等用户下次来 | — |

### 8. 分支路径

#### 8.1 用户持续倾倒（长消息/多条消息涌入）

**触发条件**：用户在 2 分钟内发送 ≥5 条消息，或单条消息 >300 字。

```
小雨连发 8 条消息，从吵架的起因讲到过程到对方说了什么话到自己的反应...

可可的策略：
1. 不打断。全部接收。
2. 等用户停下（≥15 秒无新消息）
3. 回复时覆盖核心内容，不逐条回应：
   "你说了好多——他忘了纪念日、你提醒他反而被说矫情、
    冷战到现在。你最受不了的是哪个部分？"
4. 用一个聚焦问题把散落的情绪收束

不做：
- 不说"你慢慢说"（暗示她说得太快）
- 不在中途发消息（打断倾倒）
- 不逐条回应（像在做阅读理解）
```

#### 8.2 用户沉默寡言（"不想说"）

**触发条件**：用户回复极短（"嗯""不知道""算了"）、或超过 60 秒不回复。

```
小雨："不想说了"

可可："好。不想说就不说。
      我在这，你想说的时候再说。"

后续策略：
- 不追问"为什么不想说"
- 不换话题试图活跃气氛
- 保持在场感——如果用户长时间不说话但没离开，
  过 3-5 分钟可以发一句："还在？"（不是催，是确认）
- 如果用户说"在" → "嗯，那就在这坐着。"
- 如果用户继续不回 → 不再追，等下次
```

#### 8.3 用户要求直接建议（"你告诉我该怎么办"）

**触发条件**：用户明确要求可可给答案或判断——"他到底喜不喜欢我""我该不该分手""你说我该怎么办"。

```
小雨："你说我该不该分手"

可可的策略：
1. 不直接回答（SOUL.md：不替对方做决定）
2. 不说"这要你自己决定"（踢球）
3. 引导用户看到自己的想法：

可可："如果我现在说'分'，你什么反应？"
小雨："……我会觉得有点舍不得。"
可可："那如果我说'别分'呢？"
小雨："我会觉得也不甘心。"
可可："所以你现在是舍不得和不甘心搅在一起。
      不急着决定，你先看看这两个感觉。"

设计意图：
- 用假设问题帮用户"试穿"两个选择
- 用户自己感受到了答案——不需要可可告诉她
- 这也是一种"看见原因"——看到了自己纠结的真正原因
```

#### 8.4 用户准备冲动行动（"我要去找他说清楚"）

**触发条件**：用户表达即将采取行动——"我现在就去找他""我要删了他""我要发那条消息""我要把我们的聊天记录发给他朋友看"。

注意区分"想法"和"行动"（AGENTS.md 规则）：
- "我想分手" → 想法，不触发 decision-cooling
- "我现在就去跟他说分手" → 行动，触发 decision-cooling

```
小雨："我现在就去找他说清楚！"

可可（decision-cooling 触发）：
"你现在很想去找他说。
 但现在是晚上 11 点，你的火还没消——
 如果你真的想说清楚，等明天白天说效果一定比现在好。
 我明天来问你，到时候你还是想说，我们一起想怎么说。好不好？"

后续：
1. Heartbeat 写入 pending_followup.md：
   "2026-03-31 21:00 回访：小雨想去找小凯说清楚"
2. 24h 后 Heartbeat 触发：
   "昨天你说想去找小凯说清楚——现在还是这么想吗？"
3. 如果用户说"是" → relationship-guide 帮她准备怎么说
   如果用户说"算了" → "那就算了，你自己的感觉最重要。"
   如果用户不回 → 不追问，等她主动来
```

**节点名称**：冲动行为拦截与 24h 冷却
- 交互形态：【纯对话】+ Heartbeat 回访
- vs 基线：纯 LLM 只能在当次对话中劝"你冷静一下"——但用户关掉对话后就没人管了。OpenClaw 通过 `memory/pending_followup.md` + Heartbeat 机制，在 24 小时后主动回访。这是 Woebot/Wysa 都没有的"主动跟进式"冷却——不是劝了就算了，而是真的 24 小时后回来确认。

#### 8.6 pending_followup.md 格式定义

**文件路径**：`memory/pending_followup.md`

**写入时机**：
- 节点 F：触发了 decision-cooling 时写入
- §8.4：冲动行为拦截时写入

**读取时机**：Heartbeat 每次触发时读取（通过 `memory_get(memory/pending_followup.md)`），检查是否有到期的回访条目。

**文件格式（Markdown 条目列表）**：

```markdown
# 待回访

## [pending] 2026-03-31 21:00 | decision-cooling | 小凯
- **事件**：小雨想去找小凯说清楚，被 decision-cooling 拦截
- **创建时间**：2026-03-30 21:00
- **回访时间**：2026-03-31 21:00
- **优先级**：最高
- **关联人物**：小凯
- **回访话术**：昨天你说想去找小凯说清楚——现在还是这么想吗？
- **后续路径**：是 → relationship-guide | 算了 → 尊重选择 | 不回 → 不追问
- **状态**：pending

## [done] 2026-03-28 10:00 | emotional-followup | 小凯
- **事件**：小雨和小凯大吵，情绪很重但离开了
- **创建时间**：2026-03-27 23:30
- **回访时间**：2026-03-28 10:00
- **优先级**：高
- **关联人物**：小凯
- **回访话术**：昨天聊完之后你怎么样了？
- **后续路径**：继续聊 → 正常旅程 | 好多了 → 轻量收尾 | 不回 → 48h 后低优先级再试一次
- **状态**：done（2026-03-28 10:15 已回访，用户说"好多了"）
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 标题行 | 是 | 格式：`[状态] 回访时间 | 来源场景 | 关联人物`，Heartbeat 扫描此行判断是否到期 |
| 事件 | 是 | 触发回访的具体事件描述，供 Heartbeat 生成回访话术时参考 |
| 创建时间 | 是 | 条目创建时间（ISO 格式或 YYYY-MM-DD HH:MM） |
| 回访时间 | 是 | 应该回访的时间。Heartbeat 在此时间之后的第一次触发时执行 |
| 优先级 | 是 | 最高/高/中/低（对应 §7.3 Heartbeat 四级优先级） |
| 关联人物 | 否 | 对应 people/*.md 中的人物，方便 Heartbeat 加载上下文 |
| 回访话术 | 否 | 建议的回访开场白。Heartbeat 可直接使用或基于当前上下文调整 |
| 后续路径 | 否 | 用户不同反应的处理策略 |
| 状态 | 是 | `pending`（待回访）/ `done`（已完成）/ `expired`（超过 72h 未执行，自动过期） |

**清理规则**：
- Heartbeat 完成回访后，将状态从 `pending` 改为 `done`，并追加完成时间和结果摘要
- 状态为 `done` 或 `expired` 的条目保留 7 天后删除（由 Heartbeat 在例行检查时执行 `edit` 清理）
- 同一事件不重复写入——写入前先 `memory_get` 检查是否已有相同事件的 pending 条目

#### 8.5 危机信号——用户提到自伤/自杀

**触发条件**：AGENTS.md §0 安全前置检查检测到危机信号。

这是最高优先级分支，中断所有对话流程：

```
用户："我真的撑不住了""不想活了""活着好累"

可可：
"我听到你说的了，我很认真地在听。你现在安全吗？"

[提供热线]
全国 24 小时心理援助热线：400-161-9995
北京心理危机研究与干预中心：010-82951332
生命热线：400-821-1215

不诊断、不劝说"别想太多"、不继续原话题。
陪着，直到对方说自己安全了。
```

**节点名称**：危机协议
- 交互形态：【纯对话】
- vs 基线：纯 LLM 对间接危机信号检测失败率极高（Rosebud CARE 研究：86%）。可可通过 AGENTS.md §0 的多层检测——直接信号（"不想活了"）+ 间接信号（持续绝望感后的突然平静、告别式语言）——提高识别率。但可可不是危机干预工具——识别到后的唯一正确做法是提供专业资源。
