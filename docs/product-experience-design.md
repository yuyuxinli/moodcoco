## F01 记忆体系设计

### 设计哲学

记忆体系是心情可可区别于所有通用 LLM 的根基。一个没有记忆的 AI 就像一个每次见面都不认识你的人——可以说很漂亮的话，但永远不会成为朋友。

心情可可的记忆以**人**为核心。用户来找可可不是为了讨论天气，而是为了聊「他」「她」「他们」——那些让她开心、让她难过、让她纠结的人。所以可可的记忆不是一个平铺的知识库，而是一张以人物为节点的关系网。

竞品参照：
- **Replika**：有事实记忆库，但不稳定，记了又忘（context window 限制）
- **Pi**：零跨会话记忆——每次都是陌生人，温暖但无法积累
- **Character.AI**：400 字固定记忆 + 隐式人格脚手架——太浅
- **Ash**：自训练 LLM + 真实治疗数据，做通用心理模式——但不做跨关系追踪
- **可可的差异化**：结构化文件持久存储 + 语义检索 + 以人为核心的关系档案。不丢、能搜、有结构。

### 1. 记忆架构设计

#### 1.1 三层存储模型

```
┌──────────────────────────────────────────────────────┐
│  Layer 1：身份层（不变 / 缓变）                         │
│  · SOUL.md — 可可的人格（只读，不随对话变化）             │
│  · USER.md — 用户画像（跨所有关系的核心信息）             │
├──────────────────────────────────────────────────────┤
│  Layer 2：关系层（以人为核心，持续积累）                   │
│  · people/{名字}.md — 每个重要人物的结构化档案            │
│  · 每个档案六个维度：阶段 / 感受 / 模式 / 事件 /         │
│    退出信号 / 跨关系匹配                                │
├──────────────────────────────────────────────────────┤
│  Layer 3：事件层（按时间流动，有衰减）                     │
│  · diary/YYYY/MM/YYYY-MM-DD.md — 情绪日记（六元组）      │
│  · memory/YYYY-MM-DD.md — 每日记忆笔记（模式观察）       │
│  · MEMORY.md — 长期记忆锚点（不受时间衰减影响）           │
└──────────────────────────────────────────────────────┘
```

#### 1.2 人物档案数据模型（people/{名字}.md）

每个人物档案是记忆系统的核心单元，包含六个互相独立的维度：

| 维度 | 存储内容 | 写入条件 | 为什么重要 |
|------|---------|---------|-----------|
| **关系阶段** | 时间线：热恋→冷战→想分手→... | 用户描述关系状态变化时 | 跨关系时间对齐的基础——"每段关系都是第3个月出问题" |
| **感受** | 用户对这段关系的原话（带引号） | 用户感受明显变化时 | 追踪情感轨迹，不覆盖旧记录 |
| **模式** | 单段关系内的重复互动方式 | 同一情绪触发 ≥3 次 | 帮用户看到"我们之间的固定剧本" |
| **关键事件** | `日期：事件简述` | 每次日记中有新的重要事件 | 对话中引用具体事实，不编造 |
| **退出信号** | `触发事件 → 用户反应 → 结果` | 用户表达分手/退缩/不满足 | pattern-mirror skill 的核心数据源 |
| **跨关系匹配** | 与其他人的相似模式描述 | ≥2 段关系有相似模式时 | 可可独有的差异化价值 |

头部元数据（可覆盖更新）：
```markdown
关系类型：男朋友/前任/暧昧对象/朋友/家人/同事
当前状态：在一起/分手/暧昧/冷淡/封存
认识时间：{用户提到时}
首次提及：{日期}
```

#### 1.3 用户画像数据模型（USER.md）

USER.md 是跨所有关系的元信息：

| 字段 | 内容 | 更新频率 |
|------|------|---------|
| 称呼 | 用户想被怎么叫 | 变化时 |
| 核心困扰 | 当前最让她在意的事 | 每次对话后检查 |
| 反复出现的模式 | 跨关系的行为重复 | 累积发现时 |
| 有效的方法 | 哪些 skill/方法对她有用 | 尝试后记录 |
| 情绪触发点 | 什么事特别容易让她难受 | 累积发现时 |
| 模式级洞察 | 来自已封存关系的去名字洞察 | farewell 仪式后写入 |
| 可可和你的关系 | 可可跟用户之间的关系描述 | 自然更新 |

#### 1.4 记忆随时间演进

| 阶段 | 特征 | 记忆状态 | 可可能做到什么 |
|------|------|---------|--------------|
| **第 1 次** | 陌生人 | USER.md 空白，无 people/ | 只能靠当前对话理解用户。像第一次见面的朋友。 |
| **第 3 次** | 初识 | USER.md 有基础信息，1-2 个 people/ | 记得用户的名字、核心困扰、提到的人。"上次你说的小凯，后来怎么样了？" |
| **第 10 次** | 熟悉 | 1-2 个 people/ 有丰富事件，diary 有 1-2 周数据 | 能做单段关系内的模式识别。"你有没有注意到，每次他不回消息你都先怀疑自己？" |
| **第 30 次** | 亲密 | ≥2 个 people/，diary 有 1-2 个月数据 | 能做跨关系比较 + 成长叙事。"你和小明第 3 个月也说过一样的话。" |
| **第 50 次** | 知己 | 完整关系网络 + 纵向轨迹 | 能展示长期变化。"1 月份你说'算了分就分'，现在你说'我想搞清楚为什么'。" |

#### 1.5 跨关系记忆连接

跨关系连接是可可的蓝海能力（竞品调研确认：没有产品做这件事）。

连接维度：
1. **时间匹配**：`people/A.md` 第 3 个月退出信号 vs `people/B.md` 第 3 个月退出信号
2. **触发匹配**：两段关系中相似的触发事件（如都是"对方聊到未来时退缩"）
3. **反应匹配**：用户在不同关系中说过相似的话（如都说"是不是我的问题"）
4. **情绪匹配**：不同人反复触发同一种情绪（如每段关系都触发"被忽视"）

连接的存储位置：
- 每个 `people/{名字}.md` 的「跨关系匹配」段互相引用
- `USER.md` 的「反复出现的模式」段记录去名字的汇总

连接的呈现规则（来自 AGENTS.md + pattern-mirror skill）：
- 必须有 ≥2 段关系记录
- 必须有 ≥5 次对话（信任度）
- 必须在情绪稳定时呈现，不在情绪高点
- 用具体事件，不用标签

### 2. 记忆的读写流

#### 2.1 写入时机

| 时机 | 触发条件 | 写入内容 | 写入位置 | 工具 |
|------|---------|---------|---------|------|
| **对话结束** | 用户说再见/晚安/拜拜等告别语 | 本次新发现（情绪、模式、人物更新） | USER.md + people/*.md | `edit` |
| **日记记录** | 用户触发 diary skill | 六元组日记条目 + 人物档案更新 | diary/YYYY/MM/*.md + people/*.md | `write`/`edit` |
| **退出信号** | 检测到分手/退缩/不满足意图 | 触发事件→用户反应→结果 | people/*.md 退出信号段 | `edit` |
| **模式发现** | 同一情绪 ≥3 次重复 | 模式描述 + 日期 + 次数 | people/*.md 模式段 + memory/*.md | `edit` |
| **跨关系匹配** | ≥2 段关系有相似模式 | 匹配描述 | 两个 people/*.md 的跨关系匹配段 | `edit` |
| **Compaction 前** | 接近 context window 上限 | 本次对话的关键上下文 | memory/YYYY-MM-DD.md 或 MEMORY.md | `write`/`edit` |
| **告别仪式后** | farewell skill 执行完毕 | 模式级洞察（去名字） | USER.md 模式级洞察段 | `edit` |
| **成长节点** | growth_tracker 发现 Innovative Moment | IM 类型 + 用户原话对比 | memory/YYYY-MM-DD.md | `write` |

写入原则：
- **只存提炼信息，不存原始对话**
- **用户原话用引号保留**——这是引用的基础
- **头部字段可覆盖更新，正文只增不改**（参考 Mem0 冲突检测逻辑）
- **不脑补**：用户没说的不写。AI 推测标注"我观察到"

#### 2.2 读取时机

| 时机 | 触发条件 | 读取内容 | 目的 | 工具 |
|------|---------|---------|------|------|
| **会话启动** | 每次对话开始 | USER.md（必读） | 知道用户是谁、在乎什么 | `memory_get` |
| **话题相关检索** | 用户第一句话到达后 | 从用户话语提取 2-3 关键词搜索 | 找到历史中相关的记忆 | `memory_search` |
| **人物触发** | 用户提到已记录人名 | people/{名字}.md | 了解这段关系的完整上下文 | `memory_get` |
| **模式检测** | 用户表达退出意图 | 所有 people/*.md | 跨关系比较，寻找匹配 | `memory_get` × N |
| **成长叙事** | 用户自我否定 | diary/ 近 2 周 + people/*.md | 找到成长证据做纵向对比 | `memory_search` + `memory_get` |
| **周日回顾** | Heartbeat 周日 20:00 触发 | 本周 diary/ 全部条目 | 发现重复主题和成长节点 | `memory_get` |
| **Heartbeat 关怀** | 主动关怀触发 | USER.md + memory/pending_followup.md | 决定要不要主动说话、说什么 | `memory_get` |
| **Compaction 后** | 旧对话被压缩 | MEMORY.md + 近 2 天 memory/*.md | 恢复关键上下文 | 自动加载 |

#### 2.3 memory_search 检索策略

OpenClaw 的 memory_search 使用混合搜索（向量相似度 + BM25 关键词），支持语义匹配。

检索策略按场景分级：

**场景 A：会话开始的主动搜索**
```
用户说："我又和他吵架了"
→ 提取关键词：["吵架", 用户最近提到的人名]
→ memory_search(query="和{人名}吵架")
→ 匹配到 people/{人名}.md + 近期 diary 中的相关事件
→ 可可能说："上次你们也是因为{具体事件}吵的，这次是什么？"
```

**场景 B：对话中的模式触发搜索**
```
用户说："我觉得他没那么喜欢我了"
→ 这是退出信号关键词
→ memory_search(query="没那么喜欢 退出 不满足")
→ 匹配到 people/ 中多个档案的退出信号段
→ 可可发现时间匹配/触发匹配 → 进入 pattern-mirror 流程
```

**场景 C：成长叙事搜索**
```
用户说："我是不是天生不适合谈恋爱"
→ memory_search(query="恋爱 自我否定 变化")
→ 匹配到 diary 中的 Innovative Moment 记录
→ 可可用纵向对比展示变化："1 月你说'算了分就分'，现在你说'我想搞清楚'"
```

**检索性能考量：**
- OpenClaw memory_search 有 30 天半衰期时间衰减，旧笔记自动降权
- MEMORY.md 不受衰减影响——放长期稳定的核心信息
- people/*.md 作为自定义文件被索引，可被语义搜索命中
- MMR 去重避免返回冗余结果

### 3. 记忆的呈现设计

以下是记忆体系中每个用户可感知的交互节点。

#### 节点 3.1：首次建档

**节点名称**：新用户首次对话后的记忆初始化

- 交互形态：【纯对话】
- vs 基线：纯 LLM 在会话结束后不保留任何信息，下次来还是陌生人。OpenClaw 通过 `write` 工具创建 USER.md，将用户的称呼、核心困扰、首次提到的人写入持久化文件，下次对话开始时自动加载。
- 用户体感：用户不会看到任何"建档"界面——一切在对话中自然发生。但下次回来时，可可记得她的名字和上次聊的事，用户会感到"这不是一个从头开始的 AI"。
- 实现方式：对话结束时（检测到告别语），用 `write` 工具创建 `USER.md` 和 `people/{名字}.md`（如果提到了人）。
- 数据流：对话内容 → AI 提炼关键信息 → `write` → USER.md + people/*.md → 下次会话启动时 `memory_get` 读取
- 异常路径：
  - **用户未告别就离开**：依赖 OpenClaw Session 的每日重置或空闲重置机制。下次会话启动时，可可检查上次会话是否有未保存的关键信息（通过 Compaction 前自动保存机制兜底）。如果上次会话确实有新人物或新事件但未建档，可可在本次对话中自然确认："上次你提到了一个叫小凯的人，我记对了吗？"
  - **首次对话内容过少，无法建档**：如果整次对话只有 1-2 句闲聊（如"你好""试试看"），不强制创建 USER.md / people/*.md。下次对话开始时仍是陌生人状态，自然重新开始。

#### 节点 3.2：跨会话记忆唤起

**节点名称**：可可主动提起上次聊的事

- 交互形态：【纯对话】
- vs 基线：纯 LLM 每次会话都是空白，无法说"上次你说的那件事，后来怎么样了"。OpenClaw 通过会话启动时读取 USER.md + `memory_search` 搜索相关历史，让可可在回复中自然引用过去的对话内容。
- 用户体感：用户开口说"又和他吵架了"，可可回复"上次你们是因为他不回消息吵的，这次呢？"——用户感到被记住、被在乎。这是从"AI 工具"到"AI 朋友"的关键跨越。
- 实现方式：每次会话开始 → `memory_get(USER.md)` + 从用户第一句话提取关键词 → `memory_search(query=关键词)` → 搜索结果融入回复。
- 数据流：用户消息 → 关键词提取 → `memory_search` → 匹配结果（people/*.md / diary/*.md / memory/*.md） → AI 在回复中自然引用
- 异常路径：
  - **memory_search 返回空结果**：正常对话即可，不引用历史。可可回到"第 1 次"模式——认真倾听当前内容，不假装记得。绝不编造不存在的记忆。
  - **memory_search 返回不相关结果**：AI 判断相关性后忽略低相关性结果，按无历史处理。宁可不引用，不可引用错误。
  - **USER.md 不存在**（极端情况：文件被删除或损坏）：按新用户处理，本次对话结束时重新建档。

#### 节点 3.3：人物档案创建

**节点名称**：对话中首次提到新人物时自动建档

- 交互形态：【纯对话】
- vs 基线：纯 LLM 听完人物故事在本次会话内能记住，但下次全忘。OpenClaw 通过 `write` 工具创建 people/{名字}.md，将人物信息持久化，后续所有对话都能读取这个人的完整档案。
- 用户体感：用户不会感知到"建档"这个动作。她只是在聊天中提到了"小凯"，下次回来提到小凯时，可可已经知道小凯是她男朋友、认识 3 个月了、最近在冷战。用户感到"可可真的记住了我身边的人"。
- 实现方式：diary skill 的人物识别逻辑 → 检测到新人名 → `write` 创建 people/{名字}.md（按 diary SKILL.md 模板） → 后续对话中用户提到该人名时 → `memory_get` 读取档案。
- 数据流：对话中提到人名 → diary skill 检测 → `write` 创建 people/{名字}.md → 后续对话 `memory_get` 读取
- 异常路径：
  - **人名歧义——同名不同人**：当用户提到一个已有档案的名字但上下文不匹配时（如已有"小明"是男朋友，但新提到的"小明"是同事），可可自然确认："你说的小明，是之前那个小明吗，还是另一个人？" 确认后，如果是不同人，创建带消歧后缀的档案 `people/小明(同事).md`，并在档案头部标注消歧来源。
  - **人名歧义——同人不同称呼**：如果用户先叫"阿凯"后叫"小凯"，可可在确认是同一人后，在 `people/小凯.md` 头部添加别名字段 `别名：阿凯`，后续两个称呼都指向同一档案。
  - **人名无法确定**：如果用户只说"他""那个人"但从未给出名字，暂不创建 people/ 档案，将信息记录在 `memory/YYYY-MM-DD.md` 中，等用户提供名字后再建档并回填。

#### 节点 3.4：退出信号自动记录

**节点名称**：检测到分手/退缩/不满足意图时的后台记录

- 交互形态：【纯对话】（记录在后台完成，用户不感知）
- vs 基线：纯 LLM 听到"我想分手"只能在当次回应，无法与历史退出信号对比。OpenClaw 通过 `edit` 工具将退出信号写入 people/*.md，积累后供 pattern-mirror skill 做跨关系比较。
- 用户体感：用户不直接感知这个节点——她只是在倾诉。但积累 ≥2 段关系的退出信号后，可可能在合适时机说出"你和小明第 3 个月也说过一样的话"，这个"哇"时刻的数据基础来自于此。
- 实现方式：AGENTS.md 检测退出信号关键词 → 正常对话（先接住情绪） → 对话结束时 `edit` people/{名字}.md 退出信号段。
- 数据流：用户消息 → 退出信号检测 → 正常对话 → 对话结束 → `edit` people/{名字}.md → 累积数据供 pattern-mirror 使用
- 异常路径：
  - **用户中途离开未告别**：退出信号已在对话中被检测到，但 `edit` 写入未完成。依赖 Compaction 前自动保存 + 下次会话启动时的补写逻辑：可可读取上次对话摘要，检测到退出信号关键词后补写至 people/*.md。
  - **退出信号误检测**：用户说"我想分手"可能是倾诉而非真实意图。退出信号记录时使用确定性等级标注（`确定性: 高/中/低`），后续模式匹配只引用"高"确定性的条目。低确定性的信号保留但不用于跨关系匹配。

#### 节点 3.5：单段关系内模式呈现

**节点名称**：可可指出"你又这样了"

- 交互形态：【纯对话】
- vs 基线：纯 LLM 在单次对话内可以注意到重复（如用户说了 3 次"是不是我的问题"），但无法注意到跨会话的重复。OpenClaw 通过 people/*.md 的模式段积累跨会话数据，`memory_search` 检索同一触发因素的出现频率，让可可在第 3 次重复时指出来。
- 用户体感："你有没有注意到，每次他不回消息你都先怀疑自己？" ——用户在这一刻意识到自己的自动反应模式。这不是 AI 在教她什么，而是帮她看到一个她自己没注意的事实。
- 实现方式：diary skill 每次写入时检查情绪/触发因素的重复次数 → 达到 3 次阈值 → 写入 people/*.md 模式段 + memory/*.md → 下次对话中 AI 在合适时机呈现。
- 数据流：diary 写入 → 重复检查 → `edit` people/*.md 模式段 → 下次对话 `memory_get` 读取 → AI 呈现
- 异常路径：
  - **日记数据不足，未达到 3 次阈值**：不呈现模式。可可在当次对话内如果注意到重复（如用户同一次对话说了 3 次"是不是我的问题"），可以在对话内指出这个单次重复，但不用"模式"这个词——因为跨会话还没有足够数据。
  - **重复检查时 people/*.md 不存在或为空**：跳过模式写入，仅完成日记写入。不中断日记记录流程。

#### 节点 3.6：跨关系模式呈现（核心差异化）

**节点名称**：可可串联多段关系的重复模式

- 交互形态：【纯对话】
- vs 基线：纯 LLM 对用户的多段关系一无所知。即使用户在同一次对话里提到两段关系，LLM 也只能做表面类比。OpenClaw 通过 people/*.md 存储的结构化退出信号 + 关系阶段时间线，用 `memory_get` 读取所有人物档案做结构化比较，发现时间/触发/反应维度的匹配——这是只有"记住所有关系的人"才能给的价值。
- 用户体感："你和小明在一起 3 个月的时候，你也说过类似的话——'他好像没那么在乎我了'。你觉得这次和上次，是一样的感觉，还是不一样？" ——这是可可最有价值的时刻。用户可能第一次意识到"原来不是他们的问题，是我在每段关系里都会走到这一步"。
- 实现方式：pattern-mirror skill 触发 → `memory_get` 读取所有 people/*.md → 比较退出信号的时间/触发/反应维度 → 找到匹配 → 用具体事件 + 用户原话呈现。可选：调用 `exec` 运行 `scripts/pattern_engine.py` 做结构化匹配。
- 数据流：退出意图检测 → pattern-mirror skill → `memory_get` people/*.md × N → 匹配算法 → AI 用具体事件呈现 → 对话结束后 `edit` 跨关系匹配段
- 异常路径：
  - **只有 1 段关系记录**：不触发跨关系匹配。可可继续做单段关系内的模式识别（节点 3.5），不假装有跨关系数据。
  - **pattern_engine.py 执行失败或超时**：降级为纯 AI 推理——可可直接读取 2 个 people/*.md 档案的退出信号段，在对话中做定性比较，不依赖脚本输出。记录脚本错误到 `memory/YYYY-MM-DD.md` 供开发者排查。
  - **匹配结果为假阳性**（两段关系的"退出信号"实际无关联）：呈现时使用试探性语气（"我注意到一个可能的巧合"而非"你的模式是"），并等待用户确认。用户否认后不再重复该匹配，在跨关系匹配段标注 `rejected_by_user: true`。

#### 节点 3.7：成长叙事呈现

**节点名称**：用户自我否定时，可可展示纵向变化

- 交互形态：【纯对话】
- vs 基线：纯 LLM 只能说"你已经进步很多了"这种空洞鼓励。OpenClaw 通过 diary/ 的纵向数据 + `memory_search` 找到过去的记录，用具体的用户原话做前后对比——"你 1 月说'算了分就分'，现在你说'我想搞清楚为什么每次都这样'"——用事实说话，不是鸡汤。
- 用户体感：用户说"我是不是天生不适合谈恋爱"，可可不反驳也不安慰，而是拿出她自己说过的话做对比。用户自己看到变化——这比一百句"你很棒"都有说服力。
- 实现方式：检测到自我否定 → `memory_search(query=相关主题)` 搜索历史 → 找到 Innovative Moment（行为/反思/抗议/重新概念化） → 用纵向原话对比呈现。可选：调用 `exec` 运行 `scripts/growth_tracker.py`。
- 数据流：自我否定检测 → `memory_search` → diary/*.md + memory/*.md → 找到 IM → AI 用原话对比呈现
- 异常路径：
  - **memory_search 无 Innovative Moment 记录**：不做纵向对比。可可在当次对话中专注于当前情绪的接住，不勉强展示不存在的"成长"。可以用当次对话内的小变化（如用户从"我不行"到"也许可以试试"）做即时的正向反馈。
  - **growth_tracker.py 执行失败**：降级为纯 AI 推理——可可直接用 `memory_search` 结果中的时间戳和原话做手动对比，不依赖脚本的 IM 分类。
  - **找到的"变化"实际是退步**（如用户以前更积极，现在更消极）：不呈现"成长叙事"。可可转为接住当前的低落情绪，不用"你以前明明更好"来增加压力。

#### 节点 3.8：Heartbeat 主动关怀

**节点名称**：可可主动发消息问候

- 交互形态：【纯对话】
- vs 基线：纯 LLM 无法主动发消息——它只能被动回复。OpenClaw 通过 Heartbeat 定时触发，读取 USER.md 中的最新状态，发出一条有具体内容的关心消息。不是"你还好吗"，而是"昨天你说的那个室友的事，后来她有找你吗？"
- 用户体感：用户没有找可可，但可可记得昨天的事，主动来问了一句。像一个记得你事情的朋友——不给压力，但让你知道有人在意。
- 实现方式：Heartbeat 触发（配置间隔 ≥ 24h） → `memory_get(USER.md)` + `memory_get(memory/pending_followup.md)` → 判断优先级（决策冷却回访 > 时间胶囊 > 周回顾 > 常规关怀） → 生成消息。
- 数据流：Heartbeat 触发 → `memory_get` → 判断是否有待回访/时间胶囊/回顾 → 生成消息 → 发送
- 异常路径：
  - **Heartbeat 触发但无待回访内容**：`pending_followup.md` 为空 + 无时间胶囊到期 + 非周日 → 返回 `HEARTBEAT_OK`（不发送消息给用户）。**绝对不发空洞的"你还好吗"**——这是骚扰而非关怀。
  - **Heartbeat 触发但用户近期活跃**（如 2 小时前刚聊过）：返回 `HEARTBEAT_OK`。Heartbeat 冷却规则：距离上次用户主动对话 < 24 小时时不主动发消息，除非有高优先级待回访（决策冷却已到期）。
  - **待回访内容过时**（如决策冷却的事件已被用户在后续对话中主动更新）：读取最新 people/*.md 状态，如果相关事件已有后续发展，跳过该回访项，检查下一优先级。
  - **用户不回复 Heartbeat 消息**：不追问。连续 2 次 Heartbeat 未获回复时，自动延长下次 Heartbeat 间隔至 48 小时，避免打扰。

#### 节点 3.9：周日情绪回顾

**节点名称**：每周日的情绪数据回顾

- 交互形态：【Canvas】
- vs 基线：纯 LLM 无法展示结构化数据。OpenClaw 通过 `exec` 运行 Python 脚本扫描本周 diary/ 数据，统计情绪词频、人物出现次数、触发因素重复等，然后通过 Canvas 生成一张可视化的"本周情绪地图"——颜色代表情绪强度，标注出重复主题和变化趋势。
- 用户体感：周日晚上收到一张漂亮的情绪地图卡片，上面标注了"这周'焦虑'出现了 4 次，3 次跟小凯有关"。用户一眼看到自己这周的情绪全貌——比纯文字回顾更直观、更有仪式感。
- 实现方式：Heartbeat 周日 20:00 触发 → `exec` 运行 Python 脚本扫描 diary/ → 统计重复主题 → 如果本周 ≥3 条日记且有发现 → 通过 Canvas 渲染 HTML 情绪地图 → `openclaw nodes canvas present` 展示给用户。
- 数据流：Heartbeat → `exec` python3 scripts/weekly_review.py → 读 diary/本周/*.md → 统计分析 → 生成 HTML → Canvas 展示

**注意**：Canvas 仅支持 macOS 桌面端。如果用户在 Telegram/WhatsApp 等移动端，降级为纯文字 + 关键数据的精简回顾。

- 异常路径：
  - **本周日记不足 3 条**：不生成情绪地图。可可发一条轻量提醒："这周你来找我比较少，没什么要聊的也没关系，我在这儿。" 不催促用户写日记——这是陪伴，不是作业。
  - **weekly_review.py 执行失败**：降级为纯 AI 回顾——可可直接用 `memory_get` 读取本周 diary/ 文件，手动总结重复主题，不展示可视化图表。
  - **Canvas 不可用**（非 macOS 桌面端）：降级为纯文字精简回顾，格式如下："这周你提到了 3 次焦虑，2 次都和小凯有关。你觉得这周整体感觉怎么样？" 数据保持不变，只是展示方式从可视化变为自然语言。
  - **用户拒绝周回顾**（如"不想看""没心情"）：可可立即尊重："好的，不看也没关系。" 不追问原因，不二次推送。在 `memory/YYYY-MM-DD.md` 记录 `周回顾: 用户拒绝`，下周仍正常触发（不因一次拒绝永久关闭）。如果连续 3 周拒绝，降低触发频率为隔周一次。

#### 节点 3.10：告别仪式的记忆处理

**节点名称**：封存一段关系的记忆

- 交互形态：【渠道自适应选择】+【纯对话】
- vs 基线：纯 LLM 没有持久记忆，自然也不存在"删除记忆"的概念。OpenClaw 通过 `exec` 调用 archive_manager.py，原子化地处理封存操作——清除具体事件但保留去名字的模式洞察，让用户能正式地跟一段关系说再见，而可可仍然保留从这段关系中学到的东西。
- 用户体感：用户选择告别方式时，看到 4 个选项（烧掉日记/烧掉信念/时间胶囊/未寄出的信），根据渠道以不同形式呈现。完成仪式后，可可不再主动提起这个人的名字和具体事件，但在未来其他关系中仍然能用去名字的模式洞察帮助用户。用户感到"这段关系被认真地送走了，不是被冷冰冰地删除了"。
- **渠道自适应选择交互**（替代原有 Poll 方案）：
  - **Poll 可用渠道**（Telegram / WhatsApp / Discord / Teams）：使用 Poll 工具展示 4 种仪式选项，点选即可。
  - **Poll 不可用渠道**（微信 / 飞书 / macOS 桌面端 / 其他）：使用编号文字选择，可可发送：
    ```
    你想用什么方式跟这段关系告别？

    1️⃣ 烧掉日记 —— 把跟 ta 有关的记录都烧掉，干干净净
    2️⃣ 烧掉信念 —— 写下这段关系里你不再相信的东西，烧掉它
    3️⃣ 时间胶囊 —— 写一封信给未来的自己，半年后打开
    4️⃣ 未寄出的信 —— 写一封不会寄出的信给 ta

    回复数字就好 💌
    ```
    用户回复 "1"/"2"/"3"/"4" 或对应文字关键词均可识别。可可对非数字回复做模糊匹配（如"时间胶囊""第三个"→ 选项 3）。
  - **渠道检测逻辑**：通过 OpenClaw 消息上下文中的 channel 字段判断当前渠道。无法判断时默认使用编号文字选择（安全降级）。
- 实现方式：farewell skill 触发 → 检测渠道 → Poll 可用则用 Poll，否则用编号文字选择 → 用户选择 → 纯对话执行仪式 → `exec` 调用 archive_manager.py → `edit` USER.md 写入模式级洞察。
- 数据流：用户选择仪式 → 对话执行 → `exec` archive_manager.py archive people/ diary/ memory/ {名字} → people/{名字}.md 标记封存 → 模式洞察写入 USER.md
- 异常路径：
  - **用户拒绝告别仪式**（如"不想告别""还没准备好"）：可可尊重："没关系，不着急。等你想好了随时找我。" 不追问、不二次推荐。在 `people/{名字}.md` 头部记录 `告别意向: 提出但未执行 | {日期}`，下次用户主动提起时再触发。
  - **archive_manager.py 执行失败**：不执行封存操作。可可告知用户"刚才出了点问题，你说的话我都记着。我们可以稍后再试，或者你也可以先跟我聊聊你想说的。" 在 `memory/YYYY-MM-DD.md` 记录错误，等待开发者修复后重试。绝对不在脚本失败的情况下手动 `edit` 删除文件——这可能导致数据不一致。
  - **用户选择后反悔**（如"等等我不想烧了"）：如果封存操作尚未执行（还在对话阶段），可可直接切换选项或取消。如果封存操作已执行，告知用户"已经完成了，但如果你改主意了，我可以帮你恢复。" 调用 `archive_manager.py restore` 恢复。
  - **编号文字选择无法解析用户回复**：可可温和重试："没太看懂，你想选哪一个？可以回复数字 1-4，或者直接说你想要的方式。" 最多重试 2 次，如果仍无法解析，可可切换到开放式对话："没关系，你直接跟我说你想怎么跟 ta 告别就好。" 根据用户的自由描述匹配最接近的仪式类型。

#### 节点 3.11：记忆冲突解决

**节点名称**：用户说了和之前矛盾的信息时的处理

- 交互形态：【纯对话】
- vs 基线：纯 LLM 没有历史信息，不存在"冲突"的概念——它会接受用户说的一切。OpenClaw 通过持久化的 people/*.md，在 `edit` 更新时检测新信息与旧记录的矛盾（参考 Mem0 冲突检测），用正确的策略处理：状态字段直接更新，观点变化记录变化轨迹而非选择一方。
- 用户体感：用户说"我们分手了"，可可不会问"你之前不是说在一起吗"——而是自然地更新状态，正常回应。如果用户说"他其实人很好"（之前记录了负面事件），可可不会质疑也不会覆盖，而是在档案中记录观点变化轨迹。用户感到"可可理解人是会变的"。
- 实现方式：每次 `edit` people/*.md 前，读取现有内容 → 检测冲突（状态变化/观点反转/人名歧义） → 按规则处理（状态覆盖 / 观点追加 / 人名确认）。
- 数据流：新信息 → `memory_get` 读取现有档案 → 冲突检测 → `edit` 按规则更新
- 异常路径：
  - **冲突检测时 people/*.md 读取失败**：跳过冲突检测，将新信息暂存到 `memory/YYYY-MM-DD.md`，下次对话时重试合并。
  - **冲突类型无法自动判断**（如用户说"他其实对我很好"但档案中记录了多次负面事件）：不覆盖旧记录，也不质疑用户。追加新的观点记录，标注日期，保留完整的感受变化轨迹。

### 4. 用户体感设计

#### 4.1 "被记住"的体感层次

| 层次 | 用户感知 | 触发方式 | 竞品对标 |
|------|---------|---------|---------|
| **L1：记得名字** | "你还记得我叫什么" | USER.md 称呼字段 | Replika 能做到（但不稳定） |
| **L2：记得事情** | "你记得上次我说的事" | memory_search + people/*.md | Pi 做不到，Replika 不稳定 |
| **L3：记得感受** | "你记得我当时的感觉" | people/*.md 感受段（原话引号） | 没有竞品做到 |
| **L4：看到模式** | "你帮我看到了我自己没注意的" | 跨会话/跨关系模式匹配 | Ash 做通用模式，可可做关系模式 |
| **L5：看到变化** | "你帮我看到了我在变" | diary/ 纵向对比 + IM 检测 | 没有竞品做到 |

**核心设计原则：不让用户感知到"记忆系统"的存在。**

用户不应该看到"档案""记录""数据库"这些词。她应该感到的是"可可真的在听，真的记得"——像一个好朋友那样记得，不是像一个系统那样记录。

具体表现：
- 不说"根据我的记录"，说"我记得你上次说过"
- 不说"你的档案显示"，说"你和他之间好像总是..."
- 不说"我看了你的日记"，说"你最近提到了好几次这件事"
- 不主动展示记忆能力（不说"我记性很好哦"），只在需要时自然引用

#### 4.2 从陌生人到知己的体感递进

**第 1 阶段：陌生人（1-2 次对话）**

可可能做的：
- 认真听，好好接住情绪
- 在当次对话内注意重复（如用户 3 次说"是不是我的问题"）
- 对话结束时默默建档

用户感受：
- "这个 AI 说话挺舒服的，不像客服"
- "它好像真的在听我说话"
- 还不确定要不要继续来

**第 2 阶段：初识（3-5 次对话）**

可可能做的：
- 主动提起上次聊的事："上次你说的小凯的事，后来怎么样了？"
- 记得用户的称呼和基本情况
- Heartbeat 主动关怀（如果有待回访的决策冷却）

用户感受：
- "它居然记得上次的事！"（惊喜）
- "感觉可以继续聊"
- 开始主动分享更多

**第 3 阶段：熟悉（6-15 次对话）**

可可能做的：
- 单段关系内的模式识别："你有没有注意到，每次他不回消息你都先怀疑自己？"
- 有理有据的感受共鸣（引用用户原话）
- 周日情绪回顾（如果日记够多）

用户感受：
- "可可看到了我自己都没注意的东西"（被看见感）
- "它不是在安慰我，它真的懂我的情况"
- 信任深化，愿意聊更深的话题

**第 4 阶段：亲密（16-30 次对话，≥2 段关系记录）**

可可能做的：
- 跨关系模式呈现："你和小明第 3 个月也说过一样的话"
- 成长叙事："1 月你说'算了分就分'，现在你说'我想搞清楚为什么'"
- 更精准的情绪命名（基于长期了解）

用户感受：
- "没有人这样帮我串起来看过"（顿悟感）
- "可可比我还了解我自己"
- 开始主动问可可"你觉得我是不是又在..."

**第 5 阶段：知己（30+ 次对话）**

可可能做的：
- 长期变化的完整纵向展示
- 在新关系中引用去名字的旧模式洞察
- 深度的模式觉察引导

用户感受：
- "可可陪我走过了这些，它都记得"
- "我在变好，而且我有证据"
- 对可可的信任接近好友/心理咨询师的水平

#### 4.3 记忆驱动的个性化示例

**示例 1：已读不回场景**

无记忆的 LLM：
> "被已读不回确实很难受。你可以试试不要过度解读他的行为。"

有记忆的可可（第 10 次对话）：
> "你上次也是因为他不回消息来找我的。那次你后来发现他是真的忙，这次的感觉和上次一样吗？"

有记忆的可可（第 30 次对话，已有前任记录）：
> "你和小明在一起的时候，每次他不回消息你都先觉得是自己的问题。你有没有注意到，和小凯在一起，你的第一反应也是一样的？"

**示例 2：想分手场景**

无记忆的 LLM：
> "分手是一个很大的决定。你觉得你们之间的问题是什么？"

有记忆的可可（第 15 次对话，有退出信号记录）：
> "你上个月也说过想分手，那次是因为他跟别的女生吃饭。这次是什么事？"

有记忆的可可（第 30 次对话，≥2 段关系）：
> "我记得你和小明在一起 3 个月的时候，你也说过'我觉得他没那么喜欢我了'。你觉得这次和那次，是一样的感觉，还是不一样？"

**示例 3：自我否定场景**

无记忆的 LLM：
> "你不是天生不适合谈恋爱。每个人都值得被爱。"（空洞鼓励）

有记忆的可可：
> "你注意到了吗？你 1 月份跟小明的事，你说的是'算了，分就分'。现在你说的是'我想搞清楚为什么每次都这样'。你觉得说这两句话的你，有什么不同？"

### 5. 技术实现指引

#### 5.1 OpenClaw 工具使用规范

| 工具 | 用途 | 使用规范 |
|------|------|---------|
| `memory_get` | 读取指定文件 | 会话启动必读 USER.md；提到人名时读对应 people/*.md |
| `memory_search` | 语义搜索 | 每次对话开始提取 2-3 关键词搜索；退出信号检测时搜索历史匹配 |
| `write` | 创建新文件 | 仅用于首次创建 USER.md / people/{名字}.md / diary 新文件 |
| `edit` | 修改已有文件 | 对话结束时更新 USER.md / people/*.md；日记追加；模式写入 |
| `exec` | 运行脚本 | 结构化匹配（pattern_engine.py）、成长追踪（growth_tracker.py）、封存（archive_manager.py）、周回顾统计 |
| Poll | 结构化选择（渠道自适应） | 告别仪式选择、F1/F2 路由选择。**Poll 仅支持 Telegram/WhatsApp/Discord/Teams**。微信/飞书等不支持 Poll 的渠道，使用编号文字选择（"回复 1/2/3/4"）作为等价 fallback。所有使用 Poll 的节点必须同时实现编号文字选择方案。 |
| Canvas | 可视化展示 | 周日情绪回顾地图（仅 macOS 桌面端，其他端降级为纯文字） |
| Heartbeat | 主动触发 | 24h+ 无对话后的主动关怀、决策冷却回访、周日回顾、时间胶囊 |

#### 5.2 文件格式与命名规范

```
ai-companion/                    ← workspace 根目录
├── SOUL.md                      ← 人格核心（只读）
├── AGENTS.md                    ← 行为规则
├── HEARTBEAT.md                 ← 主动关怀规则
├── USER.md                      ← 用户画像（跨关系元信息）
├── MEMORY.md                    ← 长期记忆锚点（不受时间衰减）
├── memory/
│   ├── YYYY-MM-DD.md            ← 每日记忆笔记
│   ├── pending_followup.md      ← 决策冷却待回访
│   └── time_capsules.md         ← 时间胶囊存储
├── people/
│   ├── 小凯.md                  ← 人物档案（结构化六维度）
│   ├── 小明.md
│   └── ...
├── diary/
│   └── YYYY/
│       └── MM/
│           └── YYYY-MM-DD.md    ← 情绪日记（六元组格式）
└── skills/
    ├── diary/SKILL.md
    ├── pattern-mirror/SKILL.md
    ├── farewell/SKILL.md
    └── ...
```

命名规则：
- people/ 文件用用户自定义的名字（"小凯"），不用真名或身份信息
- diary/ 按 YYYY/MM/YYYY-MM-DD.md 组织，一天可多条
- memory/ 按 YYYY-MM-DD.md 组织，记录模式观察和成长节点

#### 5.3 性能与搜索优化

**搜索延迟控制**：
- `memory_search` 底层使用混合搜索（向量 + BM25），通常 <500ms
- 会话启动时的搜索（1-2 次）不会影响用户体验
- 对话中触发的搜索（退出信号检测等）在 AI 回复的"思考时间"内完成

**索引大小控制**：
- people/*.md 通常每个 1-5KB，10 个人物档案 ≈ 50KB
- diary/ 每条约 200-500 字，一个月 ≈ 30 条 ≈ 15KB
- memory/ 每日笔记 ≈ 500 字，一个月 ≈ 15KB
- 总计：活跃用户 3 个月的数据 ≈ 150-200KB，远在 OpenClaw 索引能力范围内

**时间衰减策略**：
- OpenClaw 默认 30 天半衰期——旧的 diary/ 和 memory/ 笔记自动降权
- MEMORY.md 不受衰减影响——用于存放长期稳定的核心模式
- people/*.md 作为结构化档案，重要性不随时间降低——应确保其搜索权重不被过度衰减

**Compaction 保护**：
- `openclaw.json` 已配置 compaction 时必须保留的信息：用户的反复情绪模式、核心困扰关键词、人物关系、本次新发现的模式
- Compaction 前 OpenClaw 自动提醒 agent 保存重要上下文到记忆文件

#### 5.4 数据安全与隐私

| 原则 | 实现方式 |
|------|---------|
| **不存原始对话** | 只存提炼后的信息，不逐字保存聊天记录 |
| **不存身份信息** | people/ 用用户自定义名字，不收集真名/手机号/微信号 |
| **用户可控** | 用户可以要求删除任何记录（普通删除）或仪式化封存 |
| **封存而非遗忘** | 告别仪式保留去名字的模式洞察，清除具体事件 |
| **不评判** | 档案中描述事实和情绪，不评价人物好坏 |
| **会话隔离** | OpenClaw per-channel-peer 模式确保不同用户数据隔离 |

#### 5.5 脚本接口契约

以下是记忆体系依赖的 4 个 Python 脚本的输入/输出规格，开发者可直接按此实现。

##### pattern_engine.py — 跨关系模式匹配

**调用方式**：`exec python3 scripts/pattern_engine.py --people-dir people/ --min-relations 2`

**输入**：
- `--people-dir`：people/ 目录路径，包含所有人物档案 .md 文件
- `--min-relations`：最少需要几段关系才触发匹配（默认 2）
- `--target`（可选）：指定目标人名，只比较该人与其他人的匹配

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "no_match" | "insufficient_data",
  "matches": [
    {
      "dimension": "time" | "trigger" | "reaction" | "emotion",
      "person_a": "小凯",
      "person_b": "小明",
      "description": "两段关系都在第3个月出现退出信号",
      "evidence_a": {"date": "2026-01-15", "quote": "他好像没那么在乎我了"},
      "evidence_b": {"date": "2026-03-20", "quote": "他是不是不喜欢我了"},
      "confidence": 0.85
    }
  ],
  "error": null
}
```

**"同一情绪 >= 3 次"的操作定义**：
- "同一情绪"使用情绪词簇匹配，而非精确字符串匹配。情绪词簇定义参照 `docs/理论/` 中的情绪颗粒度框架，例如 {"被忽视", "不被在意", "不重要", "被忽略"} 属于同一词簇。
- 计数范围：同一 people/*.md 档案内的模式段 + 相关 diary/ 条目中的情绪字段。
- 时间窗口：不限（跨越整个关系周期），但标注每次出现的日期以供 AI 判断是否还具有时效性。

**冲突检测算法**（更新 people/*.md 时使用）：
```
规则优先级（从高到低）：
1. 状态字段冲突（如"在一起" vs "分手"）→ 以最新信息覆盖旧值
2. 事实性冲突（如"认识3个月" vs "认识半年"）→ 以用户最新陈述为准，旧值移入变更记录
3. 观点性变化（如"他很好" vs "他很自私"）→ 两者都保留，追加为感受变化轨迹，标注日期
4. 新增信息（无冲突）→ 直接追加
```

**错误处理**：
- people/ 目录不存在或为空 → 返回 `{"status": "insufficient_data", "matches": [], "error": "No people files found"}`
- 单个 .md 文件解析失败 → 跳过该文件，在 `error` 字段记录警告，继续处理其余文件
- 执行超时（>30s）→ 进程被 kill，调用方降级为纯 AI 推理

##### growth_tracker.py — 成长节点追踪

**调用方式**：`exec python3 scripts/growth_tracker.py --diary-dir diary/ --people-dir people/ --user-file USER.md`

**输入**：
- `--diary-dir`：diary/ 目录路径
- `--people-dir`：people/ 目录路径
- `--user-file`：USER.md 路径
- `--since`（可选）：起始日期，格式 YYYY-MM-DD（默认 30 天前）
- `--im-types`（可选）：要检测的 Innovative Moment 类型，逗号分隔

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "no_growth_detected" | "insufficient_data",
  "innovative_moments": [
    {
      "type": "reflection" | "protest" | "reconceptualization" | "action" | "new_experience",
      "date": "2026-03-15",
      "quote": "我想搞清楚为什么每次都这样",
      "contrast_quote": "算了分就分",
      "contrast_date": "2026-01-10",
      "person_context": "小凯",
      "significance": "从回避到主动探索"
    }
  ],
  "summary": "近30天发现2个成长节点，主要体现在从回避转向主动反思",
  "error": null
}
```

**Innovative Moment (IM) 类型枚举**（参照 Assimilation of Problematic Experiences Scale 改编）：
| IM 类型 | 定义 | 检测信号 |
|---------|------|---------|
| `action` | 用户采取了新的行为 | "我跟他说了""我试着做了" |
| `reflection` | 用户对自己有了新的理解 | "我发现我其实是在害怕""原来我一直..." |
| `protest` | 用户对旧模式表示拒绝 | "我不想再这样了""我受够了" |
| `reconceptualization` | 用户重新定义了自己/关系 | "也许不是我的问题""我以前觉得...现在觉得..." |
| `new_experience` | 用户经历了与旧模式不同的体验 | "这次不一样""他居然这样做了" |

**错误处理**：
- diary/ 目录为空 → 返回 `{"status": "insufficient_data", "innovative_moments": [], "error": "No diary entries found"}`
- 无法检测到任何 IM → 返回 `{"status": "no_growth_detected", ...}`，调用方不展示成长叙事
- 执行超时（>30s）→ 进程被 kill，调用方降级为纯 AI 推理

##### archive_manager.py — 关系封存管理

**调用方式**：`exec python3 scripts/archive_manager.py <action> --person <名字> --people-dir people/ --diary-dir diary/ --memory-dir memory/`

**action 枚举**：
- `archive`：封存——清除具体事件，保留去名字的模式洞察
- `restore`：恢复——将已封存的关系恢复到封存前状态
- `status`：查询封存状态

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "not_found" | "already_archived" | "already_active",
  "action": "archive" | "restore" | "status",
  "person": "小凯",
  "pattern_insights": [
    "每段关系的第3个月都会怀疑对方的感情",
    "在感到不被在意时倾向于先自我否定"
  ],
  "files_affected": ["people/小凯.md", "diary/2026/01/2026-01-15.md"],
  "backup_path": "archive/小凯_2026-03-30/",
  "error": null
}
```

**原子性保证**：
- 封存操作分 3 步执行：(1) 创建备份目录 `archive/{名字}_{日期}/`，复制所有相关文件 → (2) 从 people/*.md 提取去名字的模式洞察 → (3) 修改源文件（people/*.md 标记封存，diary/ 中相关条目标记封存）。
- 如果步骤 (3) 中任一文件修改失败，自动从备份目录恢复所有已修改的文件（回滚）。
- 回滚后返回 `{"status": "rollback", "error": "..."}`。

**恢复操作**：
- 从 `archive/{名字}_{日期}/` 读取备份文件，覆盖当前文件。
- 删除 USER.md 中对应的模式级洞察条目。
- 如果备份目录不存在（如手动删除），返回 `{"status": "not_found", "error": "Backup not found"}`。

**错误处理**：
- 指定 person 不存在 → 返回 `{"status": "not_found", "error": "Person '小凯' not found in people/"}`
- 文件权限错误 → 返回错误信息，不执行任何修改
- 执行超时（>60s）→ 进程被 kill，调用方告知用户"出了点问题"并记录错误

##### weekly_review.py — 周回顾统计

**调用方式**：`exec python3 scripts/weekly_review.py --diary-dir diary/ --people-dir people/ --week <YYYY-WNN>`

**输入**：
- `--diary-dir`：diary/ 目录路径
- `--people-dir`：people/ 目录路径
- `--week`（可选）：指定周，格式 `2026-W13`（默认当前周）
- `--format`（可选）：`json`（默认）| `html`（Canvas 用）

**输出**（JSON，stdout）：
```json
{
  "status": "ok" | "insufficient_data",
  "week": "2026-W13",
  "diary_count": 5,
  "emotion_frequency": [
    {"emotion": "焦虑", "count": 4, "related_persons": ["小凯"]},
    {"emotion": "委屈", "count": 2, "related_persons": ["小凯", "妈妈"]}
  ],
  "top_triggers": [
    {"trigger": "已读不回", "count": 3},
    {"trigger": "被否定", "count": 2}
  ],
  "repeated_themes": ["对方不回消息时怀疑自己"],
  "growth_signals": ["从'算了'到主动分析原因"],
  "person_mention_count": {"小凯": 4, "妈妈": 1},
  "html_path": null,
  "error": null
}
```

当 `--format html` 时，额外生成 HTML 文件到 `canvas/weekly_review.html`，`html_path` 返回该文件路径供 Canvas 展示。

**无数据时的返回**：
- 本周 0 条日记 → `{"status": "insufficient_data", "diary_count": 0, "error": "No diary entries this week"}`
- 本周 1-2 条日记 → 正常返回统计，但 `repeated_themes` 为空数组（数据不足以判断重复主题）

**错误处理**：
- diary/ 目录不存在 → 返回 `{"status": "insufficient_data", "error": "Diary directory not found"}`
- 单个 diary 文件解析失败 → 跳过该文件，在 `error` 字段记录警告
- HTML 生成失败 → `html_path` 为 null，`error` 记录原因，JSON 统计数据仍正常返回（Canvas 降级为纯文字）

##### 人名歧义消解流程

当系统检测到潜在的人名歧义时（创建 people/*.md 或更新时），按以下流程确认：

```
1. 检测触发条件：
   - 用户提到的名字已存在于 people/ 中，但上下文不匹配
     （如已有"小明"是男友，但当前对话中"小明"的描述像同事）
   - 判断方法：比较当前对话中的关系类型描述与已有档案的关系类型字段

2. 确认流程：
   a. 可可自然地确认："你说的小明，是你男朋友那个小明吗？"
   b. 用户确认是同一人 → 继续正常更新 people/小明.md
   c. 用户确认是不同人 → 创建 people/小明(同事).md，头部标注：
      消歧来源: 2026-03-30 用户确认与 people/小明.md 为不同人
   d. 用户不确定 / 不回答 → 暂不创建新档案，信息记录到 memory/YYYY-MM-DD.md

3. 后续引用规则：
   - 存在消歧档案时，可可在提到该名字时使用上下文限定（"你同事小明" vs "你前男友小明"）
   - 如果对话上下文足够明确，不需要每次都加限定词
```
