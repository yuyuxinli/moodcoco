## 2. F1 情绪急救 — 从爆炸降到能呼吸

### 2.1 竞品参考

| 竞品 | 做法 | 我们采纳的 |
|------|------|-----------|
| **Woebot** | CBT 框架三步走：共情确认→认知挑战→行为激活。检测到情绪激烈时不做认知干预，先做 grounding | 先接住再干预的顺序；grounding 优先于认知工作 |
| **Wysa** | 分层响应：低风险→CBT 工具；中风险→引导式冥想；高风险→危机协议。用情绪强度 1-10 量表做路由 | 情绪强度路由机制 |
| **Pi (Inflection)** | 不使用结构化框架，纯自然对话式共情。核心是"先理解再回应"，回复中先复述用户的情绪再延伸 | 自然对话式共情，不像治疗师 |
| **Rosebud CARE** | 开源危机评估框架，4 维度评估：信号识别、干预质量、伤害预防、多轮鲁棒性。测试发现 86% 的模型无法识别间接自杀暗示 | 危机检测的评估方法论；用 CARE 的 5 个测试场景做上线前基准测试 |
| **Wysa** | 82% 危机检测率，NLP 关键词+上下文检测 + PHQ/GAD 标准量表 + 始终可见的 SOS 按钮 | 多层检测机制（关键词→上下文→量表） |

## 3. F2 信号解读 — 打断焦虑螺旋

### 3.1 竞品参考

| 竞品 | 做法 | 我们采纳的 |
|------|------|-----------|
| **Woebot** | CBT 认知三角：情境→想法→情绪。用苏格拉底式提问引导用户区分自动化思维和事实 | 苏格拉底式提问法；区分事实/想象的框架 |
| **Wysa** | "Thought Checker" 工具：列出事实→列出想象→评估证据→生成替代想法 | 结构化的事实/想象分离流程 |
| **Ash** | 专注关系场景的信号解读：用户发来对方的消息，AI 给出多种可能的解读（不是唯一答案） | 多种解读而非单一结论；关系场景专精 |
| **Mosaic** | 用 NLP+BERT 检测 Gottman 的"末日四骑士"（批评/蔑视/防御/冷战）在实际聊天消息中的出现 | 结构化的关系信号检测维度 |
| **Jenova** | 对话前先问"你想让我纯听，还是想要诚实的反馈？"再决定走 F1 还是 F2 路线 | 模式检测问题：让用户自己选择共情还是分析 |

### 3.3 差距与改动

**差距 1：缺少"多种解读"的明确规则。** 用户问"他这句话什么意思"时，可可应给出 2-3 种可能解读，而非一种。

改动：AGENTS.md Step 2 增强。

```markdown
## Step 2 增强：信号解读的多解读原则

当用户拿一条消息/一个行为来问"什么意思"时：
1. 先确认事实（"他具体说了什么？什么时候说的？"）
2. 给出 2-3 种可能的解读，从善意到恶意排列
3. 引导用户判断哪种更符合整体背景
4. 绝不给唯一结论

示例：
✗ "他可能是在冷暴力你"
✓ "这条消息可能有几种意思：一是他真的很忙；二是他不太知道怎么回应你的情绪；三是他确实在回避。结合你了解的他，你觉得哪种更像？"
```

**差距 1.5：缺少"共情还是分析"的模式检测。**（参考 Jenova）

改动：AGENTS.md 新增路由规则：

```markdown
## F1/F2 路由检测

当用户带着一个关系事件来找可可，但不确定用户需要什么时：
- 先接住情绪（1-2 句共情确认）
- 然后自然地问一句意图检测问题：
  "你想让我纯听你说，还是想一起分析一下？"
- 用户选"纯听" → 走 F1 情绪急救（Step 1）
- 用户选"分析" → 走 F2 信号解读（Step 2）
- 用户没有明确选择 → 默认走 F1，在情绪稳定后自然过渡到 F2
```

**差距 2：图片分析能力未接入（P1 依赖）。**

设计：当 OpenClaw 图片分析能力可用时，AGENTS.md 增加路由规则：

```markdown
## 图片消息路由

用户发送图片时：
- 聊天截图 → 进入 Step 2 信号解读流程，先读图再引导
- 其他图片 → 正常对话，不强制解读
```

当前实现：跳过，等图片分析 provider 就绪后配置 `openclaw.json` 启用。

## 5. F4 跨关系模式追踪 — 看到自己的模式

### 5.1 竞品参考

| 竞品 | 做法 | 我们采纳的 |
|------|------|-----------|
| **无直接竞品** | 目前没有 AI companion 产品做跨关系模式追踪。**这是心情可可的蓝海（blue ocean）。** | — |
| **Ash** | 基于 Qwen3-235B 在 50K 真实治疗 session 上训练，生成"周度洞察"串联当前事件与历史模式。奖励模型权重：临床专家评分（最高）> 用户行为信号 > LLM-as-Judge | 借鉴"周度洞察"格式做周期性模式回顾（S5）|
| **BetterHelp/Talkspace** | 治疗师手动在 session notes 中追踪跨关系模式，但没有自动化工具 | 人工模式追踪的逻辑可以自动化 |
| **Replika Memory Graph** | 虽然不做跨关系分析，但其实体提取 + 关系图谱的数据结构可以参考 | 实体图谱思路 |
| **心理治疗文献** | Mikulincer & Shaver (2007) 的依恋激活模式具有跨关系一致性，可作为模式匹配的理论框架 | 退出触发器 + 依恋模式作为匹配维度 |

### 5.2 模式匹配引擎设计（S4 跨关系模式档案）

新增 `skills/diary/scripts/pattern_engine.py`：

```python
"""
pattern_engine.py — 跨关系模式匹配引擎
读取所有 people/*.md，提取退出信号，跨文件匹配相似模式。
由 diary skill 在每次对话结束时调用。
"""

# 核心功能：
# 1. parse_people_files(people_dir) -> list[dict]
#    解析所有 people/*.md，提取关系阶段 + 退出信号 + 模式
#
# 2. find_cross_patterns(people_data) -> list[dict]
#    跨文件匹配相似模式：
#    - 时间模式：多段关系在相似时间节点（第 N 个月）出现退出信号
#    - 触发模式：多段关系被相似事件触发退出（对方聊未来/热情衰减/日常摩擦）
#    - 反应模式：用户在不同关系中做出相似反应（退缩/追逐/不满足）
#
# 3. match_current_to_history(current_event, people_data) -> list[dict]
#    当前事件与历史退出信号的匹配
#    返回：匹配到的历史事件 + 匹配维度 + 相似度描述
#
# 4. update_cross_patterns(people_dir, patterns) -> None
#    将发现的跨关系模式写回 people/*.md 的"跨关系匹配"段

# 匹配维度：
MATCH_DIMENSIONS = [
    "timing",      # 时间模式（第 N 个月）
    "trigger",     # 触发事件相似
    "reaction",    # 用户反应相似
    "outcome",     # 结果相似（分手/冷战/和好）
]

# 匹配阈值：至少 2 段关系有相似模式才报告
MIN_RELATIONSHIPS_FOR_PATTERN = 2
```

### 5.3 模式急救对话设计（S1）

新增 `skills/pattern-mirror/SKILL.md`：

```markdown
# Pattern Mirror — 模式急救对话

## 触发条件
用户表达分手/退出/退缩意图，且 people/ 中有 ≥2 段关系记录。

## 流程

### Phase 1：接住情绪（必须先完成）
走 AGENTS.md Step 1 的情绪急救流程。
不在情绪高点呈现模式。
判断标准：用户从"爆发"转向"叙述"（从感叹句变陈述句）。

### Phase 2：调出历史
调用 pattern_engine.py 的 match_current_to_history()
获取匹配的历史事件。

### Phase 3：用具体事件呈现模式
只用事件，不用标签。

✓ "我记得你和小明在一起 3 个月的时候，你也说过一模一样的话——'他好像没那么在乎我了'。你觉得这次和上次，是一样的感觉，还是不一样？"
✗ "你可能有回避型依恋模式"

### Phase 4：引导用户自己看到
不给结论，用提问：
- "你觉得这次和上次，是一样的感觉，还是不一样？"
```

```python
"""
growth_tracker.py — 成长叙事追踪
纵向对比用户在不同时间点的表述，识别成长节点。
"""

# 理论基础：INT/IMA 框架（EMNLP 2025）定义了叙事疗法中的
# "Innovative Moments" (IMs) 五种类型：
#   - Action: 用户做了不同于旧模式的行为
#   - Reflection: 用户主动反思自己的模式
#   - Protest: 用户挑战旧信念（"我不是回避型"）
#   - Re-conceptualization: 用户重新定义自己（"也许我只是还没学会"）
#   - Performing Change: 用户在新场景中展现新模式
#
# 核心功能：
# 1. extract_growth_nodes(diary_dir) -> list[dict]
#    扫描 diary/*.md，找到 Innovative Moments：
#    - Action: 用户第一次没有立刻发消息追问（不同于旧模式）
#    - Reflection: "算了分就分" → "我想搞清楚为什么每次都这样"
#    - Re-conceptualization: "他不好" → "我好像在重复一个模式"
#    - Performing Change: "我不想聊了" → "我想跟你说说这件事"
#
# 2. find_contrast_pairs(growth_nodes) -> list[tuple]
#    找到可以对比的成长节点对：
#    - 同一话题在不同时间的不同表述
#    - 类似情境下的不同反应
#
# 3. format_for_conversation(contrast_pair) -> str
#    生成可可可以在对话中使用的成长叙事：
#    "三个月前你遇到这种情况，直接就走了。但这次你选择先来跟我聊聊。
#     你有没有发现，这本身就已经不一样了？"
```
