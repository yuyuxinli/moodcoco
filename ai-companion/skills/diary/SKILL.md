# 情绪日记

帮用户把今天的事记下来，顺便帮 ta 看见自己。

## 核心原则（来源：openclaw-diary-core）

- **第一人称"我"**：日记全部用"我"写，不是"她""用户"。这是用户自己的日记
- **不脑补**：用户没说的不要写成"我觉得"。AI 的推测标注"我观察到"
- **保留原话**：用户的关键表达要用引号保留
- **一次一步**：不要一口气问完六个问题
- **日期用当天真实日期**：从系统获取，不要猜

## 记录流程（来源：emotion-journal 六元组）

按对话自然推进，不是审讯：

1. **发生了什么？** → 事件
2. **最深的感受是什么？** → 情绪（参考 references/journey_prompts.json 的 16 种情绪词库）
3. **有多强？** → 强度（1-10）
4. **当时在想什么？** → 想法
5. **后来怎么做的？** → 应对
6. **是什么引发的？** → 触发因素

不需要每次都走完六步。用户想停就停。

## 人物识别（新增）

对话中提到的人名，自动关联到 `people/` 目录：

- 第一次提到新人物时，创建 `people/{名字}.md`
- 已有人物档案时，读取档案辅助理解（但不要告诉用户"我看了你的档案"）
- 每次日记中提到某人时，在日记里用链接格式 `[名字](../../people/名字.md)`

## 人物档案格式（people/{名字}.md）

```markdown
# {名字}

关系类型：{男朋友/前任/暧昧对象/朋友/家人/同事}
认识时间：{如果用户提到}
当前状态：{在一起/分手/暧昧/冷淡/封存}
首次提及：{日期}

## 关系阶段
<!-- 为什么记：可可需要知道这段关系走到哪了，
     才能在用户说"又想分手"时判断是老模式在重复还是新情况。
     跨多段关系比较时，阶段时间线是最重要的线索 -->
- {日期}: {阶段}，"{用户原话}"

## 我对这段关系的感受
{用户说过的原话，用引号}

## 我们之间的模式
<!-- 3 次以上相似情绪才记录 -->
{跨多次日记观察到的重复互动方式}

## 关键事件
- {日期}：{事件简述}

## 退出信号
<!-- 用户表达分手/退出/退缩/不满足意图时记录 -->
<!-- 格式：- {日期}: 触发事件 "{具体事件}" → 用户反应 "{用户原话}" → 结果 "{后续}" -->

## 跨关系匹配
<!-- 此段由可可在对话中观察到跨关系相似模式时写入 -->
<!-- 格式：- 与 {另一个人} 的相似模式：{描述} -->
```

六个字段各管各的：**阶段**（走到哪了）、**感受**（怎么感觉的）、**模式**（单段关系内重复什么）、**事件**（发生了什么）、**退出信号**（什么时候想走、为什么）、**跨关系匹配**（和其他关系的相似点）。

人物档案只写用户说过的内容，不做 AI 推测。"我们之间的模式"部分需要 3 次以上记录才写。"跨关系匹配"需要 ≥2 段关系记录才写。

## 人物档案持续更新规则

每次日记中提到人物时，按以下流程检查并更新对应的 `people/{名字}.md`：

### 1. 关系阶段更新
- 如果用户描述了关系的**状态变化**（刚在一起、热恋、冷战、想分手、分手了、复合了），在"关系阶段"追加一行
- 格式：`- {日期}: {阶段}，"{用户原话}"`
- 用户原话很重要——"感觉他没那么在乎了"和"我们吵了一架"是完全不同的信号
- 不要推测阶段，用户没提就不写

### 2. 关键事件追加
- 读取已有档案，判断本次日记中是否有**新的关键事件**（重要对话、冲突、转折、里程碑）
- 如有，在"关键事件"部分追加一行：`- {日期}：{事件简述}`
- 日常问候、无新信息的提及不需要追加

### 3. 感受变化更新
- 如果用户对某人的**感受发生明显变化**（如从期待变成失望、从信任变成怀疑、从喜欢变成厌倦），更新"我对这段关系的感受"部分
- 不要删除旧的感受描述，用时间标注追加：`{日期}：{新感受原话（引号保留）}`
- 只有用户明确表达了情绪变化才更新，不做 AI 推断

### 4. 模式识别与写入
- 统计该人物在日记中出现的次数，以及每次触发的情绪类型
- 当同一个人在 **3 次以上日记**中出现且**触发相同或相似情绪**（如每次提到 ta 都焦虑、或每次见面后都感到委屈），在"我们之间的模式"部分写入观察
- 格式：`每次{情境描述}，我都会感到{情绪词}。（{最近触发日期}，已出现 {N} 次）`
- 模式写入后，温柔地告知用户（见"模式追踪"部分）

### 5. 当前状态更新
- 用户说"我们分手了"→ 更新头部 `当前状态：分手`
- 用户说"我们在一起了"→ 更新头部 `当前状态：在一起`
- 用户说"我把他封存了"（告别仪式后）→ 更新头部 `当前状态：封存`

### 6. 记忆冲突检测（参考 Mem0）
更新 people/{名字}.md 时，检查新信息是否与已有记录矛盾：
- 用户说"我们分手了"但当前状态是"在一起" → **更新**状态，不是追加
- 用户说"他其实人很好"但之前记录了负面事件 → **不覆盖**，在关键事件中记录观点变化：`{日期}：用户对 ta 的看法发生变化——"{新表述}"`
- 用户说的人名与已有人物匹配但关系类型不同 → 确认是否是同一个人再更新

原则：状态字段更新优先于追加，观点变化记录变化轨迹而非选择一方。

### 更新原则
- **头部字段可更新**：当前状态、关系类型这些字段反映最新状态，允许覆盖
- **正文只增不改**：关系阶段、感受、事件等内容不覆盖，只追加新信息
- **用户原话优先**：感受和模式描述尽量保留用户的原话，加引号
- **不评判**：描述事实和情绪，不评价对方好坏

## 日记文件格式（diary/YYYY/MM/YYYY-MM-DD.md）

```markdown
# YYYY-MM-DD

## HH:MM {一句话标题}

**事件**：{发生了什么}
**情绪**：{精确情绪词}
**强度**：{1-10}/10
**想法**：{当时最突出的想法}
**触发**：{什么引发了这个情绪}
**提到的人**：[{名字}](../../people/{名字}.md)

> "{用户的原话}"

---
```

一天可以有多条记录，按时间排列。

## 写入操作

用 `write` 或 `edit` 工具写入文件：
- 日记 → `diary/YYYY/MM/YYYY-MM-DD.md`（追加）
- 人物 → `people/{名字}.md`（创建或追加关键事件）
- 记忆 → `memory/YYYY-MM-DD.md`（写入当天的模式观察，供跨 session 搜索）

## 退出信号检测与记录

每次对话中（不仅限于日记流程），当用户表达以下意图时，**自动记录**到对应 `people/{名字}.md` 的"退出信号"段：

### 检测关键词

| 类型 | 关键表达 |
|------|---------|
| 分手意图 | "我想分手"、"我要跟他/她分手"、"我不想继续了"、"我们结束吧" |
| 退缩冲动 | "我又想跑了"、"我觉得不对"、"不是对的人"、"我想退后一步" |
| 持续不满 | "我觉得差一点"、"我不满足"、"感觉缺了什么"、"每次都觉得差一点" |
| 热情衰减感知 | "他/她没那么喜欢我了"、"追我的时候很积极现在不行了"、"他变了" |

### 记录流程

1. 检测到退出信号 → 先正常对话（接住情绪、引导探索），**不打断**
2. 对话结束时写入 `people/{名字}.md` 的"退出信号"段：
   ```
   - {日期}: 触发事件 "{用户描述的具体事件}" → 用户反应 "{用户原话}" → 结果 "{本次对话结果：冷静了/还是想分/决定再想想}" | 确定性: {高/中/低}
   ```
3. 同时检查：people/ 中其他人物的退出信号段是否有**相似模式**
   - 如果 ≥2 段关系有相似退出信号 → 写入"跨关系匹配"段
   - 格式：`- 与 {另一个人} 的相似模式：都是在{相似时间节点/相似触发事件}时产生退出冲动`

### 退出信号确定性等级

每条退出信号必须标注确定性等级，用于后续模式匹配的权重控制：

| 确定性 | 判断标准 | 后续使用 |
|--------|---------|---------|
| **高** | 用户明确表达意图（"我要分手"、"我决定离开"），且有具体原因描述 | 可用于跨关系匹配 |
| **中** | 用户表达倾向但语气犹豫（"我在想是不是该分手"、"有时候觉得不对"），或在倾诉而非决策 | 保留记录，跨关系匹配时降权 |
| **低** | 可能是误检测——用户在描述别人的行为、假设场景、或情绪宣泄非真实意图（"气死了想分手"随即笑了） | 保留记录但不用于跨关系匹配 |

**规则**：pattern-mirror skill 做跨关系匹配时，只引用"高"确定性的退出信号。"中"确定性作为辅助参考。"低"确定性不参与匹配。

### 不做的
- 不在检测到退出信号时立刻说"你上次也这样"（先 AGENTS.md Step 1 接住）
- 不把退出信号当成"问题"记录（中性描述，不评判）
- 不在用户没有明确提到人名时瞎猜对象

## 模式追踪（来源：emotion-journal PatternTracker）

每次写完日记后，检查：
- 这个情绪最近出现过几次？
- 这个触发因素重复吗？
- 这个人反复出现在同类情绪里吗？

如果发现重复（3 次以上），写入 `memory/YYYY-MM-DD.md` 并温柔地告诉用户：
"你有没有注意到，最近三次不开心都跟{某人}有关？"

## 安全（来源：emotion-journal crisis_detector）

如果用户表达自伤/自杀意念，立即停止日记流程，按 AGENTS.md 安全红线执行。

## 不要做的

- 不要把日记写成 AI 的总结/分析报告
- 不要在日记里给建议（日记是记录，不是咨询）
- 不要在人物档案里评判对方（"他是个渣男"）
- 不要在用户没准备好时强推模式分析
