## 核心原则

### 1. 金发姑娘区间（Right Abstraction Level）

> "恰当的抽象层次在两者之间取得平衡：既足够具体以有效引导行为，又足够灵活以为模型提供强有力的启发式规则。" —— Anthropic

| 失败模式 | 表现 | AGENTS.md 中的例子 |
|---------|------|------------------|
| **过于刚性** | 硬编码 if-else 逻辑，脆弱 | "如果用户消息包含'心跳'且包含'快'且不包含'跑步'，则触发 sigh" |
| **过于模糊** | 高层指令，LLM 自由发挥 | "注意安全" |
| **金发姑娘区间** | 具体信号 + 灵活启发式 | "出现恐慌信号词（心跳快/手抖/停不下来）→ 强制触发 sigh Skill；但如果上下文明确是运动后的正常反应则不触发" |

### 2. 最小高信号原则

> "找到最小的高信号 token 集合，以最大化期望结果的实现概率。" —— Anthropic

- 每条规则必须是"不可推断的"——如果 AI 能从代码/上下文推断出来，就不要写
- 删掉规则后效果不变 = 那条规则多余
- Architecture overviews 不帮助 AI 定位文件（ETH Zurich 实证）

### 3. 只写人不可能推断的内容

> "研究人员建议将内容限制在不可推断的细节上，例如专用工具链或自定义构建流程。" —— ETH Zurich

AGENTS.md 应该写的：
- 自定义的对话风格规则（AI 无法从代码推断"不说应该"）
- 触发条件和路由规则（AI 无法推断"恐慌 → sigh Skill"）
- 安全红线（AI 无法推断危机干预流程）
- 特殊约束（"不超过 80 字"、"不用尊称"）

AGENTS.md 不应该写的：
- 项目结构描述（AI 能 `ls` 看到）
- 文件用途说明（AI 能读文件推断）
- 通用编程规范（AI 已训练过）

### 4. 结构化分段

> "我们建议将 prompt 组织为不同的区段，并使用 XML 标签或 Markdown 标题来划分。" —— Anthropic

AGENTS.md 推荐结构：
```
## 规则优先级（最重要）
安全红线 > 状态触发 > 对话风格 > 记忆规则

## 安全红线（最高优先级）
...

## 状态判断与行动
...

## 对话风格
...

## 记忆规则
...
```

关键：**规则之间必须声明优先级**。当规则冲突时（如"恐慌 → 触发 sigh" vs "自伤 → 危机干预"），AI 需要知道谁优先。

### 5. 示例胜过规则清单

> "对于 LLM 来说，示例就是那些'胜过千言万语的图片'。我们不推荐将一长串边界情况塞进 prompt 中。" —— Anthropic

- 3-5 个精选的、多样化的示例 > 20 条抽象规则
- 示例应覆盖：正常情况、边界情况、违规情况
- 示例格式：用户说 X → AI 应该做 Y（而不是 Z）

### 6. 人工编写，不要 LLM 生成

> "LLM 生成的 context 文件相比不提供任何 context，任务成功率降低了约 3%。" —— ETH Zurich

- AGENTS.md 必须由人工编写
- AI 可以辅助格式化，但核心规则必须来自人的判断
- 定期人工审查和更新

## AGENTS.md 稳定交付检查清单

一份合格的 AGENTS.md 应该满足：

- [ ] **有明确的规则优先级声明**（安全 > 路由 > 风格）
- [ ] **每条规则都可验证**（有具体信号词或数字标准）
- [ ] **冲突场景有解决方案**（多状态并发时的处理顺序）
- [ ] **包含 2-3 个示例**（正常/边界/违规各一个）
- [ ] **没有多余内容**（删掉后效果不变的行都删了）
- [ ] **人工编写和审核**（不是 AI 自动生成的）
- [ ] **结构化分段**（Markdown 标题清晰分隔各类规则）
