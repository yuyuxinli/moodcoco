# Evolve V2 Design

## Problem

Current evolve loop lacks macro thinking. AI reads evaluation feedback and immediately makes detail-level fixes — playing whack-a-mole instead of progressing toward the goal. Each `/loop 1m /evolve` round is a new session with no memory of previous strategic decisions.

Root causes:
1. No cross-session strategy memory — AI forgets its approach every round
2. No trajectory awareness — AI can't tell if it's improving or stuck
3. Evaluation feedback anchors attention on micro-fixes, not macro direction

## Architecture

### Core Principle

```
Simple > Complex
```

5 files. 3 agents. 1 hard constraint (independent evaluator). Everything else relies on the skill ecosystem and model capabilities.

### Three Layers

```
Code Layer:    prepare.py — loop control, state machine, trajectory analysis, stop conditions, evaluator enforcement
Agent Layer:   O / B / C — role definitions, minimal
Skill Layer:   Existing ecosystem, used at agent's discretion
```

### Core Cycle

```
O(user) → B → C → B → C → B → C → ... → Done
 (Init)    └────── Auto Loop ──────────┘
```

- **O (Orchestrator)**: Talks to user, creates program.md, dispatches B and C. Does not touch code or make judgments.
- **B (Builder)**: Reads strategy.md, writes code, commits. Executes only.
- **C (Critic)**: Scores (must call independent evaluator), reads trajectory, makes strategic decisions, writes strategy.md. The brain of the loop.

Single loop. Three agents. No inner/outer loop nesting.

### Why C Owns Strategy

C just saw the code, just ran the tests, just got /codex's independent scores — it is the agent with the freshest understanding of the current state. It decides "what next." B just opens strategy.md and executes.

### Independent Evaluator (Hardcoded in Code)

C must call an independent evaluator — a different model that is not the builder. This is enforced by code in prepare.py, not by prompt. AI cannot modify or bypass it.

```python
# prepare.py — AI MUST NOT modify this file
INDEPENDENT_EVALUATORS = ["codex", "claude"]  # CLI commands, priority order

def get_evaluator():
    """Try each in order, return first available. None = eval cannot proceed."""
    for name in INDEPENDENT_EVALUATORS:
        if is_available(name):
            return name
    return None

def validate_eval_result(result):
    """Called after every eval. No independent evaluator = eval invalid."""
    if not result.get("independent_evaluator_used"):
        raise ValueError("Eval invalid: no independent evaluator was called")
```

- Evaluator list is in code, AI cannot change it
- Every eval result is validated by code before writing to results.tsv
- If all evaluators are unavailable → loop stops, reports to user (does not fall back to self-eval)

### Skills Are Not Hardcoded

Agent definitions only specify roles and rules. Which skills to use is up to each agent at runtime (beyond the independent evaluator requirement above).

O may suggest skills to the user during Init if professional skills are not installed:

```
"Detected web app project. Recommend installing:
  - /qa → systematic testing during evaluation
  - /browse → live page inspection
  Install: git clone xxx .claude/skills/qa
  Optional — works without them, but evaluation quality improves."
```

### Inspired By

- Anthropic Harness V1: cross-session state files (claude-progress.txt)
- Anthropic Harness V2: simplification direction — fewer files, more model capability
- Microsoft Magentic-One: dual loop concept, adapted to single loop for simplicity
- GoalAct (NCIIP 2025 Best Paper): continuously updated global planning
- SWE-EVO benchmark: long-horizon failure = "can't see big picture", not coding ability
- GAN architecture: builder ≠ evaluator (separation of concerns)

## Agent Definitions

### O (Orchestrator)

**Role:** User interface + dispatch. Nothing else.

**Responsibilities:**
- Init: brainstorm with user, generate program.md
- Loop: dispatch B and C alternately (controlled by code in prepare.py)
- Stuck: stop loop, report to user
- Suggest skills to user when useful ones are missing

**Hard constraint:** Loop control is enforced by code (prepare.py), not by O's judgment.

**Skills:** Uses whatever is available at its own discretion.

### B (Builder)

**Role:** Execute. Hands only.

**Responsibilities:**
- Read strategy.md (C told it what to do)
- Read program.md (the user's goal)
- Write code
- git commit after each run (finer rollback granularity)
- Append build output to run.log
- Write build record to results.tsv

**Hard constraint:** Follow strategy.md. Do not make strategic decisions.

**Skills:** Uses whatever is available at its own discretion.

### C (Critic)

**Role:** Evaluate + strategic decisions. The brain.

**Responsibilities:**
- Score the current build (deterministic tests + LLM evaluation)
- Call independent evaluator (codex or claude CLI) — **MANDATORY, enforced by code in prepare.py**
- Read trajectory data (from analyze_trajectory())
- Read strategy.md (own previous decisions)
- Make strategic decision (6 options, picks one):
  - **Continue** — approach working, keep iterating
  - **Pivot** — change technical approach
  - **Rollback** — revert to known good commit
  - **Re-execute** — B deviated from strategy, redo (adherence check)
  - **Decompose** — B's implementation incomplete (stubs/TODO/happy path only), split into sub-tasks for B
  - **Consolidate** — pause feature work, direct B to clean up accumulated entropy (dead code, stale comments, contradictions)
- Write updated strategy.md
- Write scores to results.tsv

**Hard constraint:** Independent evaluator call is programmatically required (enforced by prepare.py). Eval without it = invalid.

**Skills:** Uses whatever is available at its own discretion.

**Temp workspace:** Gets its own working directory for multi-step analysis. Internal implementation detail, not part of file protocol.

### Responsibility Matrix

|                        | O | B | C |
|------------------------|---|---|---|
| Talk to user           | ✓ |   |   |
| Dispatch subagents     | ✓ |   |   |
| Loop control (code)    | ✓ |   |   |
| Suggest skills to user | ✓ |   |   |
| Read strategy.md       |   | ✓ |   |
| Write code             |   | ✓ |   |
| Score (independent eval)|   |   | ✓ |
| Read trajectory        |   |   | ✓ |
| Write strategy.md      |   |   | ✓ |
| Strategic decisions    |   |   | ✓ |
| git commit (each run)  |   | ✓ | ✓ |

## File Protocol (5 files)

| File | Purpose | Written By | Lifecycle |
|------|---------|-----------|-----------|
| `program.md` | User contract: goals + eval criteria + feature list | User via O (Init) | Immutable after Init |
| `adapter.py` | Deterministic test oracle: setup / run_checks / teardown | O auto-generates (Init) | User can modify |
| `strategy.md` | C's strategic decisions: approach, trajectory, next action | C (overwrite each round) | Reset on new feature |
| `results.tsv` | Append-only fact log: build records + eval scores | B + C append | Never modified |
| `run.log` | All agent output with timestamp separators | O + B + C append | Append only |

Internal (not part of protocol): `lock` (concurrency guard in prepare.py), C's temp workspace.

### Why adapter.py Is Necessary

adapter.py is the **external oracle** — the only source of deterministic, reproducible test results. Without it, all scoring relies on LLM judgment, which:
- Produces different scores for identical code across runs (NAACL 2025: 89% error rate variance)
- Makes it impossible to attribute score changes to code improvements vs test method drift
- Creates circular reasoning: LLM builds code, LLM decides how to test it, LLM judges the result (ICLR 2024: LLMs cannot self-correct without external feedback)

adapter.py provides the ground truth. C's skills provide supplementary qualitative evaluation on top.

```python
# adapter.py interface — 3 functions
def setup(project_dir):     # Start the project (e.g., web server)
def run_checks(project_dir, feature):  # Run deterministic tests → {"scores": {...}, "details": "..."}
def teardown(info):          # Clean up (e.g., kill server)
```

### program.md Format

```markdown
# Program

## Product Requirements
<what the user wants to build>

## Feature List
- [ ] Feature A
- [ ] Feature B
- [ ] Feature C

## Evaluation Criteria
dimensions:
  - name: <dimension name>
    type: deterministic | llm-judged
    cmd: <command>  # optional, for deterministic
    threshold: <float>

## Technical Constraints
- Stack: <detected from project scan>
- Dependency limits: <from user>
- No-go zones: <from user>

## Agent Rules
- Do not modify program.md
- Do not modify .claude/skills/evolve/
- Git commit after each agent run (not per feature — finer rollback granularity)
- Build output appended to .evolve/run.log
```

### strategy.md Format

```markdown
# Strategy: <feature name>

## Trajectory
<!-- analyze_trajectory() auto-fills -->
trend: rising | flat | falling | insufficient
scores: [6.5, 6.8, 7.1]
rounds_on_this_feature: 3

## Approach
<current technical approach>

## Last Round
<what was done, brief>

## Next Action
<!-- C picks exactly one. Priority top-down: use simplest sufficient action. -->

### Default (most rounds should be here)
☐ Continue — total rose or on track, keep iterating
☐ Rollback to commit <hash> — total dropped, revert and retry

### Escalation (consecutive flat/failing triggers)
☐ Pivot: <new approach description> — N+ rounds flat, change technical approach
☐ Re-execute — B deviated from strategy, redo without changing direction

### Structural
☐ Decompose: <sub-tasks list> — B's implementation incomplete, split into sub-tasks
☐ Consolidate — entropy high, B cleans up dead code / stale comments / contradictions

## Decision Signals
- total rising → Continue (default)
- total dropping → Rollback (default)
- flat for N rounds → Pivot (escalation)
- score OK but code diverged from strategy → Re-execute (escalation)
- B wrote stubs / TODO / only happy path → Decompose (structural, reactive)
- dead code / stale comments / contradictions accumulating → Consolidate (structural)

## Execution Constraint
Approach executed for N/2 rounds (minimum 2 rounds before pivoting)
```

### results.tsv Format

```
commit	phase	feature	scores	total	status	summary
a1b2c3d	build	Auth	-	-	keep	implemented JWT auth
b2c3d4e	eval	Auth	8.1/6.2	7.15	fail	code quality below threshold
```

Headers: commit, phase, feature, scores, total, status, summary

Valid phases: build, eval
Valid statuses: keep, pass, fail, crash

No contract phase. No skip status.

### run.log Format

```
========== 2026-03-29 14:32:01 | Round 12 | Build Auth ==========
npm run build output...

========== 2026-03-29 14:33:15 | Round 12 | Eval Auth ==========
test output...
```

Append-only with timestamp separators per round.

## Init Flow (4 steps)

### Step 1: Project Scan (automatic)

Scan language, framework, test framework, directory structure.

> "Detected Node.js + Express project, has vitest, entry at app/main.js"

### Step 2: Brainstorming (interactive, core step)

O talks to user following /brainstorming principles:
- One question at a time
- Multiple choice preferred
- Goal: help user clarify three things — what they want, what "good" means, what's off limits

```
Q1: "What do you want to build? One sentence."

Q2: "Core features? I suggest based on your project:
     A. JWT authentication
     B. Chat endpoint
     C. File upload
     D. Other: ___"

Q3: "What matters for quality?
     A. Tests pass (deterministic)
     B. Code quality (AI review)
     C. API design (AI review)
     D. Other: ___"

Q4: "Score threshold per dimension? Default 7/10."

Q5: "Constraints?
     A. No new dependencies
     B. Don't touch xxx directory
     C. No limits
     D. Other: ___"
```

3-5 questions, ~2 minutes. O may suggest skills for user to install based on project type.

### Step 3: Generate program.md (automatic + user review)

Auto-generate from Step 1 scan + Step 2 conversation. Show to user: "Here's your program.md. Want to adjust?"

### Step 4: Validation + Branch Creation (automatic)

Validation:
- Product requirements non-empty
- At least 1 eval dimension
- No template placeholders
- Python 3.8+, Git available
- Warn if uncommitted changes

Setup:
- `git checkout -b evolve/<tag>`
- Generate `.evolve/adapter.py` (from reference adapters + project scan)
- Create `.evolve/results.tsv` (header only)
- Create `.evolve/strategy.md` (empty template)
- Create `.evolve/run.log` (empty)
- Add `.evolve/` to `.gitignore`

> "Init complete. Run `/loop 1m /evolve` to start."

## Auto Loop

### State Machine

```
build/keep       → dispatch C (evaluate the build)
build/crash      → dispatch B (fix)
eval/pass        → dispatch B (next feature; C resets strategy.md)
eval/fail        → dispatch B (C already updated strategy.md with decision)
All features pass → Done
```

No skip. Stuck = code stops the loop, reports to user.

### Per-Round Flow

```
/evolve triggered
  │
  ▼
prepare.py: should_stop() runs FIRST          ← Code decision, AI not started yet
  │
  ├─ Returns True → DO NOT dispatch AI. Output report to user. Loop stops.
  │
  └─ Returns False → Continue:
       prepare.py reads results.tsv → state machine routes
         │
         ├─ Need B? → O dispatches B subagent
         │   B reads: program.md + strategy.md
         │   B does: write code → commit → append run.log + results.tsv
         │
         ├─ Need C? → O dispatches C subagent
         │   C does: adapter.run_checks() + own analysis + independent evaluator (MANDATORY)
         │   C reads: strategy.md + trajectory (analyze_trajectory)
         │   C decides: continue / pivot / rollback
         │   C writes: strategy.md + results.tsv
         │
         └─ All done? → O reports to user, loop stops
```

### Stop Conditions (Hardcoded in prepare.py, AI Cannot Modify)

```python
# prepare.py — AI MUST NOT modify this file

HARD_LIMITS = {
    "max_rounds_total": 100,          # ~1-2 days of /loop 1m /evolve
    "max_rounds_per_feature": 30,     # ~half a day per feature
    "max_consecutive_crashes": 5,     # build keeps crashing
    "max_consecutive_fails": 10,      # eval keeps failing
    "max_flat_after_pivot": 3,        # pivoted but still no improvement
}

def should_stop(results_tsv, feature):
    """
    Called BEFORE AI is dispatched. Returns (stop: bool, reason: str).
    AI does not participate in this decision.
    """
    progress = read_progress(results_tsv)
    trajectory = analyze_trajectory(results_tsv, feature)

    if progress["total_iterations"] >= HARD_LIMITS["max_rounds_total"]:
        return True, "Total round limit reached"

    if progress["feature_iterations"] >= HARD_LIMITS["max_rounds_per_feature"]:
        return True, f"{feature}: per-feature round limit reached"

    if progress["consecutive_crashes"] >= HARD_LIMITS["max_consecutive_crashes"]:
        return True, f"{feature}: consecutive crashes"

    if progress["consecutive_fails"] >= HARD_LIMITS["max_consecutive_fails"]:
        return True, f"{feature}: consecutive eval failures"

    # Adaptive: pivoted N times but trajectory still flat
    if (trajectory["trend"] == "flat"
        and progress["pivots_on_this_feature"] >= HARD_LIMITS["max_flat_after_pivot"]):
        return True, f"{feature}: pivoted {HARD_LIMITS['max_flat_after_pivot']} times, still no improvement"

    return False, ""
```

When stopped, output:
```
⛔ <feature> stopped by code after N rounds
   Reason: <from should_stop>
   Recent scores: [x, y, z]
   Last strategy: <from strategy.md>
   Action needed: adjust program.md / lower threshold / provide hints
   Re-run /evolve to continue.
```

## analyze_trajectory()

Added to prepare.py:

```python
def analyze_trajectory(results_tsv: str, feature: str, window: int = 3) -> dict:
    """
    Extract recent eval scores for a feature, determine trend.

    Returns:
        {
            "trend": "rising" | "flat" | "falling" | "insufficient",
            "scores": [float, ...],
            "rounds": int,
            "latest": float
        }

    Logic:
        - Fewer than window eval rows → "insufficient"
        - latest - earliest > +0.5 → "rising"
        - latest - earliest < -0.5 → "falling"
        - Otherwise → "flat"

    Parameters:
        - window=3: last 3 eval rounds (tunable)
        - ±0.5 threshold on 10-point scale (tunable)
        - Only reads eval phase rows, ignores build/crash
    """
```

## Design Principle: Natural Selection

The loop has a built-in natural selection mechanism:

```
C (Claude) = generates strategy (variation)
Independent evaluator (codex/claude CLI) = scoring (selection pressure)
adapter.py = deterministic ground truth (environmental constraint)
results.tsv = fossil record (what survived, what didn't)
```

C proposes approaches. The independent evaluator + adapter.py judge them objectively. Bad strategies score low and get replaced. Good strategies survive and propagate. C must adapt to the selection pressure, not the other way around.

Two layers of selection pressure:
1. **adapter.py** — deterministic tests, reproducible, unfakeable (the physics of the environment)
2. **Independent evaluator** — cross-model LLM judgment, different blind spots than C (the predator)

Without either, C could fool itself. adapter.py provides the hard floor. The independent evaluator catches what deterministic tests miss.

**Escalation:** When C's strategies repeatedly fail selection (consecutive pivots ≥ max_flat_after_pivot, still flat), prepare.py's `should_stop()` halts the loop. The system admits "my variation + selection cycle isn't producing fit strategies" rather than spinning forever. This decision is made by code, not by C.

## Future: V3 Candidates

### Parallel Competition Mode

When serial natural selection (one strategy at a time) isn't enough, allow C to spawn 2-3 variant strategies in parallel:
- O dispatches 2-3 B agents (separate git branches)
- C + /codex evaluates all variants, selects winner
- Winner continues, losers discarded

Not in V2. Serial pivot is sufficient until proven otherwise.
